# NotebookLM 用参照資料：工程管理 AI 配台システム（planning_core 中心）

この文書は、Google NotebookLM などに「ソース」として読み込ませ、質問応答や要約の材料にすることを想定して書いています。工場向け配台計画シミュレーションの目的、全体の流れ、データの所在、Gemini の使い方、直近で拡張されたロジック（グローバルコメントによる休業、need の追加人数、チーム選定、利用量トラッキング）を、できるだけ一文一文で意味が通じるように説明します。実装の最終的な正確さはリポジトリ内の `planning_core.py` に従ってください。

---

## 第1部　システムの目的と全体像

### 1.1 何をするシステムか

このシステムは、工場における「どの設備で、いつ、誰が、どの加工タスクをどれだけ進めるか」を、Excel 上のマスタデータとタスク入力に基づいて自動的に割り当て（配台）し、結果を Excel で出力するためのものです。人的リソースのスキル（OP／AS）、各設備に必要な人数、勤怠（出勤・休憩・休暇）、タスクの優先度や納期、特別な指示文などを考慮します。

### 1.2 中核となるプログラム

中核は単一の Python モジュール `planning_core.py` です。VBA やマクロから環境変数でパスを渡して呼び出す設計になっています。スクリプトと同じフォルダが実行時のカレントディレクトリに設定されるため、`master.xlsm` や `log` フォルダは相対パスで解決されます。

### 1.3 外部から使う主な入口

- **環境変数 `TASK_INPUT_WORKBOOK`**  
  マクロ側の「加工計画」や配台計画が入ったブック（例：`.xlsm`）のフルパスを指定します。このブックの中に「配台計画_タスク入力」シートや「メイン」シート、設定シートなどが含まれます。

- **`run_stage1_extract()`**  
  段階1の処理。加工計画データの抽出、中間 Excel の生成、配台不要に関する設定シートの同期や Gemini による補完などが含まれます。`task_extract_stage1.py` から呼ばれる想定です。

- **`generate_plan()`**  
  段階2の処理。マスタの skills・need・勤怠とタスク入力を読み込み、日付ごとにループして設備と担当者へ割り付け、結果ブックと個人別スケジュールを `output` 配下に出力します。`plan_simulation_stage2.py` から呼ばれる想定です。

### 1.4 ざっくりした処理の流れ（2段階）

**段階1**では、元データ（例：加工計画 DATA）から配台計画用のタスク一覧を整え、`output/plan_input_tasks.xlsx` のような中間成果物を出します。また「設定_配台不要工程」シートの保守や、ルールの AI 補完、ブックの保存（openpyxl または COM）が絡みます。

**段階2**では、`master.xlsm` からスキル・必要人数ルール・各メンバーの勤怠を読み、環境変数で指定されたブックから配台対象タスクを読みます。その後、日付の昇順でループし、その日出勤しているメンバーの空き時間と設備の空きを見ながら、タスクにチームを割り当てます。結果は日付付きの `output/年/月/日/` フォルダに、設備別の時間割・タスク一覧・カレンダー風の結果などとして Excel で保存されます。実行ログは `log/execution_log.txt` に追記・上書きされます。

### 1.5 主なファイルとフォルダ（実行ディレクトリ基準）

- **`master.xlsm`**  
  skills シート、need シート、メンバーごとの勤怠シート（カレンダー）を含む統合マスタ。

- **`log/`**  
  実行ログ、Gemini のプロンプト・応答の保存、利用量累計 JSON、グラフ用 CSV、メインシート向けサマリテキストなど。

- **`output/`**  
  段階1の中間ファイル、段階2の日付別フォルダ内の `production_plan_multi_day_*.xlsx`、`member_schedule_*.xlsx` など。

## 第2部　段階2（generate_plan）の内部の流れ（概念）

段階2は `generate_plan()` が本体です。ここでは「直近拡張」と関係が深い順に、概念だけ説明します。

1. **スキルと need の読込**  
   `load_skills_and_needs()` が `master.xlsm` の skills と need を読み、メンバー一覧、設備（工程）一覧、工程名＋機械名ごとの**基本必要人数**、特別指定ルール、さらに**配台時追加人数**（余剰時に増やせる人数の上限）を辞書として返します。戻り値は6要素で、最後が `surplus_map` です。

2. **勤怠の読込と AI 解析**  
   `load_attendance_and_analyze(members)` が各メンバーシートを読み、備考列などを必要に応じて Gemini で構造化し、日付ごと・メンバーごとの `is_working`、出勤退勤時刻、休憩、効率などを組み立てます。

3. **グローバルコメントの読込と工場休業の上書き**  
   メインシートの「グローバルコメント」見出しの直下のテキストを読みます。同じ文字列は後段で「メイン再優先特記」として Gemini に渡ることもありますが、**工場全体の休業日**については、ルールベースのパーサが先に働きます。パーサが日付を検出すると、その日は全員 `is_working=False` に上書きされ、備考に工場休業のタグが付きます。その後、計画対象の日付リストが確定します。

4. **タスク入力の読込とキュー構築**  
   配台計画シートから DataFrame を読み、配台不要の行を除き、特別指定備考を Gemini で解析した結果とマージし、タスクのキュー（残量、優先度、納期、開始希望日など）を構築します。

5. **メイン再優先特記（Gemini）**  
   グローバルコメント全文を解釈し、「スキル無視」「必要人数を1に」「配台制約の撤廃」などのフラグや、依頼NOごとの主担当指名を JSON で返させます。キャッシュがあれば API を呼ばずに再利用します。

6. **日付ループでの割付**  
   各日について、出勤しているメンバーの `avail_dt`（次に作業可能な時刻）を初期化し、その日開始可能なタスクに対してチーム候補を列挙します。チームは need で決まる最小人数から、最小＋追加人数上限までの人数について組合せを試します。各候補について、その日にこなせる単位数と終了時刻を計算し、**比較用タプル**が最小のチームを採用します（詳細は後述）。採用後、メンバーと設備の空き時刻を更新し、タスクの残量を減らします。

7. **Excel 出力とログ**  
   設備別ガント風シート、タスク一覧、カレンダー、稼働率などを書き出します。Gemini 利用サマリを log に保存し、可能ならメインシートの P 列へ openpyxl で反映します。

## 第3部　Gemini API 利用量の記録と可視化

### 3.1 なぜ記録するか

コスト管理と利用状況の把握のため、Gemini の各応答についてトークン数や推定料金を累積し、ファイルに残します。コンソールに毎回の単価を出さず、ログと JSON に集約する方針です。

### 3.2 累計 JSON の場所と役割

ファイル名は `log/gemini_usage_cumulative.json` です。バージョン1形式で、累計の呼出し回数、プロンプト／候補／思考などのトークン合計、モデル別の内訳、推定 USD 累計などが入ります。

### 3.3 期間別バケット（buckets）

同じ JSON 内に `buckets` オブジェクトがあります。API を1回呼ぶたびに、その時点の PC ローカル時刻で次のキーにトークンと回数と推定コストを加算します。

- **by_year** … 例：`2026`
- **by_month** … `YYYY-MM`
- **by_week** … ISO 週番号 `YYYY-Www`
- **by_day** … `YYYY-MM-DD`
- **by_hour** … `YYYY-MM-DDTHH`（時まで）

複数 PC で同じファイルを共有すると時刻基準が混ざるため、JSON 内に「period_key はローカル時刻」という注意書き（timezone_note）を入れています。既存の累計ファイルに buckets が無かった場合、読み込み時に空の構造を補いますが、**過去の呼出しを遡ってバケットに埋め直すことはしません**。導入以降の呼出しからしか積み上がりません。

### 3.4 グラフ用 CSV

`log/gemini_usage_buckets_for_chart.csv` に、長形式で行を出力します。列には粒度（year / month / week_iso / day / hour）、期間キー、呼出し回数、各種トークン、推定 USD・JPY などがあります。UTF-8 BOM なので Excel で開きやすく、ピボットやフィルターでグラフ化できます。

### 3.5 メインシート向けサマリテキスト

`build_gemini_usage_summary_text()` が、当実行分と累計を人が読める複数行テキストにします。累計の後ろに「期間別集計」として、年は全件、月・週・日・時は新しい順に件数を制限して列挙し、上記 CSV のパスを案内します。`write_main_sheet_gemini_usage_summary` のタイミングで CSV も更新します。

## 第4部　メインシート「グローバルコメント」の二重の役割

### 4.1 読み取り位置

`TASK_INPUT_WORKBOOK` で指定されたブックの「メイン」または「Main」、あるいはシート名に「メイン」を含むシートを探し、セル値が「グローバルコメント」とみなせる見出しセルの**真下のセル**を文字列として読みます。表記は NFKC 正規化や部分一致でゆるくマッチします。

### 4.2 役割その1：メイン再優先特記（Gemini）

同じ全文が `analyze_global_priority_override_comment` に渡り、JSON で次のような意味を取り出します。例：スキル要件を無視する、チーム人数を need より1人に固定する、設備専有や時刻ガードなどを緩める、依頼NOと氏名の対応で主担当を上書きする、など。これらは配台ループで最優先で参照されます。

### 4.3 役割その2：工場休業日（ルールベース・AI 不要）

全文を改行や句読点でチャンクに分け、各チャンクが「工場単位の休業」を示すかどうかをキーワードで判定します。該当するチャンクから日付文字列を抽出し、西暦付き・月日のみ（年は計画基準年）など複数形式に対応します。抽出された各日付について、勤怠データ上のその日の全メンバーを非稼働にし、備考の先頭に「工場休業（メイン・グローバルコメント）」という印を付けます。その日は稼働メンバーが0人になるため、配台ループでは設備への割付がスキップされます。

工場休業と判定しやすい例としては、「臨時休業」「工場が休み」「休場」「生産停止」「加工しない」などの表現です。個人の休みだけが書かれている文と誤判定しにくいよう、「工場」「全社」などスコープを示す語と組み合わせるルールになっています。

### 4.4 マスタに行が無い日

グローバルコメントで休業とした日が、マスタ勤怠の日付列に存在しない場合、上書きの対象が無く警告ログだけが残ることがあります。計画ループがその日をそもそも走査しない場合、休業の効果が表に出にくいので、カレンダー上その日を全員分持つ運用が確実です。

## 第5部　need シート：必要人数と「配台時追加人数」

### 5.1 Excel 上のイメージ

need シートは、列方向に工程名と機械名の組が並び、行方向に「基本必要人数」と「特別指定1〜99」が続く構造です。実装はヘッダ行を A 列のラベル文字列から自動検出します。**「必要人数」を含む行が基本必要人数行**です。

その**直下の行**を「配台時追加人数」行として読みます。A 列に「追加人数」「増員」「余剰」などの語があれば確実に認識し、ラベルが弱い場合でも、特別指定行ではなく工程×機械列に数値があれば追加人数行とみなすフォールバックがあります。多くのブックでは Excel の5行目付近に当たります。

### 5.2 基本必要人数と追加人数の意味

- **基本必要人数**  
  その設備・工程のタスクを回すために最低限そろえる人数（OP／AS の組合せは skills 側で判定）。

- **配台時追加人数（上限）**  
  最低人数を満たしたうえで、「余剰人員がいるなら追加で入れてよい人数」の上限です。現場の運用として、配台結果で余った人を最大何人までこの設備に乗せられるかをマスタで表現します。

### 5.3 プログラム内での解決

`resolve_need_required_op` がタスクの工程名・機械名・依頼NO（特別指定の条件照合）から整数の必要人数を返します。`resolve_need_surplus_extra_max` が同様のキーで追加人数の上限を返し、0〜50 にクリップします。特別指定ルールは現状、追加人数側は上書きしません（関数引数に task_id を渡す余地は将来用）。

### 5.4 配台シート側の上書き

タスク行に「必要人数」列などで上書きがあればそれが優先され、メイン再優先で「人数を1に」が有効なときは強制的に1になります。

### 5.5 チーム候補の列挙

ある日のあるタスクについて、`req_num` を最小人数、`max_team_size = min(req_num + extra_max, その日のスキル適合メンバー数)` とし、`req_num` から `max_team_size` までの各人数について、メンバーの組合せを生成します。担当 OP 指名（特別指定やグローバル指名）がある場合は、その人を OP として含む組合せに絞るか、人数が足りなければ指名を無視して一般の組合せにフォールバックする分岐があります。いずれにせよ、チーム内に少なくとも1名の OP が必要で、AS だけの組合せは捨てます。

## 第6部　加工時間と「増員しても速くなりすぎない」モデル

現場の感覚に合わせ、人数を増やしても単位あたり時間が線形に短くなり続けないようにしています。

単位あたりの実効時間は、おおむね「タスクのベース時間 ÷ メンバー平均効率 ÷ タスク効率」に、増員係数を掛けた形で計算します。増員係数 `_surplus_team_time_factor` は次の思想です。必要人数を `rq`、実際のチーム人数を `n`、マスタで許された追加人数の上限を `em` とします。`n <= rq` または `em == 0` のとき係数は1（増員の効果なし）。それ以外、実際に増やした人数 `extra = n - rq` について、`frac = min(1, extra / em)` とし、係数は `1 - R * frac` です。ここで `R` は定数 `SURPLUS_TEAM_MAX_SPEEDUP_RATIO` で、既定値 0.05 です。つまり追加枠を使い切っても、単位時間が短くなるのは最大で約5%です。途中の人数では線形に補間されます。

これにより、「余剰を載せた方がその日の処理単位数がわずかに増える」ことはありますが、劇的な加速にはなりません。そのうえでチーム選定で人数を優先するモードでは、開始時刻が少し遅くても多人数を選びやすくなります。

## 第7部　チーム選定の比較ルール（辞書式最小）

複数のチーム候補のうちどれを採用するかは、比較用タプルを作り、Python のタプル比較で**小さい方**を選びます。

### 7.1 既定モード：余剰人員の活用を優先

環境変数 `TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF` がオフでないとき（既定はオン）、タプルは次の順です。

1. **マイナスのチーム人数**（`-len(team)`）。人数が多いほどこの値は小さくなるので、まず人数が多いチームが有利です。
2. **チーム全体の作業開始時刻**（早いほど有利）。
3. **マイナスのその日処理できる単位数**（多いほど有利）。
4. **スキル優先度の合計**（小さいほど有利）。

### 7.2 従来モード：開始時刻を最優先

同じ環境変数を `0` や `false` などでオフにすると、タプルは「開始時刻 → 単位数 → 優先度合計」だけになり、以前の「早く始められるチーム優先」に戻ります。

### 7.3 開始時刻の決め方（チーム全体）

チームの開始時刻は、メンバー各自のその時点での空き時刻の**最大値**です。したがち、遅い人に足を引っ張られる形になり、人数が増えるほど開始が遅くなりやすい構造です。従来モードではこれが1人採用を促しやすかった背景です。既定モードでは人数を先に比較するため、開始が少し遅くても多人数が選ばれやすくなります。

## 第8部　デバッグとトレース

### 8.1 DEBUG_TASK_ID

環境変数 `DEBUG_TASK_ID`（既定例は依頼NO）を指定すると、タスクキューの並びや特定日の割付対象に、その依頼NOが含まれるかなど、開発用の INFO ログが増えます。

### 8.2 TRACE_TEAM_ASSIGN_TASK_ID

別の環境変数 `TRACE_TEAM_ASSIGN_TASK_ID` に依頼NO（例：`W3-14`）を設定すると、そのタスクについて配台ループ内で、必要人数・追加上限・当日の候補メンバー一覧、再優先フラグ、OP 指名の解決結果、**人数ごとの「その人数で最良のチーム」**、最終的に採用したチームと比較タプルの意味が INFO で出ます。マクロから Python を起動するだけでは環境変数が継承されないことがあるため、バッチや PowerShell で変数をセットしてから同じプロセスで段階2を実行する必要があります。

### 8.3 実例の解釈（W3-14）

依頼NO W3-14 のように、特別指定で「担当者を富田さんへ固定」とある場合、preferred_operator が解析結果に入り、配台は富田を OP として含む組合せを優先します。need マスタ上で「検査×熱融着機　湖南」の基本必要人数が1で追加上限が1なら、試すのは1人チームと2人チームです。既定の人数優先モードでは、有効な2人チームが存在すればそれが選ばれやすくなります。OP が実質1人しかいない日などでは、結局1人に見える結果になります。

## 第11部　短い用語集

- **配台** … タスクに対し、日時・設備・担当チームを決めること。
- **OP / AS** … skills シートのセルで示される役割。OP が主担当側。チームには少なくとも1 OP が必要。
- **need** … 設備（工程×機械）ごとの人数要件と特別指定。
- **余剰・追加人数** … 必要人数を超えて割り当て可能な人数の上限（マスタの配台時追加人数行）。
- **再優先特記** … メインのグローバルコメントを Gemini が解釈した結果。配台制約を例外で緩めたり、依頼ごとに主担当を指名したりする。
- **buckets** … Gemini 利用量を年・月・週・日・時に集計した辞書。
- **辞書式最小** … タプルの先頭から順に比較して小さい方を選ぶ方式。
