## 別ボタンで足したい動作の設計案

やりたいことは次の独立した処理です。

1. シート上の **「配台試行順番」に入っている値を「並び順のキー」として float 解釈**（例: 1, 2, 1.5）  
2. **その float の昇順**で DataFrame の行を並べ替え（**同値は元の行順で安定**させると安全）  
3. 並べ替え後に **1, 2, …, n と自然数を振り直す**  
4. 既存と同様、**ヘッダー行は据え置き、データ行だけ** `refresh_plan_input_dispatch_trial_order_via_xlwings` と同じ **行列の組み立て・`ws.range(...).value` 書き戻し**のパターンで Excel に反映  

実装のコツ:

- **マスタ読込・`fill_plan_dispatch_trial_order_column_stage1` は呼ばない**（ユーザーの手入力キーを捨てないため）。  
- **列のクリアもしない**。  
- 空セル・解釈不能な文字は、既存の段階1ソートに倣うなら **「有効キーより後ろ」** に回すのが自然です。  
- 処理後は **1..n の正の整数**になるので、段階2の **`_task_queue_all_have_sheet_dispatch_trial_order`**（全行が正の整数か）とも整合します。

## 仕様で決めておくとよい点

- **同じ float の行が複数あるとき**: 安定ソートで元の並びを維持するか、別タイブレーク列を使うか。  
- **空・不正セルがある行**: 末尾にまとめるか、エラーで止めるか。  
- **既存ボタンと同じ `_apply_planning_sheet_post_load_mutations` を回すか**: 行同期などは揃えたいが、**「小数は並べ替えキー」だけ**にしたいなら、**軽い版（ミューテーション省略 or 最小限）**にすると意図がブレにくいです。

## 仕様の整理（あなたの回答どおり）

| 項目 | 採用 |
|------|------|
| 同じ float が複数行 | **エラーで停止**（重複キーは不可） |
| 空・不正セル | **エラーで停止**（「末尾へ」は使わない） |
| 並びの根拠 | **シート上の「配台試行順番」セルだけ**（納期・need・上書き・マスタによる自動試行順は一切使わない） |

このため、**既存の「再計算」経路とは別関数**にし、次は**呼ばない**のが筋です。

- `fill_plan_dispatch_trial_order_column_stage1`（マスタ・ルール由来の試行順）
- `_apply_planning_sheet_post_load_mutations`（設定シート連携・上書き系など、行や列の意味を変えうる前処理）

読み取り → 検証 → float 昇順ソート → 1..n 付与 → 行列を書き戻し、までを **xlwings の現在のシート内容だけ**で完結させます。

## 実装時の検証ルール（提案）

- **対象行**: ヘッダー直下から、いわゆる「データ行」をどう定義するかだけ決める必要があります。  
  - 例: 「依頼NO と工程名のどちらかが空の行はデータ外としてスキップ」など、既存の計画シートの慣習に合わせる。  
  - **スキップした行**まで「配台試行順番を必須」にするかどうかで挙動が変わります（全部必須にすると、末尾の空行までエラーになりがちです）。
- **float の解釈**: Excel が数値で入っていればそのまま、文字列なら `parse_float_safe` 等で厳密に。  
  - **NaN / 空 / 解釈不能** → 即エラー。  
  - **重複した float** → 即エラー。  
- **負数・0 を許すか**: 並びの「キー」として 0 や負も昇順でよいなら可。段階2は最終的に **1..n の正の整数**になるので、キーとして負を許しても問題は出にくいですが、「キーも正のみ」にしたいならその時点でエラーにするとよいです。

## 「上書きに左右されない」との関係

`_apply_planning_sheet_post_load_mutations` を回さないことで、**設定シートや上書きルールが行内容を書き換えたうえで並べ替える**ことは避けられます。  
並べ替え・連番化の入力は **実行直前に見えているセルだけ**になります。
