## 2. Gemini 利用量の集計・出力

### 2.1 累計 JSON

- **パス**: `log/gemini_usage_cumulative.json`（定数 `GEMINI_USAGE_CUMULATIVE_JSON_FILE`）
- **内容**: 全実行を通じた呼出し回数・トークン合計・モデル別集計・推定 USD 累計など。
- **期間別バケット**（`buckets`）:
  - `by_year`（例: `2026`）
  - `by_month`（`YYYY-MM`）
  - `by_week`（ISO 週 `YYYY-Www`）
  - `by_day`（`YYYY-MM-DD`）
  - `by_hour`（`YYYY-MM-DDTHH`）
- **時刻の意味**: キーは **API 応答を記録した PC ローカル時刻**（`datetime.now()`）。`timezone_note` にその旨を JSON 内に保持。
- **既存ファイル**: 旧 JSON に `buckets` が無い場合は読込時に空構造を補う。**過去ログのバックフィルはしない**（以降の呼出しから蓄積）。

### 2.2 グラフ用 CSV

- **パス**: `log/gemini_usage_buckets_for_chart.csv`（`GEMINI_USAGE_BUCKETS_CSV_FILE`）
- **形式**: UTF-8 BOM（Excel 向け）
- **列**: `granularity`, `period_key`, `calls`, `prompt_tokens`, `candidates_tokens`, `thoughts_tokens`, `total_tokens`, `estimated_cost_usd`, `estimated_cost_jpy`
- **出力タイミング**: `write_main_sheet_gemini_usage_summary` 実行時、`calls_total > 0` のとき再生成。

### 2.3 サマリテキスト

- `build_gemini_usage_summary_text()` がメイン用テキストを生成。
- 累計ブロックの後に **【期間別集計】**（年は全件、月・週・日・時は新しい順に件数制限）と CSV パス案内を付与。

## 3. メイン「グローバルコメント」と工場休業

### 3.1 読み取り

- 関数: `load_main_sheet_global_priority_override_text()`
- **位置**: `TASK_INPUT_WORKBOOK` のメイン系シートで、見出しセル「グローバルコメント」（表記ゆれ可）の **直下セル**。
- 同じテキストは **メイン再優先特記** の AI 解釈にも渡る（別ロジック）。

### 3.2 工場休業日の自動判定（ルールベース）

- 関数: `parse_factory_closure_dates_from_global_comment(text, default_year)`
- **default_year**: 計画の基準日の年（データ抽出日ベースの `run_date.year`）。
- **分割**: 改行・`。`・`;`・`；` でチャンク分割し、**チャンクごと**に「工場休業っぽいか」と日付を判定。
- **工場休業の目安キーワード**（チャンク内）: 例）`臨時休業`、`工場`＋`休/休業/休み/停止`、`休場`、`稼働/生産/ライン`＋`停止/なし`、`加工しない` 等、`休業`＋`工場/全社/当日` 等。
- **日付表記**: `YYYY年M月D日`、`YYYY/M/D`、`M月D日`（年は default_year）、`M/D`（月日）。
- **適用**: `apply_factory_closure_dates_to_attendance(attendance_data, members, closure_dates)`
  - 該当日×メンバーで `is_working=False`、備考先頭に `工場休業（メイン・グローバルコメント）` を付与。
- **注意**: マスタ勤怠に存在しない日は警告のみ（その日が `sorted_dates` に無いと配台ループに載らない）。

### 3.3 処理順序（段階2）

`load_attendance_and_analyze` の直後にグローバルコメントを読み、工場休業日を勤怠に上書きしてから `sorted_dates` を確定する。

## 4. need シート（必要人数と配台時追加人数）

### 4.1 レイアウト（概念）

- A 列等で **工程名行・機械名行** を検出したうえで、**「必要人数」を含む行**を **基本必要人数** 行とする。
- その **直下行**を **配台時追加人数** 行として検出（ラベルに「追加人数」「増員」「余剰」「配台＋追加」等、または工程×機械列に数値があり特別指定でない場合）。

### 4.2 読込結果

- `load_skills_and_needs()` の戻り値は **6要素**:
  `skills_dict, members, equipment_list, req_map, need_rules, surplus_map`
- `req_map` / `surplus_map` は `工程名+機械名` キーに加え、工程名・機械名単独のフォールバックキーを持つ（読込順の注意は実装どおり）。
- **特別指定1〜99** は従来どおり必要人数の上書き。`surplus_map` を特別指定で上書きする仕様は **現状なし**（将来用に `resolve_need_surplus_extra_max` は `task_id` を受け取る）。

### 4.3 解決関数

| 関数 | 役割 |
|------|------|
| `resolve_need_required_op(process, machine_name, task_id, req_map, need_rules)` | 当該タスクの **最低必要人数**（正整数） |
| `resolve_need_surplus_extra_max(process, machine_name, task_id, surplus_map, need_rules)` | **追加で入れられる人数の上限**（0〜50 にクリップ） |

### 4.4 配台ループでのチーム人数

- `req_num` … 行の「必要OP」列があればそれ（1以上）、なければ `resolve_need_required_op`。
- `ignore_need_minimum`（メイン再優先 AI）が真のときは `req_num` を 1 に固定。
- `extra_max = resolve_need_surplus_extra_max(...)`
- `max_team_size = min(req_num + extra_max, len(capable_members))`
- チーム候補は人数 `req_num` 〜 `max_team_size` の **全組合せ**（担当 OP 指名がある場合はその制約付き）。

## 5. 加工時間モデル（増員時の速度）

- **単位あたり実効時間**（概念）:
  - ベース: `base_time_per_unit / 平均効率 / タスク効率`
  - 増員係数: `_surplus_team_time_factor(rq_base, len(team), extra_max)`
- **式の意味**:
  - `rq_base = max(1, req_num)`
  - `extra = max(0, len(team) - rq_base)`、`em = extra_max`
  - `extra <= 0` または `em <= 0` のとき係数 **1.0**
  - それ以外: `frac = min(1, extra / em)` とし  
    `係数 = 1 - SURPLUS_TEAM_MAX_SPEEDUP_RATIO * frac`
- **定数**: `SURPLUS_TEAM_MAX_SPEEDUP_RATIO = 0.05`（**追加枠を使い切っても単位時間が短くなるのは最大約 5%**）。

## 6. チーム選定の優先順位（辞書式最小）

関数 `_team_assignment_sort_tuple(team, team_start, units_today, team_prio_sum)` の戻り値が **小さい**方が採用。

### 6.1 既定（余剰活用優先）

環境変数 `TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF` が **オフでない**とき（既定: オン）:

1. **`-len(team)`**（人数が多いほど有利）
2. `team_start`（早いほど有利）
3. **`-units_today`**（その日に処理できる単位数が多いほど有利）
4. `team_prio_sum`（スキル優先度の合計が小さいほど有利）

### 6.2 従来（開始時刻最優先）

`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF=0`（または `false` / `no` / `off` / `いいえ`）:

1. `team_start`
2. `-units_today`
3. `team_prio_sum`

### 6.3 その他の制約（従来どおり）

- チームに **少なくとも 1 名の OP** が必要（AS のみの組合せは却下）。
- `team_start = max(メンバーの avail_dt)`、設備空き・原反同日・指定開始・マクロ実行時刻下限などは既存ロジック。

## 7. デバッグ・トレース用環境変数

| 変数 | 役割 |
|------|------|
| `DEBUG_TASK_ID` | 既定 `Y3-26`。キュー・日次ループ等のデバッグログ対象依頼NO。 |
| `TRACE_TEAM_ASSIGN_TASK_ID` | 例: `W3-14`。配台ループで **人数別の最良候補** と **採用結果** を INFO ログに詳細出力。 |
| `TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF` | `1`（既定）= 人数優先、`0` 等 = 開始時刻優先。 |

マクロから Python を起動する場合、これらは **起動プロセスの環境変数** に設定する必要があります（未設定だと TRACE ログは出ません）。
