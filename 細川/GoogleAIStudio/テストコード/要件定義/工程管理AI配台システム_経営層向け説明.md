## 1. このシステムは何をするか（一言）

工場の **「いつ・どの機械で・誰が・何をするか」** を、**加工計画データと勤務・スキル情報**から **自動で組み立て**、**日次〜複数日の生産計画表**と**担当者別の予定**を **Microsoft Excel 上**にまとめる仕組みです。現場が慣れ親しんだ表形式のまま、配台（割付）作業の負荷を下げることを狙います。

---

## 2. なぜ必要か（ビジネス上の位置づけ）

| 観点 | 説明 |
|------|------|
| **現場負荷** | 依頼・納期・設備・人の制約が重なると、配台調整は時間がかかり属人的になりやすい。 |
| **変更・人員の急変** | 依頼内容や担当者が**急に変わり再配台が必要**な場面で、手作業に比べ**再配台の難しさと工数の増大を抑え、回避しやすくする**。 |
| **見える化** | 設備ごとの時間割、メンバーごとの予定を一覧化し、調整・説明の材料にする。 |
| **ルールの統一** | 優先度や納期、スキル条件などを、システム側のルールとして繰り返し適用しやすくする。 |
| **例外の扱い** | 「この案件だけ特別に」といった指示を、画面上の備考やメインシートの記載から反映する設計がある（詳細は要件定義書）。 |

## 3. 利用の流れ

現場の配台は、まず **「いま配台すべき仕事を揃える」** 次に **「機械と担当者に振り分ける」** という流れになります。本システムは、この流れを **Excel 上の表** でつなぎ、2つの作業に分けて支援します。

1. **ステップ1 — 未完了の仕事を、配台用の入力表に並べる**  
   加工計画などに書かれた仕事のうち、**まだ終わっていないもの**だけを抜き出し、システムが読み取る **入力表** に並べます。この表が、その日・その期間の **やることリスト** になります。

2. **ステップ2 — 割付案を計算し、計画表として出力する**  
   入力表に加え、**マスタ**（スキル、必要人数、その日の出勤など）を組み合わせて、**どの機械に何をいつ**、**誰にどの仕事を** 割り当てるかを計算します。結果は **生産計画表** と **担当者別の予定** として Excel に出力されます。

**人の判断を挟むとき**は、ステップ1の表に **上書きしたい内容** や **特別な指示** を書き足してからステップ2を実行すると、配台案に反映されます。急ぎのときはステップ1と2を **続けて** 実行してかまいません。

## 4. 「AI」は具体的に何をしているか

ここでいう AI は、**Google が提供するクラウド上の生成AI（Gemini）** を利用します。**インターネット経由**でサービスにアクセスします。

現時点の**試作バージョン**では、**文言の解釈をパラメータ化すること**に課題があるため、**オプション扱い**となっています。

**主に任せていることの例**（人が細かくルールを書かなくても、文章や表の文言から意味を取り出す用途）:

| 対象・領域 | Gemini に任せること（概要） |
|------------|-----------------------------|
| 配台計画上の **備考・特記事項** | セルの文章を解釈し、**解析結果をパラメータ**（構造化された値など）としてプログラムが受け取る |
| **配台不要** としたい工程 | 条件を、機械が処理しやすい形に整理する |
| **勤務や休暇まわりの備考** | 読み取り補助 |
| メインシートの **自然言語による全体指示** | 解釈（例: 特定の案件の扱いを変える） |

**重要**: 本システムでは **Gemini を利用できる状態を前提** に主要機能が設計されています。API キー（利用権限）を用意し、社内の情報セキュリティ方針に沿って管理する必要があります。利用には **通信料に加え、Google の課金・利用枠** が発生し得ます（契約内容に依存）。

## 5. システムの主要パーツと役割

| いま表現しているもの | 実際のイメージ |
|----------------------|----------------|
| **Excel（マクロ付きブック）** | 画面・ボタン・表の置き場。現場が触る主たる場所。 |
| **マスタ用 Excel** | スキルや出勤など、計算の土台となるデータ。 |
| **裏側の計算プログラム** | 表の読み書きと配台ロジックを担当。Excel から自動で起動される。 |
| **出力フォルダ** | 計算結果のファイルやログが保存される場所。 |

**Windows PC と Excel が動く環境**で、初回だけ **環境のセットアップ**が必要、という理解で足ります。

**堅牢性を優先する場合の UI**：**Excel を前面の操作画面にしない**構成への移行もあり得ます（例：**JavaFX**、**Python + Qt** などのデスクトップアプリへ再実装）。**開発コストは増加します。**

## 6. 期待効果と留意点・リスク

### &lt;期待効果&gt;

- 配台表作成の **作業時間短縮**、担当者による **ブレの低減**（ルールの再現性）。
- **納期・優先度・スキル** など複数条件を同時に考慮した案を素早く出せる。
- 出力結果が**慣れ親しんだ Excel**であり、**二次加工が容易**。

### 留意点・リスク（経営・管理側の論点）

| 項目 | 内容 |
|------|------|
| **クラウドAI依存** | Gemini 利用には **ネットワーク** と **契約・キー管理** が必須。止まると設計どおりの機能が使えない。 |
| **コスト** | API の **従量課金・枠** は利用量次第。運用ルール（誰がどれだけ実行するか）の整理が望ましい。 |
| **情報の取り扱い** | 送信・保存の範囲は、**個人情報・顧客情報・機密**の観点で社内規程と照合すること。**通信は通常 TLS（HTTPS）で暗号化**されるが、データは**クラウド上で処理**され、保存・学習利用などの扱いは**プラン・契約で異なる**。無料・有料の別だけで「送信がない」「全面に保護される」とは限らないため、**Google の最新の取り扱いと契約**を確認すること。 |
| **Excel 前提** | 大規模な同時利用やモバイル中心運用には向かない構成。拡張時は Web 化など別アーキテクチャが検討対象になり得る。 |
| **環境の均一性** | PC ごとにセットアップ差があると「動く／動かない」が分かれやすく、**障害の発生も起きやすい**。**配布手順の整備**が重要で、**関係者が協調して**環境をそろえる取り組みが欠かせません。 |

## 7.5 開発経緯と、ソフトウェア委託／内製の判断について

### 当初の想定と、実際に進んだ内容

- 当初は、**骨格を作成したうえで外部業者へ依頼する**方向の想定でした。
- 一方で、**外部委託の方針は確定しておらず**、また**「骨格」の定義**（何を指すか・どこまでが範囲か）も不明瞭でした。

その結果、**特別ルールの指定漏れや不具合を含みつつも、完成形に近いイメージまで本格的にコーディングが進む**形となりました。

### 開発体制とツール選定（Cursor）

**3〜4 か月程度**の期間をいただけるのであれば、AI によるフルバイブコーディングに頼らず、**ロジックを完全に把握したうえでの実装**も選択肢でした。

しかし今回は、**骨格づくりにそれだけの期間を確保できない**前提です。**短期間で開発を進めるため**、**Cursor** を採用する方針をとりました。

**開発ツール**（バイブコーディング）

| 項目 | 内容 |
|------|------|
| **使用ツール** | **Cursor**（AI を統合したコードエディタ）。バイブコーディングで使用。 |
| **サブスク費用の目安** | **Cursor のサブスクリプション**により、おおむね **月額 20〜60 米ドル程度**のランニング（為替・プラン改定により変動し得ます）。 |
| **開発時間をさらに短縮したい場合** | **Pro+**（**月額 60 米ドル程度**）への加入が必要になります。 |

### ソフトウェア会社へ委託する場合の前提

現状、**コードのロジック把握が十分に追いついていない**部分があります。

そのため、ソフトウェア会社へ依頼する場合でも、おおむね次の流れになる見込みです。

- **Cursor で生成・編集した成果物**をベースに、修正・追記しながら要件を固める
- その結果、**要件定義書の整備にも膨大な時間**がかかる

### 当面の見立て（永続性・費用・運用）

**永続性・保守性**の観点では、ソフトウェア会社による**再作成・再実装**が望ましい一方、**かかる費用は大きい**と考えます。

当面は、**このまま開発を完走させ**、**社内の開発リソースを増やす方策を模索しながら運用**するのが現実的ではないか、という見立てです。

### 業務・工場稼働への影響

**本システムが利用できない場合でも、従来どおり手動の配台に戻すだけ**です。

**工場の稼働そのものに致命的な支障が生じるものではない**と考えています。

### 経営判断のお願い

**補助金の枠**がどの程度あるかは、現時点では**不明**です。

上記を踏まえ、**ソフトウェア会社への作成依頼と内製で進めるか**について、**経営判断をいただきたい**と考えています。

| 項目 | 内容 |
