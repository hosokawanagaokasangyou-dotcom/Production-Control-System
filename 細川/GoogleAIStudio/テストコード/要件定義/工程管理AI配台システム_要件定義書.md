# 工程管理AI配台システム 要件定義書

**文書目的**: 本システムの業務ロジック・技術構成・データ仕様を整理し、ソフトウェアの新規開発・保守・移管を依頼する委託先が要件を把握できるようにする。  
**想定読者**: 受託開発会社の PM / アーキテクト / 実装担当  
**前提**: 現行は **Microsoft Excel（.xlsm）＋ローカル Python 3** によるハイブリッド構成。仕様の一次情報はリポジトリ内の `python/planning_core.py` および `生産管理_AI配台テスト_xlsm_VBA.txt` に存在する。**Google Gemini API は本システムの要件として必須**（運用上「設定」B1 に有効な認証を常に配置すること。未設定時は実装上フォールバックする箇所があるが、設計された機能は成立しない）。  
**文書バージョン**: 0.4　**最終更新**: 2026-03-28

---

## 1. システム概要

### 1.1 業務目的

工場の **加工計画（タスク）** に対し、**設備（工程・機械）** と **担当オペレータ（OP）** の割付（配台）を自動計算し、日次〜複数日の **生産計画表** および **メンバー別スケジュール** を Excel で出力する。

### 1.2 構成要素

| 要素 | 役割 |
|------|------|
| **生産管理_AI配台テスト.xlsm** | 操作 UI・マクロ・設定シート・結果取り込み・LOG 表示。Python 起動の親プロセス。 |
| **master.xlsm** | スキル・必要人数（need）・勤怠カレンダー由来の出勤データ等のマスタ。`planning_core` が参照。 |
| **Python（planning_core 中心）** | 段階1・段階2の計算コア、openpyxl による xlsx 生成、Gemini API 連携（必須）。 |
| **output/** | 段階1中間成果物、段階2の日付別成果 xlsx、ログ。 |

### 1.3 処理の二段階モデル

| 段階 | トリガ | 概要 |
|------|--------|------|
| **段階1（タスク抽出）** | VBA → `task_extract_stage1.py` | マクロブックの「加工計画DATA」から未完了タスクを抽出し、`output/plan_input_tasks.xlsx` を生成。VBA が「配台計画_タスク入力」シートへ取り込み。 |
| **段階2（計画生成）** | VBA → `plan_simulation_stage2.py` | 「配台計画_タスク入力」を読み、マスタ勤怠・スキルに基づき配台シミュレーション。成果 xlsx を生成し、VBA がマクロブックへシート取り込み。 |

ユーザーは段階1後に **配台計画シート上で上書き列・特別指定** を編集し、段階2で反映させる運用を想定する。段階1→段階2の **連続実行** も可能。

## 2. 機能要件

### 2.1 段階1（タスク抽出）

**入力**

- マクロブック内シート **「加工計画DATA」**（列定義は `planning_core` の `TASK_*` / `SOURCE_BASE_COLUMNS` と一致が必要）。

**抽出ロジック（要旨）**

- **完了扱い** の行は除外（完了キーワード・実出来高等による換算残量が 0 など）。
- 残 **換算数量**（総量 − 完了相当）が正で、**依頼NO**・**工程名** が有効な行のみ対象。
- **「配台計画_タスク入力」** 向け列集合にマッピング（上書き用空列を含む）。
- **参照列「（元）…」** はマスタ need 等から再計算（`planning_core._refresh_plan_reference_columns`）。
- **設定_配台不要工程** シートのルールにより、条件に合致する行へ **配台不要** を自動設定可能（AI・JSON 式評価を含む）。
- 同一依頼NOで同一機械に複数行ある場合の **分割行** に対する **配台不要自動設定**（` _apply_auto_exclude_bunkatsu_duplicate_machine`）がある。

**出力**

- `output/plan_input_tasks.xlsx`、シート名 **「タスク一覧」**。
- 日付列の表示形式、上書き列の視覚的ハイライト（黄）等の後処理あり。

**VBA 側**

- 環境変数 **`TASK_INPUT_WORKBOOK`** にマクロブックのフルパスを設定のうえ Python 実行。
- 生成 xlsx を **「配台計画_タスク入力」** に取り込み、既存体裁フォントの退避・復元、クエリ更新（RefreshAll）等。

### 2.2 段階2（配台計画シミュレーション）

**入力**

- **「配台計画_タスク入力」**（列順・列名は `plan_input_sheet_column_order()` と一致）。
- **master.xlsm**: skills / need / 勤怠（メンバー別出勤・休憩・休暇区分・備考等）。
- **マクロブック「メイン」シート**（名前解決は VBA/Python で「メイン」「Main」「メイン」を含むシート名）: **グローバルコメント**・**再優先特別記載** 等、計画全体への自然言語指示（Gemini 解析）のソースになり得る。

**計画基準日時**

- 原則 **「加工計画DATA」のデータ抽出日**（列 `データ抽出日`）。無い場合は実行時刻にフォールバック。

**勤怠・稼働**

- カレンダー／出勤簿ロジックに基づき、日ごと・メンバーごとに **稼働可否**、**始終業**、**休憩帯**、**中抜け**（AI または文言推定）を構造化。
- メインのコメントから **工場休業日** を解析し、勤怠に反映する処理あり。

**配台アルゴリズム（要旨）**

- タスクを **優先度・納期・進行状況・特別指定（AI 解析含む）** でキュー化（`build_task_queue_from_planning_df`）。
- **日付ループ** で、設備（機械）ごとにタイムラインを更新しつつ OP を割当（チーム編成・need の必要人数・余剰人員の扱い等）。
- **休憩を除いた実働分** で加工時間を換算（`get_actual_work_minutes` / `calculate_end_time`）。
- **設備ガント**、**設備毎時間割**、**タスク一覧**、**メンバー別** 等の結果シート用データを生成し、**日付階層の output フォルダ** に xlsx 保存。
- 矛盾（特別指定と AI 解析の不整合等）を検出し、可能なら **Excel 上の配台シートに視覚的ハイライト**。ブックを Excel で開いたままだと Python からファイルを直接保存できないことがあるため、そのときは **`log\planning_conflict_highlight.tsv` などの補助ファイル（タブ区切り）に内容を書き出し**、段階2終了後に VBA がそのファイルを読んで **開いているシートへセルを塗る**（`ApplyPlanningConflictHighlightSidecar`）。

#### タスクの優先順位（`task_queue` の並び＝同一日内の割付検討順）

段階2では `generate_plan` 内で `task_queue.sort(...)` により **先頭に近いタスクほど同条件で先に機械・OP を占有しやすい**。タプルは **辞書式順（小さい方が先）**。実装キー（`planning_core.generate_plan`）の要約は次のとおり。

1. **納期ソースの強さ**（`due_source_rank`）：**指定納期_上書き（セル）** → **特別指定備考 AI の目標納期系** → **AI の開始日（締め扱い）** → **回答納期** → **指定納期** → なし（最大ランク）。数値が小さいほど先。
2. **AI／上書きによる「完了期限系」の指定**（`has_done_deadline_override`）：**ある**行を **ない**行より先。
3. **加工途中**（実績数量あり、`in_progress`）：**ある**方を先。
4. **配台計画の「優先度」列**（`PLAN_COL_PRIORITY`）：**数値が小さいほど高優先**。未入力は実装上 **999** 扱い。
5. **当日基準日以前が納期根拠**（`due_urgent`）：**緊急**（納期根拠日が基準日以前）を先。
6. **納期根拠日**（`due_basis_date`）：**早い**日付を先。
7. **開始可能日**（`start_date_req`）：**早い**日付を先。
8. **依頼NO のタイブレーク**（`_task_id_priority_key`）：同一プレフィックスなら **ハイフン後の数値が小さい** 方を先（例: Y3-24 を Y3-34 より先）。
9. **必要人数（need／行上書き解決後）**：**多い**方を先（実装では人数にマイナスを付与してソート）。

補足：**メインシート「再優先特別記載」**（Gemini 解釈）により、スキル無視・チーム1名固定・配台制約の撤廃など **キュー以前のグローバル上書き** が入る場合がある。

#### 配台の優先順位（チーム候補・スキルセル）

- **チーム編成の候補同士の優劣**（`_team_assignment_sort_tuple`）：環境変数 **`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF`（既定 1＝有効）** のとき、**(1) 人数が多いチーム** → **(2) チーム全体の作業開始が早い** → **(3) その日にこなす単位数が多い** → **(4) メンバーごとの skills 交差セル優先度の合計が小さい** の順で優先。無効時は **開始が早い** を最優先とし、その後 **単位数**・**スキル優先度合計**。
- **master.xlsm「skills」交差セル**（`parse_op_as_skill_cell`）：**OP／AS の後に付く整数が小さいほど、その設備への割当で先に選ばれる**。数字省略の `OP`／`AS` は優先度 **1**。OP/AS で始まらないセルは当該組み合わせではスキルなし扱い。

**出力ファイル（代表）**

- `production_plan_multi_day_*.xlsx`（結果_* シート群・列設定シート等）。
- `member_schedule_*.xlsx`（メンバー名シート → VBA で **個人_*** にリネーム取り込み）。

**VBA 側**

- 上記 xlsx を開き、既存 **結果_*** / **列設定_結果_タスク一覧** を削除してからコピー。
- 列幅・罫線・見出し色・**メイン** の結果リンク・メンバー一覧・出勤表示の更新。
- 段階1/2 成功後 **全シート UsedRange のフォントを BIZ UDPゴシックに統一**（自動）する仕様あり。

### 2.3 Google Gemini API 連携（**必須要件**）

本システムの **要件として Gemini API を利用可能な状態にすること** を求める。認証未設定時、実装は一部処理をスキップするが、**特別指定備考の構造化・配台不要ルールの JSON 化・勤怠解釈・メイン「再優先特別記載」の解釈・工場休業日の抽出** 等、設計上の主機能が期待どおり動かないため、**本番・検収環境では API キー（または暗号化認証 JSON）の配置を必須**とする。

- **主な用途**: 配台計画 **特別指定_備考** の解析、**設定_配台不要工程** の日本語ロジック→JSON、**勤怠備考・結果カレンダー** 周りの解釈、**メイン「再優先特別記載」** の自然言語→配台フラグ（スキル無視・1名チーム・制約撤廃・依頼NO別担当指名など）、その他計画補助。
- **認証**: マクロブック **「設定」B1** に Gemini 認証 JSON のパス（平文または暗号化形式）。復号パスフレーズは **Python ソース内定数**（セキュリティ方針としてシートに書かない）。
- **依存パッケージ**: `google-genai`（`requirements.txt`）。
- **運用**: API 利用状況のサマリを **メイン P 列** 等へ反映する処理あり。料金・クォータは別途運用設計すること。

### 2.4 その他マクロ機能（UI・運用）

- **環境構築**: `python/setup_environment.py` 実行（pip / requirements / xlwings アドイン）。ブック直下ではなく **`python\` 配下** を優先参照。
- **全シートフォント統一**（リスト選択 / 手入力 / BIZ UDP 固定）。
- **設定_配台不要工程** シートの確保・Power Query 更新・TSV フォールバック同期。
- **master.xlsm を開く** ボタン配置用マクロ。
- master 側（別ファイル `master_xlsm_VBA.txt`）には **カレンダーからメンバー出勤簿生成** 等、マスタ整備用マクロが含まれる。

### 2.5 Python 実行環境・必要コンポーネント（**xlwings 含む**）

#### ランタイム・起動方法

- **OS**: Windows（Excel デスクトップとセット）。
- **Python**: 3 系（VBA「環境構築」は **Python 3.12** の winget / 公式インストーラ URL を想定）。コマンドは **`py -3`**（Python Launcher）を前提にマクロから `cmd.exe` 経由で起動する。
- **カレントディレクトリ**: `planning_core` はモジュール読込時に **テストコード直下**（`python\` の親）へ `chdir` する実装。ブック・`master.xlsm`・`output\`・`log\` と同じ階層に `python\` を置くこと。

#### pip パッケージ（`python/requirements.txt`）

| パッケージ | バージョン目安 | 役割 |
|------------|----------------|------|
| **pandas** | ≥2.0 | タスク DataFrame、`read_excel` 等による表データ処理。 |
| **openpyxl** | ≥3.1 | **主経路**：閉じた xlsx/xlsm の読み書き、スタイル・シート生成（段階1/2 成果物、設定シート保守の保存試行など）。 |
| **xlwings** | ≥0.30 | **環境標準コンポーネント**として `requirements.txt` に含める。下記「xlwings アドイン」とセットで導入する。 |
| **google-genai** | ≥1.0 | Google **Gemini API** クライアント（必須要件に連動）。 |
| **cryptography** | ≥42.0 | Gemini 認証 JSON の **暗号化形式**の復号、`encrypt_gemini_credentials.py` 等。 |

#### xlwings（パッケージ＋Excel アドイン）— 要件としての位置づけ

- **pip の `xlwings`**: 上表のとおり必ずインストール対象に含める。
- **Excel アドイン（`xlwings.xlam`）**: `python/setup_environment.py` の最後で **`xlwings addin install`** を実行し、ユーザーの Excel スタートアップ（通常は `%APPDATA%\Microsoft\Excel\XLSTART` 等）へ配置する。マクロ **「環境構築」（`InstallComponents`）** も同スクリプトを呼び出すため、**初回セットアップ完了＝アドイン配置まで含む**。
- **現行アーキテクチャとの関係**:
  - **段階1・段階2の本体**は VBA が **`WScript.Shell` で `py -3 -u python\task_extract_stage1.py` / `plan_simulation_stage2.py` を子プロセス起動**する方式であり、**計算ループ自体は xlwings のランタイム API に依存しない**（コメント上も planning_core 側は主に openpyxl）。
  - それでも **xlwings は「Excel と Python を公式に橋渡しする標準スタック」として requirements に含め、アドインを配布物と同等に扱う**。将来の **RunPython / UDF / リボン連携** や、リポジトリ内 **検証用スクリプト**（COM/xlwings 系テスト）との整合のため。
- **運用**: アドインインストール後は **Excel の一度の終了・再起動**でリボン反映を確認すること。信頼できる場所にブックを置く・マクロ有効化は Excel セキュリティ設定に従う。

#### pywin32（COM）— 明示されていないが実装で使用

- `requirements.txt` には **列挙されていない**が、`planning_core` は **Excel で対象ブックを開いたまま**のとき、**`win32com.client`（pywin32）** でブックに接続する **COM フォールバック経路**を持つ。
- **xlwings を pip インストールすると pywin32 が依存として入る**ことが多く、その場合 COM 経路も利用可能になりやすい。pywin32 が無い環境では当該 COM 経路のみ使えず、**openpyxl でブックを閉じた状態から保存する**か、**補助 TSV＋マクロ反映**に切り替わる。**補助 TSV**とは、保存や COM が使えないとき Python が `log\` 等へ出力する **タブ区切りテキスト**であり、VBA（例: `設定_配台不要工程_AからE_TSVから反映`）が読み込んで **Excel 上のセルへ書き戻す**仕組みである（英語圏の「sidecar＝本編の横に付く別ファイル」という意味でコード内に使われることがある）。

#### 初回セットアップ手順（要点）

1. マクロブックを保存可能なフォルダに配置し、**`python\`** に `planning_core.py`・`requirements.txt`・`setup_environment.py`・エントリ `.py` を配置。  
2. マクロ **「環境構築」**（パスワード確認後 `InstallComponents`）またはコマンドプロンプトで  
   `cd /d <テストコード直下>` → **`py -3 python\setup_environment.py`**  
3. 処理内容: **pip 自身の更新** → **`pip install -r python\requirements.txt`** → **`xlwings addin install`**。  
4. **HTTPS**: Gemini 利用のためインターネット接続・プロキシ設定が必要な場合あり。

## 3. データ要件（シート・列の整合）

委託先は **Excel 見出し文字列と Python 定数の 1:1 一致** を維持することが必須。主要な定数名は `planning_core.py` 先頭付近の `TASK_*` / `PLAN_*` / `ACT_*` / `EXCLUDE_RULE_*` / `RESULT_*` 等。

**特に重要なシート名（例）**

- 加工計画DATA、配台計画_タスク入力、設定、設定_配台不要工程、LOG、メイン、結果_*、個人_*、列設定_結果_タスク一覧、結果_カレンダー(出勤簿) 等。

**設定シート（マクロブック）**

- **B1**: **Gemini 認証 JSON パス（必須運用）** — 本番では空にしないこと。  
- **B4 / B5**: Python 生成 xlsx に埋め込むフォント名・サイズ（B4 空のときは書体名を openpyxl で付与しない設計）

## 4. 非機能要件

| 区分 | 内容 |
|------|------|
| **実行環境** | Windows、Excel デスクトップ、Python 3（`py -3` ランチャー想定）。**§2.5** の pip 一式・**xlwings アドイン**を含む。 |
| **パス** | マクロブックと **同一フォルダ** に `python\`、`output\`、`log\`、`master.xlsm`（または `planning_core.MASTER_FILE` で指定）を配置する運用。UNC パスでは `pushd` 利用等の対策あり。 |
| **ログ** | `log/execution_log.txt`、ブック内 **LOG** シート。 |
| **同時実行** | Excel でブックを開きっぱなしにすると openpyxl のファイル上書き保存が失敗することがある。その場合は **COM（pywin32）で Excel 経由保存**を試すか、**`log\` の補助 TSV を VBA が読んでセルへ反映**する経路に切り替わる（§2.2・§2.5）。 |
| **セキュリティ** | Gemini キーは JSON・暗号化ファイルで管理。ソースに埋め込まれた復号用定数の取り扱いは契約で規定すること。 |
| **外部 API** | **Google Gemini API は必須**（本番・検収）。ネットワーク出口・プロキシ・キーローテーションは運用で確保する。 |

## 5. 外部インタフェース

- **環境変数（プロセス）**: `TASK_INPUT_WORKBOOK`（必須）、その他 `planning_core` 内 `os.environ` 参照のデバッグ用変数（例: `TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF` でチーム優先モード切替）。
- **ファイル I/O**: 入力 xlsm/xlsx、出力 xlsx、ログ、オプションで `json\` や `API_Payment\` 等（既存コード参照）。
- **Google Gemini API（必須）**: HTTPS（`google-genai`）。**設定 B1 未設定時は設計機能が欠落する**ため、システム要件として常時利用可能とする。

## 6. 移管・再実装時の注意（委託先向け）

1. **単一巨大モジュール** `planning_core.py` に業務ロジックが集中している。リファクタ時は段階1/2境界・`generate_plan` 内日付ループ・タスクキュー・設備タイムラインを分解単位の候補とする。  
2. **VBA と Python の契約**（シート名・列名・環境変数・出力ファイル命名・取り込み順）を変更する場合は **双方を同時改修** すること。  
3. **Excel 依存**を減らし Web/API 化する場合、**配台コア**をサービス化し、Excel はクライアントに退ける案が一般的。  
4. 現行の **フォント・罫線・メインシート UI** は VBA で強く依存しているため、UI フレームワーク変更時は同等の運用要件（リンク、メンバー表、LOG）を要件に明記すること。
