# 配台ルール（現行仕様）

実装の正は `python/planning_core/_core.py` の `generate_plan`・`build_task_queue_from_planning_df`・日次割付ループおよび周辺関数です。本書はその挙動を業務向けに要約したものです。

**開発運用**: 配台ロジックをソースで変更したときは、**本書（`配台ルール.md`）と `配台ルール.html` を必ず同内容に更新する**（Cursor は `.cursorrules` および `.cursor/rules/md-paired-html-sync.mdc` を参照）。

## 改定（2026-04）休憩・終業直前の配台

- **終業直前**: 業務上の意味は次のとおり。**退勤上限までの残りが `ASSIGN_END_OF_DAY_DEFER_MINUTES` 分以下**かつ、**`ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS` ロール分以上はその日に回せない**ときは、**新たに加工を始めない**（既存の連続ブロックの途中表示は別）。
  - **`ASSIGN_END_OF_DAY_DEFER_MINUTES`**: **Python 既定 45（分）**（環境変数・`設定_環境変数` の B 列が未設定のとき）。**`0` を明示**すると終業直前の下記ロジックは**無効**。
  - **小残ロール（依頼全体）**: ウィンドウ内でタスクの **`remaining_units` 切り上げ** が **`ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS`（既定 5）以下**のとき、その日の開始不可（`_defer_team_start_past_prebreak_and_end_of_day` が `None`）。
  - **閾値ロール数以上回せない場合**: 同じウィンドウ内で、試行開始から終業まで **`calculate_end_time` の実働換算**で収容できるロール数（レガシーはその日の割当ブロックの `units_today`＝`min(収容上限, 残ロール切り上げ)`、試行順1ロールは終業までの収容上限そのもの）が **`ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS` 未満**のとき候補却下（`_eod_reject_capacity_units_below_threshold`）。いわゆる「閾値ロール数以上回せないので新規開始しない」と実装上一致する。
  - **同一依頼の連続ロール（実装）**: 設備占有キー（`machine_occupancy_key`）ごとの直前加工の依頼NO（`machine_handoff["last_tid"]`）が、いま割り当てようとしている行の依頼NOと**一致するとき**は、上記の**小残による当日不可**および**収容閾値未満の候補却下**を**行わない**（`_eod_same_request_continuation_exempt`）。**依頼が切り替わる**／**その日その設備で最初の加工**のときは従来どおり終業直前デファーを適用する。
  - **補足**: 全残ロールで終業直前を止めたい場合は `ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS` を十分大きくする（例: 999999）。**既存ブック**で `設定_環境変数` に **`ASSIGN_END_OF_DAY_DEFER_MINUTES` が 0** の行だけが残っている場合は、B 列を **45 等**に直すか、行を削除して VBA の不足追記で**新既定**を取り込んでください。
- **設備時間割の進度**（例 `14/54R`）: **累積完了ロール / そのタスク総ロール**であり、**左側は残りロール数ではない**（残りは総量−完了相当。終業直前の残ロール判定は **`remaining_units` の切り上げ**）。
- **休憩**: 試行開始が **休憩帯内**のときは **`_defer_team_start_past_prebreak_and_end_of_day` が休憩終了時刻へ繰り下げ**（設備下限・`avail_dt` 等は `refloor_fn` で再適用）。**帯外でも**、当該ブロックに必要な **連続実働分数**（試行順は 1 ロール換算の切り上げ分、レガシーは同趣旨）が **次の休憩開始まで足りない**ときは、**その休憩の終了まで同様に繰り下げ**（午後休憩の直前にだけ載せられず、その後が空になる事象の防止）。繰り下げ後に終業超過・終業直前デファーに該当すれば `None`。**1 回の割当ブロック**が、繰り下げ後も **次の休憩開始または終業までの連続実働分数**に収まらないときは候補却下（`_contiguous_work_minutes_until_next_break_or_limit`）。`calculate_end_time` は休憩を跨いだ実働換算に使うが、**上記で休憩またぎの連続配台は通さない**。
- **廃止（無視）**: 環境変数 **`ASSIGN_DEFER_MIN_REMAINING_ROLLS`**（旧・同日 defer しきい値）と **`ASSIGN_PRE_BREAK_DEFER_GAP_MINUTES`**（旧・休憩直前ギャップ）は **読まない**。設定シートに残っていても無害。

### 2. 順次配台・日内フロー（業務案と実装の対応）

- **業務案（未実装）**: 「納期が近い1件ずつだけ配台し、超過時に都度計画基準+1」型のメインループは、現行の **全日程×日次マルチパス**構造とは別のため **未実装**です。
- **実装されている日内配台**: 環境変数 **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST`**（既定 `1`）で、**`_trial_order_first_schedule_pass`** が有効なとき、当日候補を **配台試行順（`dispatch_trial_order`）の昇順**で回し、**試行順が先でもその日入らなければ同じフェーズ内で次の試行順へ進む**（§B-2 以外は同一タスクについて**その日入る限り連続ロール**）。**§B-2 は完全二相**: **フェーズ1**で熱融着**検査行以外**を詰め、**フェーズ2**は **同一依頼の EC が全日で完走した後**にのみ検査が候補化（**前進配台では EC に 1 ロールでも残がある間は検査を載せない**。翌稼働日へ進み EC だけを繰り返す）。交互にすると EC 担当が検査へ回り **EC がブロック**されやすいため、フェーズ1で EC 等を先に詰める。通常の日次ループ完了後、**EC 側が完走済みの B-2 検査だけ**を日付先頭から **`_run_b2_inspection_rewind_pass`** で再走査し、`timeline_events` を人・設備のブロックテーブルとして **日跨ぎ占有を保持したまま検査を詰める**。再走査中の検査は **`_roll_pipeline_inspection_assign_room`** と **`_roll_pipeline_b2_inspection_ec_completion_floor_dt`** で制御。`0` にすると従来の **`_day_schedule_task_sort_key` ソート順**の一日内ループにフォールバックします。
- **依頼NO直列**: 環境変数 **`STAGE2_SERIAL_DISPATCH_BY_TASK_ID=1`** のときのみ、**残がある最も早い依頼NO**（処理順リスト内）の行だけが当日候補になり、該当依頼が詰まると他依頼は進みません（既定は無効）。**処理順リスト**は、各依頼NOについて **配台試行順番（`dispatch_trial_order`）の最小値**が小さい依頼を先に並べ、タイブレークは **計画シート読込で当該依頼の先頭行に近い `planning_sheet_row_seq`** と **依頼NOキー**（`_serial_dispatch_order_task_ids`）。
- **配台計画シートの「配台試行順番」列**: **列順の先頭**（`plan_input_sheet_column_order`）。**段階1**では、加工計画DATA からの抽出直後（`_merge_plan_sheet_user_overrides` の前）に当列を**空にクリア**し、既存「配台計画_タスク入力」からは**引き継がない**（`PLAN_STAGE1_MERGE_EXTRA_COLUMNS` は **ロール単位長さ** のみ）。続けてマスタ need・グローバルコメント解析を含め **段階2 冒頭と同じ** `_generate_plan_task_queue_sort_key` → **`_reorder_task_queue_b2_ec_inspection_consecutive`** → **`_assign_sequential_dispatch_trial_order`** で **1..n** を付与（`fill_plan_dispatch_trial_order_column_stage1`）。**段階1の Excel 出力直前**に `_sort_stage1_plan_df_by_dispatch_trial_order_asc` で**当列の昇順**に行を並べ替える（欠損・非正整数は末尾、同キーは元の行順）。**段階2**では、キューに載る **全行**についてシート上の **配台試行順番が正の整数で埋まっている**ときだけ、その数値順（同値は `planning_sheet_row_seq`）でキューを並べ替え **`dispatch_trial_order` にそのまま反映**し、**§B-2/3 の EC 隣接繰り上げは行いません**（手編集用）。**1行でも欠損**があるときは従来どおり **自動計算**（隣接化＋連番）にフォールバックします。
- **納期基準超過の巻き戻し再試行**（セクション **2.3**）は **既定オフ**です。`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=True` のときのみ `_collect_task_ids_missed_deadline_after_day` トリガで実行されます。

### 3. 納期基準日当日の完了時刻 16:00（参照定数）

定数 **`PLAN_DUE_DAY_COMPLETION_TIME`（16:00）** を `planning_core/_core.py` に定義済みです。**割付終了時刻への組み込みは今後の拡張**です（現状はシフト終了・既定 17:00 枠など従来ロジック）。

### 4. 同一納期基準内のタイブレーク（依頼NO）

グローバルキュー（§2.0）では、納期基準（`due_basis_date`）が同じ帯のあと **need シート左列優先**、さらに **依頼NO** でタイブレークします（機械名だけの NFKC 正規化キーはグローバルソートに使いません）。

- 通常の依頼NO（例 `Y3-24`）: **最後の `-` 以降が数値**ならその整数が小さいほど先。
- **ハイフンが2つ以上**（例 `Y4-1-1`）: **最初の `-` の直後の区切りの数値**を比較（例では **1**）。

### 5. 同一依頼NO・異なる工程×機械の順序

**加工内容**をカンマで分割した工程順と **工程名** の照合は従来どおり **§A-1**（`process_sequence_rank`・`_task_blocked_by_same_request_dependency`）です。

## 適用の優先順位

矛盾があるときは次のとおりです（上ほど強い）。

**ルール群の階層（明示）**

① **§B（個別ルール群）**は **§A（全依頼ルール群）より優先**します。§B と §A が食い違うときは、該当する依頼・工程・機械の組み合わせでは **§B を採用**します。  
② **§B 群内**の適用の強さは **`B-1` を最優先**とし、続く **`B-2` と `B-3` は同一の `b_tier = 1` 帯**（EC 先行と未着手の検査／巻返し）でまとめます。番号はルールの別名付けです。実装ではグローバル `task_queue.sort` で **加工途中 → 納期基準（`due_basis_date`）** を見たうえで **`b_tier` により B-1 → B-2/B-3 帯 → その他**の順になります。

**本文セクションとの対応**

1. **§B 個別に指定する追加ルール**（例: B-1 検査×熱融着機湖南の加工途中と試行順、B-2 EC→検査ロールパイプライン、B-3 EC→巻返しロールパイプライン）
2. **§A 全加工依頼に適用する追加ルール**（例: A-1 加工内容の工程順）
3. **本文の番号付き一般ルール**（「1. 配台の対象外」以降）

**§B-1** は **検査×熱融着機湖南かつ加工途中**の行に対し、**配台試行順**（グローバルでは §2.0 のキーで決まったうえで付与される `dispatch_trial_order`、日内 **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0`** 時は `dispatch_trial_order` を rank より前）を **§A-1 の rank より先に**効かせます（下記 §B-1・§2.0）。**§B-2 / §B-3** は **ロールパイプライン定数・ロール枠・（既定の）設備占有・キュー／日内ソートでの後続前倒し** 等で制御します（B-3 は B-2 と同趣旨で、後続工程が **巻返し×EC機　湖南**）。**§A-1 は B-2 検査行・B-3 巻返し行にも適用されうる**ため、実際に後続が候補に残るかは **§B-2.6** の **`_trial_order_flow_eligible_tasks` チェーン**（§A ブロック → 枠・占有・試行順）で決まります。**§B 対象外**は **§A-1 / §A-2** を従来どおり適用し、**§B-2/§B-3** 対象の未着手後続行はグローバルキューで **`b_tier = 1`** により §A より先の帯に並びます。

## A. 全加工依頼に適用する追加ルール

### A-1 「加工内容」列に基づく工程順序（同一依頼NO）

**データ**: マクロブック **「加工計画DATA」** および段階1経由の **「配台計画_タスク入力」** に列 **`加工内容`** がある場合に有効です（段階1で `SOURCE_BASE_COLUMNS` に含まれるため、抽出データに引き継がれます）。

**ルール**:

- 同一 **依頼NO** で複数行（複数工程）があり、いずれも **配台不要ではない** とき、**加工内容** の文字列を **カンマ `,` 区切り**で分割した順序が **工程の正しい順序**です（先頭が最初の工程）。
- 例: `EC,検査` → **先行工程に残ロールがある間、後工程には `_task_blocked_by_same_request_dependency` により割付を試みません**（§A-1）。  
  **§B-2**（`EC`→`検査`×熱融着機　湖南）および **§B-3**（`EC`→`巻返し`×EC機　湖南）の後続行は、**`_trial_order_flow_eligible_tasks`** 内で **`_roll_pipeline_inspection_assign_room`**・**設備占有** などにより制御され、§A より上位のフィルタチェーンは **B-2.6** 参照（§B が本文一般ルールより優先）。
- 各行の **工程名** が分割後のどの要素と一致するかで順序番号を決めます（NFKC・空白除去で照合）。**加工内容に現れない工程名**の行には、この順序制約はかかりません（ただし **結果シートでは不整合としてタスクIDを赤表示**します。下記「9.」）。
- **配台不要**の行はキューに入らないため、ブロッカーにもなりません。

**実装上の補足**: 既定の日内フロー（**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`**）では **配台試行順**を軸にマルチパスし、§B-2 は **完全二相**（先に EC 等、次に検査）です。検査候補化は **ロール枠**（`_roll_pipeline_inspection_assign_room`）で制御します。**リワインド**で検査を載せるときは **ロール数枠**と **時刻下限**（`_roll_pipeline_b2_inspection_ec_completion_floor_dt`）で制御します（§2.0.2・B-2.2）。旧フロー（`0`）では **§2.0.1** の複合キー順でループします。

### A-2 同一依頼NOの直列化（rank が無い行・加工内容に載らない工程）

**目的**: 同一 **依頼NO** で **異なる工程（別行）** は、**別設備でも同時に加工しない**（現場の流れを1本化）。

**データ**: 段階1→配台計画シートを **上から**読み込んだときの **同一依頼NO内の行番号**（`same_request_line_seq`）。

**ルール**:

- **§A-1** で両行に `process_sequence_rank` が付くときは **rank のみ**で前後を判定（本節は追加判定しない）。
- **どちらか一方でも rank が無い**ときは、**シート上で上にある行**（`same_request_line_seq` が小さい方）の **残ロールがゼロになるまで**、下の行には割付を試みません。
- **§B-2**（上記 EC×検査）は **`_task_blocked_by_same_request_dependency` を特別に抜ける実装ではなく**、**B-2.6** のフィルタ（**ロール枠 `assign_room`**・設備占有等）で前進／リワインドの挙動が決まります。

**実装**: `planning_core/_core.py` の `build_task_queue_from_planning_df`（`same_request_line_seq` 付与）と `_task_blocked_by_same_request_dependency`。

## B. 個別に指定する追加ルール

**§B は「配台ルール（現行仕様）」本文の一般ルール（§1 以降）より優先**します。§A と食い違う場合も、**§B に該当する依頼・工程の組み合わせでは §B を採用**します（上記「適用の優先順位」参照）。

**ブロック構成**: 今後 **`B-1`, `B-2`, …** とルールを追加する想定です。各 **`B-n` を独立したブロック**（対象・ルール・§A との関係・定数）として記述します。

### B-1 — 検査（熱融着機　湖南）かつ加工途中のとき、配台試行順を優先

**対象**: **§B-2.1 の後続行と同一**（工程名 **`検査`**、機械名 **`熱融着機　湖南`**）かつ **`in_progress`（加工途中・実績あり）** が真のタスク。

**ルール**

- **グローバル** `task_queue.sort`（`_generate_plan_task_queue_sort_key`）: **`b_tier = 0`**（§B-1）。対象行は常に加工途中のため **§2.0 の先頭キーでは一般行と同様に先に並び**、**同一加工途中・同一納期基準（`due_basis_date`）**では **`b_tier` により** B-2/B-3 帯より先です。対象行同士のタイブレークは **§2.0** のとおり **`b2_queue_sub`（B-1 同士では常に同値）→ need 列位置 → 依頼NO** です（`planning_sheet_row_seq` はグローバルキーに含めません）。**`dispatch_trial_order` はソート＋§B-2 隣接調整の直後に 1..n で付与**されます。
- **日内** `_day_schedule_task_sort_key`: **§2.0 と同趣旨の先頭キー**のあと、対象行は **`dispatch_trial_order`（昇順）を `process_sequence_rank`（§A-1）より前**に置きます。**§B-2 帯の並び（`b2_roll_pipeline_stage`）より前**のキー列に置くため、**加工途中の検査（§B-1）**では試行順が **EC 先行の §B-2 段**よりも先に効きます（§2.0.1）。
- **§B 群内の順位**: **B-1**（本節）を **B-2**（ロールパイプライン）より **常に先**にソートキーへ反映しています。

**実装**: `planning_core/_core.py` の `_generate_plan_task_queue_sort_key`、`_day_schedule_task_sort_key`。

### B-2 — EC（EC機　湖南）→ 検査（熱融着機　湖南）ロールパイプライン

**ロジック（方針）**

1. **先行の EC を優先**して全日で進め、通常日次ループ完了後に **検査のみ日付先頭から再走査（リワインド）** して配台します。再走査時は `timeline_events` をブロックテーブルとして使用し、EC で既に埋めた人・設備の時間帯を保持します。
2. **§B 群内**では **§B-1**（加工途中の熱融着検査と試行順）に次いで、**EC 行と未着手の検査行**を **§A より先の同一帯**（`b_tier = 1`）にまとめ、**同一帯内では EC を検査より先**（`b2_queue_sub` / 日内は `b2_roll_pipeline_stage`）に並べます。ロール枠・バッファは B-2.2 の式に従います。

#### B-2.1 対象（コード定数・同一依頼NOの行同士のみ）

| 段階 | 工程名 | 機械名 |
|------|--------|--------|
| 先行 | `EC` | `EC機　湖南` |
| 後続 | `検査` | `熱融着機　湖南` |

#### B-2.2 ルール（ロールパイプライン・前進とリワインド）

- **ロール単位**でパイプラインします。EC 側の **完了ロール数**を `ec_done`、検査側を `insp_done` とします（計画上の残量から算出）。
- **前進配台（日次ループ）**: `roll_pipeline_inspection` 行は **`_roll_pipeline_inspection_assign_room` のロール枠があるとき**に候補化します（`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0` の経路でも同趣旨）。`ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2` のため、EC 完了が 2 ロール未満の間は検査枠が 0 になります。
- **段取りバッファ**（`ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2`）と **`_roll_pipeline_inspection_assign_room`** の式は、**前進・リワインドの両方**で「検査を何ロール進めてよいか」と **EC ロール終了時刻下限**（`_roll_pipeline_b2_inspection_ec_completion_floor_dt`）を整合させるために使います（§B-2.3 参照）。
- **同一依頼のキューに EC 行が無い**検査行は、ロール枠を事実上かけない（`ROLL_PIPELINE_INSP_UNCAPPED_ROOM`）。`need`・スキルに従い通常配台します（EC 不在で永久スキップしない）。
- **上限（`assign_room`）の意味合い**: **EC 全ロール完了前**は `max(0, ec_done - 2 + 1) - insp_done` 型の枠。**EC 全ロール完了後**は、上記だけでは検査の許容本数が `ec_done` より 1 本少なくなり最終ロールが詰まるため、上限を **`ec_done` まで引き上げ**、検査も EC と **同じ本数まで**完了できるようにします（`_pipeline_ec_fully_done_for_tid` / `_roll_pipeline_inspection_assign_room`）。
- **検査の優先・再走査**  
  - **グローバル** `task_queue.sort`: **§B-1**（加工途中）の次に **`b_tier = 1`**（本節の検査行・未着手）が来ます。  
  - **同一日内** `_day_schedule_task_sort_key`（主に **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0`**）: **§B-1** 対象は **`dispatch_trial_order` を rank より前**へ。同じ `加工内容` 由来の **rank 内**では **§B-2** の検査を先に試すほか、**パイプラインに割当可能枠がある**（`_roll_pipeline_inspection_assign_room` > 0）ときは **工程順 rank を 1 段繰り上げ**、**他依頼の先行工程（rank 相当）と同じ波**で割付を試みます。  
  - **日跨ぎ再走査**: `_run_b2_inspection_rewind_pass` が `sorted_dates` の先頭から B-2 検査だけを再配台し、EC 完了後に「日を戻して検査」を実現します。

#### B-2.3 §A-1・§A-2 との関係

- **`_task_blocked_by_same_request_dependency`** は、**§B-2 の EC→検査**について、**`_roll_pipeline_inspection_assign_room` > 0** のときだけ **§A-1 の「先行 EC に残があれば検査不可」を適用しない**（それ以外の先行工程・他ペアは従来どおり）。
- **`_roll_pipeline_inspection_assign_room`** … **前進・リワインド配台の両方**で、検査ロールを **何本まで進めてよいか**の上限と最終ロール詰まり回避に用います（B-2.2）。**当日候補**では、**枠が 0 のときは除外**します。
- **同一依頼内の他の前工程**（B-2 以外）に残がある場合も §A-1 が検査をブロックし得ます。**他依頼NO**には B-2 定数は適用されません。

#### B-2.4 同一物理機械での検査占有（環境変数）

- 既定オン: **`PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE`**（`1` 相当。`0` / `false` / `off` 等で無効。名称は歴史的なもの）。**§B-2 検査**が**既に割付開始**し残ロールがある行がある **機械名占有キー**（上記 §4.1 と同じ `_physical_machine_occupancy_key_for_task`）上では、**他タスクを試行しない**（`_exclusive_b1_inspection_holder_for_machine`）。オフにすると時間割上は依頼が混在し得るが、長期占有ブロックは避けられる場合があります。

#### B-2.5 定数の変更

- `planning_core/_core.py` 内の `ROLL_PIPELINE_*` および `ROLL_PIPELINE_INITIAL_BUFFER_ROLLS` を編集してください。

#### B-2.6 実装上の候補フィルタ（`_trial_order_flow_eligible_tasks`）

当日候補に入る前に、同一依頼の **`_task_blocked_by_same_request_dependency`**（§A）· **（既定オン）キュー全体の配台試行順シリアル**（`_task_blocked_by_global_dispatch_trial_order`：`start_date_req` ≤ 当日かつ残ありの行のうち **`dispatch_trial_order` の最小値**に載らない行は除外。納期が近い試行順の大きい行も割り込まない。無効化は環境変数 **`STAGE2_GLOBAL_DISPATCH_TRIAL_ORDER_STRICT=0`**）· **B-2/B-3 後続向け（`roll_pipeline_inspection` または `roll_pipeline_rewind`）かつ `_roll_pipeline_inspection_assign_room` ≤ 0（ロール枠なし）**· **上記 B-2.4 の設備占有**· **同一物理機械上の試行順強制**（`_equipment_line_lower_dispatch_trial_still_pending`）の順で判定されます。後続行は前進配台・`_run_b2_inspection_rewind_pass` のどちらでも、枠・占有・時刻下限に従って載ります。

#### B-2.7 同一依頼内の担当者集合の排他（EC と検査／巻返し）

- **ルール**: 同一 **依頼NO** の §B-2 または §B-3（`roll_pipeline_ec` と後続の `roll_pipeline_inspection` または `roll_pipeline_rewind` がキューに両方ある）について、一方の工程の **`assigned_history` に一度でも記録された担当者**は、他方の工程の **チーム探索候補（`capable_members`）から除外**します。
- **環境変数（トグル）**: **`PLANNING_B2_EC_FOLLOWER_DISJOINT_TEAMS`** … **既定オン**（`1` 相当）。**`0` / `false` / `no` / `off` / `いいえ` / `無効`** で **上記の相互除外を無効化**し、EC と後続の両方で同一人物が候補に残り得ます（`PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE` と同趣旨の真偽解釈）。
- **実装**: `planning_core/_core.py` の `_filter_capable_members_b2_disjoint_teams`（`_assign_one_roll_trial_order_flow` および `STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0` の日次割付ループの双方で適用）。履歴の `team` 文字列と `avail_dt` のキーは **NFKC** で突合します。
- **トレース**: 配台トレース対象依頼NOでは、担当者分離で候補が削られたときに  
  `ブロック判定: B-2担当者分離 ... 候補除外=... 残候補=...` を `dispatch_trace_<依頼NO>_rNN.txt` に出力します。
- **追加トレース（候補試行）**: 配台トレース対象依頼NOでは、`_assign_one_roll_trial_order_flow` 内で  
  `候補固定: 担当OP指名=...`、`候補却下: OP不在`、`候補却下: 実働不足` など、チーム候補が不採用になった理由を出力します。
- **成立しない場合**: 除外の結果 **OP が必要人数に満たない**等でロールが付かないことがあります（要員・スキル・指名の見直し）。

#### B-2.8 工程間WIP上限（総ロール数・依頼NO／加工日をまたぐ）

**目的**: 工程の進捗差により、**検査前ロール**が作業スペースへ蓄積しすぎることを防ぐ（現場の許容量制約）。

**対象**: §B-2／§B-3 のロールパイプラインと同じ系（EC → 検査／巻返し）だが、**依頼NOを跨いだ総量**として制限する。

**定義（総ロールWIP）**:

- **総EC完了ロール数** \(EC\_DONE\_TOTAL\): 全 `roll_pipeline_ec` 行について `initial_remaining_units - remaining_units` を合算
- **総後続完了ロール数** \(FOLLOW\_DONE\_TOTAL\): 全 `roll_pipeline_inspection` / `roll_pipeline_rewind` 行について `initial_remaining_units - remaining_units` を合算（**検査＋巻返しを合算**）
- **検査前WIP（総ロール）**:  
  \(WIP = \max(0, EC\_DONE\_TOTAL - FOLLOW\_DONE\_TOTAL)\)

**ルール**:

- **WIP が上限以上**のとき、**EC（隣接するEC工程）を新たに配台しない**（候補から除外）。

**環境変数（上限ロール数）**:

- **`WIP_LIMIT_EC_BEFORE_INSP_ROLLS`**（既定 `15`）  
  - **`0` 以下**を明示すると **無効**（上限制約なし）
  - 実装は `planning_core/_core.py` の `_trial_order_flow_eligible_tasks` 冒頭で総WIPを算出し、EC候補を弾く。

### B-3 — EC（EC機　湖南）→ 巻返し（EC機　湖南）ロールパイプライン

**方針**: **§B-2 と同一のロールパイプライン挙動**（完全二相・`_roll_pipeline_inspection_assign_room`・`_roll_pipeline_b2_inspection_ec_completion_floor_dt`・`_run_b2_inspection_rewind_pass`・`PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE`・`_filter_capable_members_b2_disjoint_teams` 等）を適用する。**B-2 で「検査」行だった位置に相当する工程を「巻返し」**とし、機械名は **EC機　湖南**（先行 EC と同一機）。

#### B-3.1 対象（コード定数・同一依頼NOの行同士のみ）

| 段階 | 工程名 | 機械名 |
|------|--------|--------|
| 先行 | `EC` | `EC機　湖南` |
| 後続 | `巻返し` | `EC機　湖南` |

#### B-3.2 実装フラグ・集計

- キュー構築時: **`roll_pipeline_rewind`**（`planning_core/_core.py` の `ROLL_PIPELINE_REWIND_PROCESS` / `ROLL_PIPELINE_REWIND_MACHINE`）。
- 後続の完了ロール数は **`_pipeline_b2_follower_roll_done_units`** で **熱融着検査行と巻返し行を合算**する（同一依頼に両方の行があれば枠計算に両方が入る）。

**説明の読み替え**: §B-2.2〜B-2.7 の「検査」を、本節の後続（巻返し）として読み替えてよい。

### B-4 — スリット（スリット機1　湖南）→ SEC（SEC機　湖南）工程間WIP上限（総ロール数・SEC優先）

**目的**: スリット後（SEC前）のロールが作業スペースへ蓄積しすぎることを防ぎ、**SECを優先して進める**。

**対象**: 工程名 `スリット`×機械名 `スリット機1　湖南` と、工程名 `SEC`×機械名 `SEC機　湖南`。

**定義（SEC前WIP: 総ロール）**:

- **総スリット完了ロール数** \(SLIT\_DONE\_TOTAL\): 対象スリット行について `initial_remaining_units - remaining_units` を合算
- **総SEC完了ロール数** \(SEC\_DONE\_TOTAL\): 対象SEC行について `initial_remaining_units - remaining_units` を合算
- **SEC前WIP（総ロール）**:  
  \(WIP = \max(0, SLIT\_DONE\_TOTAL - SEC\_DONE\_TOTAL)\)

**ルール**:

- **WIP が上限以上**のとき、**スリットを新たに配台しない**（候補から除外）。
- 上記の状態では、結果として **SECが相対的に優先**され、WIPが解消される方向へ進む。

**環境変数（上限ロール数）**:

- **`WIP_LIMIT_SLIT_BEFORE_SEC_ROLLS`**（既定 `20`）  
  - **`0` 以下**を明示すると **無効**（上限制約なし）
  - 実装は `planning_core/_core.py` の `_trial_order_flow_eligible_tasks` 冒頭で総WIPを算出し、スリット候補を弾く。

#### B-4.1 5ロール以上でSEC開始（加工内容がスリット→SECの依頼のみ）

**ルール**: 加工内容が **`スリット,SEC`**（スリットが先・SECが後）と判断できる依頼について、  
**スリット完了 − SEC完了 < 5** の間は **SEC を候補から除外**し、スリットが **5ロール以上**完了した時点で SEC を開始する。

**実装**: `planning_core/_core.py` の `_trial_order_flow_eligible_tasks`。判定にはタスク辞書の `process_content_tokens`（加工内容トークン）を使う。

#### B-4.2 スリット行とSEC行の隣接（自動算出の配台試行順）

**ルール**: 加工内容が **スリット→SEC** の依頼について、**自動算出の配台試行順**では  
同一依頼NO内で **スリット行の直後にSEC行が来る**ようにキューを軽く並べ替える（隣接させる）。

**実装**: `planning_core/_core.py` の `_apply_dispatch_trial_order_for_generate_plan`（シートで全行の試行順が指定されている場合は適用しない）。

## 1. 配台の対象外（キューに載せない）

- **完了扱い**の行（完了キーワード等で除外）
- 配台計画シートの **「配台不要」** がオン
- **同一依頼NO × 同一機械名**で複数行あるグループのうち、工程名が **「分割」** の行は、セルが空なら自動で配台不要
- マクロブック **「設定_配台不要工程」**（C列＝常に配台不要の工程×機械、または E列の条件が真のときその組み合わせを配台不要とみなす）  
  - **段階1**（`run_stage1_extract`）で `apply_exclude_rules_config_to_plan_df` により、上記を **計画 DataFrame（中間 xlsx／以降の配台計画シート取り込み元）** に反映する。  
  - **段階2**（`load_planning_tasks_df`）では C/E を計画シートへ**再適用しない**。配台計画シートの列 **「配台不要」**（段階1出力・手編集の結果）だけを解釈する。

### 1.1 未加工列（必須）・残量算定と配台対象外の特例

- **列「未加工」は必須**（`load_planning_tasks_df` 経路の `_ensure_dataframe_has_unprocessed_column`）。列が無い、またはいずれかの行で **数値として読めない**（空・不正）ときは **`PlanningValidationError`** で段階2を中止します。
- **実出来高・実加工数から済相当へフォールバックしない**。**配台用の残り長さ(m)・済相当(m)・換算数量(100m 切上)** は **`_plan_row_dispatch_qty_metrics`** が **未加工列のみ**で算定します（`build_task_queue_from_planning_df` がこれを呼び出し）。
  - **未加工 > 0**: 残りm = 未加工、済相当m = max(0, 換算数量(raw) − 未加工)。
  - **未加工 ≤ 0**（セルが数値 **0** として読める場合）: **換算数量（100m 切上げ）**を残量の基礎とし、それが **ロール単位長さ** 未満なら **ロール単位長さまで引き上げ**（コメントどおり最小加工単位＝1 ロール・済相当m = 0）。業務上「換算数量あり・**実加工数 0**・**未加工 0**＝**全数未加工**」と読むケースを、この分岐で配台キューに載せます（リポジトリルール `processing-plan-data-full-unprocessed.mdc` と整合）。
- **キューへ載せない特例（加工済み扱い）**: 次を**すべて**満たす行は **`_plan_row_exclude_as_completed_mikan_unprocessed_zero_actual_done_rule`** でスキップします（上記「完了キーワード」や「配台不要」とは別経路）。
  - **未加工**列があり数値 **0**
  - **実加工数** が **0 以外**
  - **加工完了区分** が **「未完」**、または **`0:未完` のようにコロン区切りで末尾が「未完」**（「未完了」等は **未完扱いに含めない**）
- **ロール本数の初期化**: `from_unprocessed_qty` が真で **ロール単位長さ > 0** のとき、**残りm ÷ ロール単位長さ**の **切り上げ**を **`initial_remaining_units` / `remaining_units`** の初期値に使います。未加工由来でないフォールバック経路では **残りm / ロール単位**の実数商になります（`build_task_queue_from_planning_df` 内）。

## 2. タスクの並び順（割付の試行順）

`generate_plan` 内で構築した `task_queue` は、**日付ループに入る前**に **`_apply_dispatch_trial_order_for_generate_plan`** で並びと **`dispatch_trial_order`** を確定します。**全行シート指定**のときは列順ソート＋列値の反映のみ。**欠損行がある**ときは **`_generate_plan_task_queue_sort_key` で `task_queue.sort`** のあと §B-2/3 隣接化と **`1..n` 付与**です（**配台計画シートの物理行順そのものではなく**、この確定後の並びが試行番号の基準です）。

**各日の割付順**は環境変数 **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST`** で分岐します。

- **`=1`（既定）**: 当日の割付本体は **`_trial_order_first_schedule_pass`**（§2.0.2）。`_day_schedule_task_sort_key` はこの経路では**割付ループの主ソートには使いません**。
- **`=0`（旧フロー）**: 当日は **`_day_schedule_task_sort_key`** でソートした順に 1 本のループで試行します（試行順はキーの一部）。

### 2.0 グローバルキュー（`task_queue.sort` のタプル順）

### 2.0.1 日内の試行順（`_day_schedule_task_sort_key`）

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0`** のときだけ、同一日内の割付ループがこのキーで **`sorted(...)`** されます。先頭は **§2.0 と同趣旨**（**`in_progress` → 納期基準（`due_basis_date`） → `b_tier` → `b2_queue_sub` → need 列位置 → 依頼NOタイブレーク**）のあと、**§B-1** 用に **`dispatch_trial_order` を `process_sequence_rank` より前**へ置く **`(0, dispatch_trial_order)` / `(1, 0)`**、続けて **`process_sequence_rank`** → **`same_request_line_seq`** → **`dispatch_trial_order`** → **`b2_roll_pipeline_stage`（0=EC、1=未着手検査、2=その他）** → **配台計画の優先度** → **`_result_task_sheet_sort_key`**（配台試行順番・依頼NO…）です。  
このため **`dispatch_trial_order` は並びの要素の一つ**であり、§B 段・納期・need 列・rank 等と**同時に**効きます（試行順だけが絶対の第一キーではありません）。**ただし** **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`** のときは、候補化段階で **`STAGE2_GLOBAL_DISPATCH_TRIAL_ORDER_STRICT=1`（既定）** により、**より小さい試行順に未完了が残る限り大きい試行順は当日の割付候補に入りません**（レガシー日次ループ `=0` 時も同関数で同様）。同一日内の **`sorted` キー**とは別層の制約です。

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`（既定）** のときは、割付の主経路は **§2.0.2** を参照してください。

**結果_タスク一覧**の行順は **`_result_task_sheet_sort_key`** により **配台試行順番**（`generate_plan` 冒頭の **`_apply_dispatch_trial_order_for_generate_plan` で確定した** 1..n）が第一キーです。

### 2.0.2 配台試行順優先の日内割付（`_trial_order_first_schedule_pass`・既定）

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`** のとき、各日・各マルチパスで次を行います。

1. **`tasks_today`**（その日 `start_date_req` 以前で残あり）を起点に、**`_trial_order_flow_eligible_tasks`** で候補を絞る。除外され得るものの例: §A-1 の先行工程待ち、**（既定オン）より小さい配台試行順に当日詰めうる未完了がある**ときの大きい試行順（`_task_blocked_by_global_dispatch_trial_order`、環境変数 **`STAGE2_GLOBAL_DISPATCH_TRIAL_ORDER_STRICT`**）、§B-2/§B-3 の **後続ロール枠ゼロ**（`_roll_pipeline_inspection_assign_room` ≤ 0）、（既定で）後続の同一物理機械占有（B-2.4）、**同一物理機械上でより小さい `dispatch_trial_order` がまだ当日詰めうる**ときの後続試行順（`_equipment_line_lower_dispatch_trial_still_pending`）など。
2. 残った候補を **`dispatch_trial_order` 昇順**に並べる。
3. **1 パス**は **二相**。**フェーズ1**: 候補のうち **§B-2 の熱融着検査行および §B-3 の巻返し行以外**を **`dispatch_trial_order` 昇順**で並べ、各タスクを **`_drain_rolls_for_task`** により**可能な限り連続ロール**で詰める（EC・他依頼を含む）。**フェーズ2**: §B-2 検査／§B-3 巻返し行を同じく昇順で `_drain_rolls_for_task` する（候補化条件はロール枠・占有・時刻下限）。同一パス内ではフェーズ1が終わるまで後続を試さない（交互にすると EC 担当者が後続へ回り **EC がブロック**されやすい）。**リワインド**で後続を載せるときも **`_assign_one_roll_trial_order_flow`** 内で **ロール数枠**と **EC ロール終了時刻に基づく開始下限**（`_roll_pipeline_b2_inspection_ec_completion_floor_dt`）を織り込む。
   なお、同一タスクの連続ロールで使う **継続チーム優先**は **同日内のみ**です。翌日の最初のロールでは前日チームを固定せず、当日候補（勤怠・スキル・設備空き）から再選定します。
4. 日次ループ側の **`while`**（`_sched_max_passes`）が続く限り、**再度 1 から**同様のパスを繰り返す。

**読み替えのポイント**

- **試行順の若いタスクが優先**される（各フェーズは昇順リストの先頭側から検討）。
- **各フェーズ内**では、あるタスクについて**可能な限り連続ロール**を詰め、**外側の日次マルチパス**で同じ二相を繰り返す。
- **候補に入らなかったタスク**は、その日は試行順に関係なく割付対象にならない。

### 2.0.3 `dispatch_trial_order` と納期シフト再試行

**初回** `generate_plan` では、**`_apply_dispatch_trial_order_for_generate_plan`** が **配台試行順**を確定します。**配台計画シートの「配台試行順番」がキュー全行で正の整数**のときは、その昇順（同値は `planning_sheet_row_seq`）にソートし列値をそのまま **`dispatch_trial_order`** に載せます（**§B-2/3 の EC 隣接化は行いません**）。**1 行でも欠損**があるときは **`_generate_plan_task_queue_sort_key` でソート** → **`_reorder_task_queue_b2_ec_inspection_consecutive`** → **`_assign_sequential_dispatch_trial_order`** で **1..n** を付与します。

**`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=True`** のときのみ、**納期シフト再試行**（当該依頼の内部 `due_basis_date` を +1 日・割当クリア）の **直後**に **`_apply_dispatch_trial_order_for_generate_plan`** を **再実行**し、上記と同じ規則で試行順を **付け直します**。**既定オフ**では外側ラウンドが増えないため、この処理は **初回のみ**です。

### 2.1 開始日 `start_date_req`

- **配台に使う原反日付**: 列 **`原反投入日_上書き`**（`PLAN_COL_RAW_INPUT_DATE_OVERRIDE`）に日付があるときは **その日付**を「原反投入日」として以下のルールに使う。**空白**のときは列 **`原反投入日`**（加工計画DATA 由来）を使う。
- **原反投入日が無い**とき（上記の結果として採用日が無いとき）: **`start_date_req` = 計画基準日**（加工計画DATA の **抽出時間** を優先し、列が無い・有効値が無いときは **データ抽出日**。いずれも得られないときは実行時刻＝`run_date` の元になる）。
- **原反投入日がある**とき: **`start_date_req = max(計画基準日, 原反投入日)`**（原反前倒しはしない）。
- **特別指定**（セル／特別指定備考 AI の `start_date` マージ結果）に開始日上書きがあっても、**原反投入日がある行は `max(指定開始日, 原反投入日)` を採用**する（原反投入日より前への前倒しは不可）。
- メイン **再優先特別記載** の **工場休業日** は勤怠上全日非稼働。
- `generate_plan` 冒頭で、**`start_date_req` が誰も出勤しない日**のとき **直近の稼働日**へ繰り上げ。

### 2.2 原反投入日と同日開始

- **採用した原反投入日**（2.1 のとおり、**原反投入日_上書き**優先）と**同日**に加工を開始する場合、**その日の加工開始下限**は **`same_day_raw_start_limit` = 13:00**（`time(13, 0)`。開始日上書き有無にかかわらず適用。`_trial_order_flow_day_start_floor` でも原反当日は 13:00 帯を織り込む）。

### 2.3 配台残と納期の後ろ倒し再試行（任意・既定オフ）

**方針**: **配台試行順を正**とし、納期基準日を過ぎても **内部で `due_basis_date` を +1 してタイムラインを巻き戻す再試行は既定では行いません**（`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=False`）。カレンダーは **1 通し**で完走し、残りは **配台残**等として結果に残ります。

**トリガ**（実装 **`_collect_task_ids_missed_deadline_after_day`**）: **各日のシミュレーション終了時点**で、**残量があり**、**`due_basis_date`（内部の納期基準）が存在し**、`start_date_req` ≤ 当日 で、かつ **当日 ≥ `due_basis_date`**（納期当日を含み「期限内に完遂できなかった」）となった **依頼NO** を集めます。  
**上記フラグがオフのとき**はこの集合を **検出しても再試行に入らず**、当日処理はそのまま続行します（ログの「納期超過リトライ」も出ません）。

**従来挙動を有効にする場合**（`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=True`）: この **ミスした依頼NO** 集合に対し **納期後ろ倒し再試行**に入ります（**割当履歴の有無は条件に含めません**。以前の「配台残」の口語定義とは異なる点に注意）。

- **フラグ**: `planning_core/_core.py` の `STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING`（既定 **`False`**）。**`True`** にすると **内部納期基準 +1 日・巻き戻し・カレンダー先頭から再走査**の外側ラウンドが走り得ます。
- **1 ラウンドの内容**: 対象依頼NOの各行について **内部の `due_basis_date`（納期基準）のみ +1 日**（`_shift_task_due_calendar_fields_one_day`）。ソート・納期超過判定・再試行の優先度はこのシフト後の値を使う。**結果_タスク一覧の「計画基準納期」列**（列名は互換のため従来どおり）は **`due_basis_date_result_sheet`**（キュー構築時の元日付）のまま表示し、+1 連鎖は反映しない。**回答納期・指定納期**もシート取り込み値のまま。`remaining_units` と `assigned_history` を初期化し、**勤怠の自動拡張で増えた日付キーはマスタの日付集合へ巻き戻して**から、再度全日程を走査します。
- **シフトしない依頼**: いずれの行も **`due_basis_date` が無い**依頼は、後ろ倒しの対象にせず通常の配台残のままです。
- **勤怠最終日を超える場合**: `due_basis_date` を持つ行があり、**+1 日が勤怠マスタ上の最終計画日を超える**依頼は、そのラウンドではシフト対象外（カレンダー不足側の分類）になります。
- **上限**: `STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS`（既定 **5**）は **依頼NOごと**（シミュレーション全体の合計ではない）。当該依頼について納期基準+1の巻き戻し再シミュをこれ以上行わず、ログに警告し得ます。上限到達後も未完了が残る当該依頼の行には「納期見直し必要」を付けます。

**納期シフト再試行で外側ラウンドが増えた直後**（対象依頼の割当クリア・タイムライン掃除のあと）、**`_apply_dispatch_trial_order_for_generate_plan`** を再実行し、**`dispatch_trial_order` を再確定**します（全行シート指定時は列順へのソートと列値の反映のみ、欠損行があるときは **`_generate_plan_task_queue_sort_key` でソート → §B-2/3 隣接化 → `1..n` 付与**）。**§2.0.3** と整合します。**既定オフ**ではこの再確定は走りません。

## 3. 誰を候補にするか（スキル・人数）

### 3.1 稼働

- `master.xlsm` 各メンバー勤怠で、**加工ライン配台の候補**は **`eligible_for_assignment`** が真の人のみ（勤怠備考 AI の **`配台不参加`** が true の日は除外）。**出勤表示**は **`is_working`**（全日休み・欠勤は `is_holiday` 由来で false）。月次点検など「出勤はするが配台だけ不可」は `is_working` true・`配台不参加` true になり得る。
- 勤怠 **備考** が空でも **休暇区分** だけ入っている行は、文脈解釈のため **同じ勤怠 AI** に渡す（表記揺れ・婉曲表現はキーワードマッチではなく AI で解釈）。
- **出退勤の優先**: その行で **出勤時間** と **退勤時間** の**両方**がマスタに入っているときは、**セル値**を `start_dt` / `end_dt` の基準に用い、勤怠 AI の **出勤時刻 / 退勤時刻** では上書きしない（中抜け・休日・配台不参加などは従来どおり AI／列から解釈）。**どちらか一方だけ**欠けるときは、従来どおり AI と列の **OR** で補完する。
- 個人の空き開始は **`start_dt`**（退勤は `end_dt`）
- **残業終業（master メンバー勤怠の列）**: 定時の **`退勤時間`**（および勤怠 AI の **`退勤時刻`**）とは別に、任意列 **`残業終業`**（その日の退勤上限時刻）を置ける。`load_attendance_and_analyze` で **`is_holiday` でない行だけ**解釈する。適用後に退勤が出勤以前になる場合は **残業終業を無視**して定時退勤に戻す。`master_カレンダーからメンバー出勤簿を生成` では **K 列**として出力し、再生成時も **日付キーで手入力を復元**する（VBA `PreserveMemberAttendanceHandColumns`。旧 L 列のみに値があるシートは退避時に **K が空なら L を読む**フォールバックあり）。

### 3.2 スキル（`skills` シート）

- **工程名＋機械名**の交差セルが **OP** または **AS**（＋優先度整数）のメンバーのみ候補
- チームは **少なくとも1名が OP** でなければ不採用（AS のみの組は不可）
- 優先度は **数値が小さいほど**チーム選定で有利

### 3.2.1 機械カレンダー（`機械カレンダー` シート）

`master.xlsm` の **`機械カレンダー`** シート（VBA で作成。`skills` と同型の **1 行目＝工程名・2 行目＝機械名**、**A 列＝日時（1 時間刻みの**毎正時**スロット）**・**B 列＝曜日**、**C 列以降＝設備列**）を段階2で読み込みます。**スロットの開始・終了「時」**は **`メイン` シート A12（工場稼働開始）・B12（工場稼働終了）**から取得し（**時台のみ**使用。空または不正なときは **6:00〜22:00** の毎正時）、VBA `master_機械カレンダーを作成` が行を生成します。**メイン相当シート名**は **`メイン`**・**`メイン_`**・**`Main`** の順で完全一致を試し、無ければ名前に **メイン** を含み **カレンダー** を含まないシートのうち **文字数が最短**のものを採用します（`master.xlsm`／配台ブックとも、Python の `openpyxl` 経路と VBA `MasterGetMainWorksheet`／`GetMainWorksheet` が同趣旨）。

- **占有の解釈**: 設備列のセルに **空白以外の文字** が入っていると、その行の日時から **1 時間** `[slot_start, slot_start+1h)` の間、その列に対応する **設備キー**および **同一機械名に集約したキー** の双方について、加工配台の **`machine_avail_dt`** 上で **使用不可** として扱われます（`equipment_list` による正規化キーと突合。複数工程列が同一機械名のときはマージ）。**数値の 0**（整数・浮動小数、`0.0`）は **空欄と同様に非占有**とみなします（誤って 0 が入ったセルが全日「占有」になるのを防ぐ）。
- **トラブルシュート（特定時刻以降どの設備にも配台が入らない）**: その時刻付近まで加工が載ったあと **`machine_avail_dt` が機械カレンダーの占有ブロック内に入り**、`_bump_dt_past_machine_calendar_blocks` で **終業時刻まで繰り上がる**と、以降は全設備で新規加工が入りません。**該当日の午後スロット**に不要な文字・記号が入っていないか、**メイン A12/B12 の工場終了時刻**が意図より早くないか、**出勤簿の退勤**が早すぎないかを確認してください。
- **当日のシード**: `_seed_avail_from_timeline_for_date` の **直後** に **`_apply_machine_calendar_floor_for_date`** を呼び、占有と重なる `machine_avail_dt` を **`_bump_dt_past_machine_calendar_blocks`** で **ブロック終了以降** に繰り上げます。
- **占有中の更新**: ロール確定で **`machine_avail_dt` を機械名占有キーで更新**した **直後** に **`_bump_machine_avail_after_roll_for_calendar`** を呼びます（試行順フロー・レガシー日次ループの両方）。
- **未整備時**: シートが無い・空・読込失敗時は **何も占有しない**（従来どおり）。
- **定常外の見た目（master 側）**: **`メイン` A15（定常開始）・B15（定常終了）** が有効なとき、**`master_機械カレンダーを作成` 後**に **1 時間スロットが半開区間 [定常開始, 定常終了) と重ならない行**の **A 列（日時）・B 列（曜日）** を薄いオレンジ系で塗ります（A15/B15 が空・不正のときは着色なし）。

#### 3.2.1.1 日次始業準備（`設定_機械_日次始業準備`）

`master.xlsm` に任意で置く **`設定_機械_日次始業準備`** を段階2 `generate_plan` 冒頭で読み込み（`load_machine_daily_startup_settings`）、**設備の壁時計**（`machine_avail_dt` と整合する **加工開始時刻の下限**）に織り込みます（`_machine_effective_floor_for_assign`・`_resolve_machine_changeover_floor_segments` 等。`_changeover_plan_segments_and_machining_lower_bound` は **日次始業用セグメントのみ**を組み立てます。識別子に `changeover` が残るのは歴史的呼称です）。あわせて **`メイン` A15/B15** は `_read_master_main_regular_shift_times` で読み、**A15（定常開始）**を **`_STAGE2_REGULAR_SHIFT_START`** に保持します（日次始業の壁時計アンカー用。B15 は日次始業ロジックでは未使用）。シートが無い・列が揃わない・データが空のときは **従来どおり**（加算なし）。

**`設定_依頼切替前後時間`**（工程名・機械名・準備分・後始末分）: **現行の `planning_core` では読み込みません**。依頼NO切替に伴う **加工前準備・後始末の壁時計加算・タイムライン出力・日末の後始末追記**は **行いません**（`master.xlsm` にシートが残っていても無視。再導入は別途実装）。

1. **`設定_機械_日次始業準備`**（1 行目見出し・2 行目以降データ）
   - **機械名**・**日次始業準備_分**（別名: `始業準備_分` / `日始業準備_分`）
   - **同一カレンダ日・同一占有キー**で、その日 **最初の加工ロール**を載せるときだけ適用します。当日すでに当該機械に **加工**イベントがある場合（タイムラインシード後・またはシミュレーション内で先に加工を載せた後）は **二度目以降は日次始業準備を置きません**（`machining_today_occ`）。
   - **区間の置き方（定常開始 A15 が有効なとき）**: **勤怠に紐づけず**、当該計画日について **半開区間 `[定常開始, 定常開始 + 日次始業準備分)`** を **壁時計の固定ブロック**としてタイムライン・設備占有に載せます（その時間帯は **その機械で加工しない**／加工開始下限は少なくともこの区間の終了以降）。`_changeover_plan_segments_and_machining_lower_bound` がこのセグメントを出力し、`_machine_effective_floor_timedelta_only` でも先頭ロール時に **`max(従来の mf, 定常開始+N分)`** で整合します。
   - **A15/B15 が読めない・開始≥終了で無効なとき**（`_STAGE2_REGULAR_SHIFT_START` が無いとき）の **日次始業のみ**、従来どおり **代表スキル OP** の出勤・休憩に沿った `calculate_end_time` による forward にフォールバックします。

**誰の勤務・休憩に合わせるか（時刻の forward）**: **日次始業**のうち **A15 が無効**で代表スキル OP の勤務線で forward するとき、当日 `eligible` かつ当該 **工程+機械**で **OP スキル**を持つ者のうち **優先度が最小の1名**（`_pick_skilled_op_for_changeover_interval`）の **出勤区間・休憩**に沿って `calculate_end_time` で区間を置きます。**A15 有効時**の `[定常開始, 定常開始+N分)` ブロックは **勤怠と無関係**（前項のとおり）。

**タイムライン上の人員（表示・`avail_dt`・ミラー）**:
- **日次始業**（`machine_daily_startup`）: **主 `op`・補 `sub` は空**（人の配台は載せない）。**`task_id`（依頼NO）も空**（機械単位の区間であり特定依頼に紐づけない）。**設備占有キー**のみミラー・壁時計に効かせます。**A15 無効時のみ**、区間の forward に代表スキル OP の勤務線を使います（セグメントの `op` は空のまま）。

**スキル OP が1人も取れないとき**: 代表者の勤務線を引けないため、**壁時計に分を足す**従来互換（`_machine_effective_floor_timedelta_only`）へフォールバックします（**日次始業**は **A15 有効時** `_STAGE2_REGULAR_SHIFT_START` により **定常開始+N分**の `max`、**A15 無効時**は従来の **分の加算**）。

**結果_設備ガント**: **`event_kind` = `machine_daily_startup`** のスロットは **黄色系**（実装色 `FFEB9C`）で塗り、**連続区間の先頭マス**の表示は **`(日次始業準備)`** とします（依頼NOハッシュ色の対象外）。

**結果_設備毎の時間割（薄緑着色）**: 設備列の表示に **`(日次始業準備)`** が含まれるセルを薄緑にします（`planning_core` の `_apply_equipment_schedule_prep_cleanup_fill` と整合）。

**タイムライン出力**: **日次始業**はロール確定時に **加工行より前**に **別イベント**として `timeline_events` へ追記し、`DispatchIntervalMirror` にも登録します。**加工行**には `event_kind` = `machining` を明示します。**余剰サブ追記**（`append_surplus_staff_after_main_dispatch`）の対象は **加工イベントのみ**（従来どおり）。**日次始業**には余剰サブを足しません。

**ミラー**: **日次始業**などのセットアップ区間が既存占有と **設備キー**（担当者が載る場合は **担当者**）で重なる場合は、その日の当該ロール割当を **棄却**します（試行順フロー・レガシー日次ループとも）。

**メモ**: 加工の `team_start` は、上記で求めた **加工開始下限**（必要なら日次始業終了時刻）と **チームの `avail_dt`** の遅い方など、既存ロジックで決まります。日次始業と加工の間に **意図的な空き**が出る場合があります。

### 3.2.2 メイン定常時刻（A15/B15）と出勤簿

- **〇月カレンダー → メンバー出勤簿**（`master_カレンダーからメンバー出勤簿を生成`）で、記号 **`通常` / 空白 / `国` / その他（デフォルト扱い）** の **出勤・退勤**は **A15/B15** を採用します（読めないときは **8:45～17:00**）。**前休**は **休憩時間1_終了～定常終了**（生成マクロでは通常行と同じ休憩1、既定 **12:50～B15**）、**後休**は **定常開始～休憩時間1_開始**（既定 **A15～12:00**）。Python の `infer_mid_break_from_reason` は勤怠行の **休憩時間1_開始/終了**列（空なら `DEFAULT_BREAKS`）で同趣旨に補正します。
- 配台ブック **メイン**の勤怠色分け（結果出勤表示と **定常**の一致判定）は **master の A15/B15** を参照します（従来の A12/B12 ではありません）。

### 3.3 必要人数 `req_num`

1. 配台計画シートの **必要OP** 列に正の整数があればそれを使用
2. 無ければ **`need` シート**：`工程+機械名` → 機械名のみ → 工程名のみの順で参照
3. **特別指定1〜99** は、条件が依頼NOに一致したうえで上書き（**order が小さいほど優先**）

### 3.4 メイン「再優先特別記載」（グローバル AI）

- **`ignore_need_minimum`** … `req_num` を **1** に固定（`need`・行の必要OPより優先）
- **`ignore_skill_requirements`** … 全員 **OP 扱い**かつ優先度を実質同一にし、スキルマトリクスを無視
- **`abolish_all_scheduling_limits`** … 設備専有・原反同日13:00下限・指定開始時刻・マクロ実行時刻下限などを適用しない（併せて skill / need も緩和される経路あり）
- **`global_day_process_operator_rules`** … Gemini がグローバルコメントから解釈した **「日付（YYYY-MM-DD）＋工程名の部分一致＋複数メンバー名」** の配列。該当する **稼働日**に、該当する **工程名**（タスクの工程名列）のタスクを配台するとき、列挙したメンバーを **同一チームに必ず含める**（氏名の解決は **担当OP指名と同じ**：当日の **OP/AS 候補**に対して照合）。**依頼NOが分かる1名の主担当**は従来どおり **`task_preferred_operators`**（**オブジェクトのみ。配列不可**）、**「◯月◯日の△工程にＡとＢを配台」** 型は本キーへ出す想定。解決できない指名は警告ログのみ。 **_assign_one_roll_trial_order_flow**（試行順優先フロー）と **従来の組合せ探索ループ**の両方でプリセット・候補チームをフィルタする。最低人数 `req_num` は必須メンバー数を下回らないよう引き上げる。
- **誤スキーマ時の救済** … モデルが **`task_preferred_operators` に配列**や **`workstation_id` 等の誤キー**で長文だけを返した場合でも、正規化の前段で内容を読み、**可能な範囲**で **`global_day_process_operator_rules`**（日付・工程名・「ＡとＢを配台」型の推定）および **`scheduler_notes_ja`** へ落とし直す（推定できない文だけはメモに残す）。

### 3.5 余剰増員（`need` の「配台時追加人数／余力時追加人数」行）

`master.xlsm` の **need** シートで、基本必要人数行の**直下**（Excel 上は概ね **5 行目付近**）に置かれる行を読み、工程×機械列ごとに **必要人数を満たしたうえで追加で入れられる人数の上限**（`extra_max`／`surplus_map`）を定義します。

**既定（2026-04 以降）**

1. **メインの配台シミュレーション**では、この上限を **チーム人数の組み合わせ探索には使わない**。メインでは **`req_num`（基本必要人数）ちょうど**でチームを組む（`_assign_one_roll_trial_order_flow` および非試行順優先フローで `extra_max` を 0 扱い）。
2. **全日程の割付シミュレーションが終了した後**、`append_surplus_staff_after_main_dispatch` が各タイムラインブロックを時系列に見て、**まだ `req_num + extra_max` に達していない**ブロックに対し、次をすべて満たす人を **サブ（`sub`）に追記**する（対象は **`event_kind` が加工（省略時も加工）の行のみ**。**日次始業準備**（`machine_daily_startup`）は **サブ追記の対象外**（人員を載せない区間のため）。  
   - 当日 **`eligible_for_assignment`**  
   - **OP または AS** で当該工程×機械にスキル適合（メイン割付と同じキー解決）

   - 追記順はスキル優先度のうえ、**AS を OP より先**に並べる  
3. **ブロックの開始・終了時刻は再計算しない**（メインで決めた `end_dt` のまま）。追記は **結果表示・サブ欄・個人スケジュール上の同席表現**が主目的。

## 4. 時刻・設備の制約

### 4.1 チーム開始時刻

- 各メンバーの **`avail_dt`（その時点の空き）** の **最大値**（最も遅い人に合わせる）
- **`abolish_all_scheduling_limits` がオフ**のとき、次を順に適用  
  - **同一物理機械**は1タスクずつ：`machine_avail_dt` のキーは **マスタ「機械名」列を正規化した値**（`machine_occupancy_key`）であり、**工程+機械の設備列が複数あっても同時刻に重ならない**。結果シートの **設備列（工程+機械）** は従来どおりだが、壁時計の占有は機械名単位で共有する。  
  - 原反同日の **13:00 下限**（`same_day_raw_start_limit`／`_trial_order_flow_day_start_floor`）  
  - タスクの **earliest_start_time**（セル／AI）  
  - 計画基準日が **実行日と同じ日** のとき、**抽出時間（なければデータ抽出日）の日時より前**には開始しない

### 4.2 休憩・中抜け

- **`breaks_dt`**（所定休憩＋中抜け）：勤怠備考 AI の中抜け、または AI が空のときは `infer_mid_break_from_reason` による推定を `breaks_dt` に載せ、**`calculate_end_time`** でその日処理可能な単位数を算出します。
- **割付時の追加制約**（2026-04）: 終業直前・休憩帯内開始・休憩またぎの扱いは **改定（2026-04）** および環境変数 **`ASSIGN_END_OF_DAY_DEFER_MINUTES`** / **`ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS`** を参照。

### 4.3 増員時の作業速度

- 必要人数を超えた人数について、単位あたり時間に短縮係数 **`_surplus_team_time_factor`** を掛ける（**メイン割付で `extra_max` を使う場合**に効く。既定の「後追いサブのみ」ではメインは `req_num` 人数のため短縮はかからない）。
- 短縮の上限は定数 **`SURPLUS_TEAM_MAX_SPEEDUP_RATIO`**（既定 **0.05**＝最大約5%）の線形モデル

### 4.4 標準シフト枠（参照）

- **`DEFAULT_START_TIME`** / **`DEFAULT_END_TIME`** … ソース上の既定は **8:45** / **17:00**（コメントどおりの日内配台・結果グリッドの基準）。
- **上書き**: 段階2 `generate_plan` の実行中、`master.xlsm` の **`メイン` A12・B12** に有効な時刻が入っているときは **その値で一時上書き**し、**結果_設備毎の時間割・結果_設備ガント・個人スケジュール**など **`DEFAULT_*` を参照する出力**が **同じ枠**に揃います（A12/B12 のどちらか欠損・開始≥終了・openpyxl で読めないときは既定のまま）。
- **定常外の見た目（段階2）**: **`メイン` A15・B15** が有効なとき、**結果_設備毎の時間割**・**結果_設備毎の時間割_機械名毎**の **「日時帯」列**（`HH:MM-HH:MM`）のうち **半開区間 [定常開始, 定常終了) と重ならないセル**、および **結果_設備ガント**の **時刻列見出し（10 分刻み）** の同様のセルを、薄いオレンジ（`RESULT_OUTSIDE_REGULAR_TIME_FILL`）で塗ります（欠損・不正時は着色なし）。**工場稼働外**（`メイン` A12/B12）との関係は **結果_設備毎の時間割** と同様に、VBA 取込後 `取込後_結果シートへマスタ時刻を反映` で **日時帯列**に薄い青／定常外は薄いオレンジの優先を適用します（Python の xlsx 単体では定常外オレンジのみ）。**結果_設備ガント**は **時刻列の見出し（10 分刻み）を 3 行目に 1 回だけ**置き、**窓枠固定は E4**（行 1〜3・列 A〜D を固定。Python 出力でも `freeze_panes` を設定し、マクロ取込み時 `結果_設備ガント_列幅を設定` でも **E4 をアクティブにしてから** `FreezePanes` を掛け直し、**`A1.Activate`** のあと **表示倍率 85%**（`ActiveWindow.Zoom`）を設定し、`ScrollColumn = 1`・`ScrollRow = 1` で表示位置を左上に戻します（`E4` 操作後に横スクロールが右寄りになるのを防ぐ）。段階2 完了時は全 `結果_*` を 100% にしたあと **`結果_設備ガント_表示倍率を設定` で 85%** に戻します。）**A 列**は各日のデータ行（計画／実績を含む）を **縦結合**し、その日の **`【YYYY/MM/DD】`** を **左 90° 回転（縦書き相当）**で表示します（旧レイアウトの「日付だけの横断見出し行」は廃止）。**レポート 1〜2 行目**は **全幅を横結合**した見出し・メタ行とします。**時刻が入る列（E 列以降）だけ**は **タイムスロット列を横結合せず**、同一タスク／休憩／空きの連続区間は **地色の帯**にし **ラベルは先頭マスのみ**（オーバーフロー表示。**A〜D・タイトル帯は結合可**）。**B〜D** が **機械名・工程名・タスク概要**、**E 列以降**が時刻グリッドです（**タスク概要**列は **依頼NO のみ**を半角スペース区切り。**固定列の担当者は廃止**；タイムライン帯の角丸ラベル等で担当者を補助表示する場合は `GANTT_TIMELINE_SHAPE_LABELS`／`member_labels` を参照）。**日付ブロック同士の境目**に **行高 5 pt・背景黒**の区切り行を 1 行挿入します。データ行の行高は **52 pt**（タイトル行 **40 pt**・メタ **26 pt**・表頭（3 行目）**65 pt** は Python 側）。フォントは帯内ラベル **14pt**・左ブロック **14pt**・表見出し **14pt**・時刻見出し **14pt**・タイトル **24pt**（`_write_results_equipment_gantt_sheet`）。**印刷レイアウト**は Python 側で **A3 横 / 余白「狭い」 / 「ページに合わせる」＝横 1 ページ・縦は自動（`fitToPage`＋`fitToWidth=1`・`fitToHeight=0`）/ 1 日 1 ページ（手動改ページ）/ 行 1〜3 を印刷タイトル行**として設定します（`_write_results_equipment_gantt_sheet` の `page_setup` / `page_margins` / `print_title_rows` / `row_breaks`）。**手動改行（`rowBreaks`）の `brk@id` は仕様上 0 基準**のため、実装では **日ブロック先頭の 1 基準行番号から 1 を引いた値**を渡し、**日の途中で 1 行ずれる誤りを防ぐ**。

## 5. チーム選定の優先基準

そのタスク×その日について **成立する候補チームをすべて集めたうえで**、候補ごとにタプルを比較し、**辞書式で小さい方**を採用します。  
比較には **その日の候補全体の最早開始時刻 `t_min`** を使います。

### 5.0 環境変数の優先関係

1. **`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF=1`**（従来モード）… 下記を **無条件で** 人数最優先にします（スラック分は使いません）。
2. 上記が **オフ**（既定 `"0"`）で、**`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES`**（既定 **60**）が **正**のとき … **開始優先＋1時間以内は人数を厚く**（下記「スラック付き」）。
3. **`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES=0`** かつ余剰優先オフ … **開始のみ**優先（下記「開始のみ」）。

### 5.0.1 従来モード（`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF=1`）

- **`(-チーム人数, 開始時刻, -その日の単位数, 優先度合計)`**  
  → **人数が多い** → **開始が早い** → **その日の処理量が多い** → **スキル優先度合計が小さい**

### 5.0.2 スラック付き（既定の割合）

- **`t_min`** … その日の **すべての成立候補**のうち **最も早い** `team_start`
- 候補の遅れ **`team_start - t_min`** が **`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES` 分以下**のとき（**同じ「帯」**）:  
  **`(0, -チーム人数, 開始時刻, -その日の単位数, 優先度合計)`**  
  → **スラック時間以内の開始の遅れ**であれば **人が集まる（人数の多い）チーム**を優先
- 上記の **スラックを超えて遅い**候補は:  
  **`(1, 開始時刻, -チーム人数, -その日の単位数, 優先度合計)`**  
  → **開始が早い**方を優先し、同帯では人数・単位数・優先度でタイブレーク

### 5.0.3 開始のみ（スラック 0）

- **`(開始時刻, -その日の単位数, 優先度合計)`**

### 5.0.4 マスタ「組み合わせ表」プリセット（`TEAM_ASSIGN_USE_MASTER_COMBO_SHEET`）

- **`TEAM_ASSIGN_USE_MASTER_COMBO_SHEET` が有効**（既定 `"1"`）のとき、工程×機械キーごとの **プリセット編成**は、**成立すれば即採用で打ち切りにはしません**。
- 各プリセットについて **成立可否を試行**し、成立したものは **`team_candidates` に追加**されます。続けて **通常の組み合わせ探索**（`itertools.combinations` 等）で得た成立チームも同じリストに載ります。
- **採用は 1 件**で、上記 **候補全体**に対し §5.0.1〜5.0.3 の **`_team_assignment_sort_tuple`（`t_min` を含む）**で辞書式最小を選びます。  
  → シート上の **組合せ優先度は「試行順」にしか効かず**、**より早い `team_start` のチーム**（プリセット外でも可）が選ばれ得ます。
- 結果履歴の **組合せ表#n**（A列 ID）: プリセット候補からその行として採用したときは従来どおり ID が付きます。加えて、**組合せ探索のみ**で最良が選ばれて ID が欠けている場合でも、採用チームのメンバー集合（NFKC・trim）がマスタ「組み合わせ表」のいずれかのプリセット編成と一致すれば **`_lookup_combo_sheet_row_id_for_preset_team`** で ID を補完し、**履歴1〜n で同じ表示**になります（同日連続ロールの `preferred_team` 短絡など、従来は履歴2以降で欠け得たケースを含む）。いずれのプリセットとも一致しない採用では付きません。

### 5.1 主担当（lead OP）

- **担当OP指名**が当日の OP 候補に解決でき、かつチームに含まれる → その人を lead
- それ以外 → OP のうち **(スキル優先度, 名前)** が最小の人

### 5.2 担当OP指名の入力源と照合

- 配台計画の **担当OP_指定** と、特別指定備考 AI の **`preferred_operator`**（セル優先）
- メイン再優先の **`task_preferred_operators`** がある場合は **最優先で上書き**
- メイン再優先の **`global_day_process_operator_rules`** は **日付×工程**が一致するタスク×日のチームに、列挙した **全員を必須同席**（上書きではなく **チーム構成の拘束**。担当OPセル・AI と併用時は **和集合**を必須メンバーにする）
- 名前は NFKC 正規化・**富田/冨田**（姓側の表記寄せ）で、当日 **`op_today`**（その設備で OP の人）と照合
- **姓・名の表記ゆれルール**（`skills` の全メンバーで判定）  
  - **姓**は正規化後 **完全一致のみ**（別姓を名のゆれで誤マッチさせない）。  
  - **`skills` に同一姓のメンバーが2人以上いない** ときだけ **名**は、完全一致に加え **どちらかの文字列が他方を含む**場合も一致とみなす（略称・表記のゆれ）。  
  - **同一姓がロスターにいる** ときは **名も完全一致**のみ。  
  - 姓・名の区切りは **最初の半角／全角空白**（マスタの「姓　名」形式に合わせる）。名ゆれモードで **姓だけ**入力したとき、同姓候補が複数いれば **解決しない**（曖昧のため）。
- **一致しない場合**はログに記録し、**指名は無効**となり、通常の組み合わせ探索のみ（別の OP で埋まり得る）

## 7. 計画の日付範囲

- 勤怠データに含まれる日のうち、**計画基準日（加工計画DATA の抽出時間を優先、なければデータ抽出日）以降**が対象です（`generate_plan` 内で `sorted_dates` を基準日以降に絞り込み）。
- **現行（段階2）**: **`STAGE2_EXTEND_ATTENDANCE_CALENDAR = False`** のため、上記の勤怠に載っている日付の範囲**だけ**で各パスの日ループを行い、**勤怠の自動拡張（日付の複製追加）は行いません**。最終日までに割り切れないタスクは **配台残**または **配台不可**のまま残り得ます。
- **配台残の納期巻き戻し再試行**（任意）: **`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=True`** のとき **セクション 2.3** のとおり、内部の計画基準を +1 し当該依頼の割当を戻して**同じ勤怠範囲でもう一度**カレンダー通しを回します（勤怠日付そのものを増やすのとは別メカニズム）。**既定はオフ**です。
- **オプション**: `planning_core/_core.py` で **`STAGE2_EXTEND_ATTENDANCE_CALENDAR = True`** にすると、**残タスクがあるのに最終勤怠日までに終わらない場合**、直近の勤怠日と同じ出勤パターンを **1 日ずつ複製**して日付を延長し、残ロールがなくなるまで配台を続けます。拡張の上限は **`SCHEDULE_EXTEND_MAX_EXTRA_DAYS`（既定 366 日）**です。

### 7.1 ロール単位長さ（配台計画_タスク入力）

- **列名**: **`ロール単位長さ`**（定数 `PLAN_COL_ROLL_UNIT_LENGTH`）。シート上では **製品名** の**右隣**（`plan_input_sheet_column_order`）。
- **段階1**: `run_stage1_extract` で **換算数量・ロール単位長さを確定**する。内容は次のとおり（いずれも段階2では再実行しない）。
  - **`infer_unit_m_from_product_name`**: 製品名に現れる寸法区切り（半角 `x`/`X`、または `×`・全角Ｘｘ 等は正規化後に同様）について、**最後の `NNNX MM`（各 2〜6 桁）ペアの右側（`X` の後ろ）**をロール長の候補とする（実データ照合: テストコード直下 `製品名,ロール単位の長さ.txt`）。ペアが無いときは最後の `X` 直後の 2〜6 桁。**いずれも解釈できないとき**（寸法なし、`X` 後が 2〜6 桁でない等）は換算数量には落とさず、既定 **`INFER_ROLL_UNIT_LENGTH_DEFAULT_NO_MATCH_M`（100m）** を **`ロール単位長さ`** の推定に用いる。製品名セルが欠損（`None` / NaN）のときだけ従来どおり `fallback_unit`（換算数量等）を返す。
  - **換算数量が推定ロール長より小さいとき**: 列またはフォールバックで得た **1ロール長**が推定より小さく、かつ **換算数量**が推定ロール長未満なら **推定ロール単位長さを採用**（`_coerce_roll_unit_m_when_converted_qty_below_roll`）。推定不能、またはシート値が推定より大きいときは上書きしない。
  - **換算数量の 100m 下限（段階1出力）**: 正の **換算数量**が **100m 未満**なら **100m**（`PLANNING_MIN_QTY_M`・段階1出力の当列を上書き）。
  - **ロール単位長さの切り上げ（下二桁繰り上げ）**: 正の **ロール単位長さ**を **`ROLL_UNIT_LENGTH_CEIL_STEP_M`（100m）の倍数**に切り上げる（`_ceil_roll_unit_length_m_to_next_step`）。例: 40→100、20→100、125→200。推定・補正の**直後**に適用し、さらに **`_merge_plan_sheet_user_overrides` の後**に列全体へ再適用する（マージで引き継いだ手入力値も同じ規則にそろえる）。
  - **寸法なし＋ロール長＝換算数量の矯正（マージ後）**: `_heal_stage1_roll_unit_no_dim_when_roll_matches_qty_mistake` が、製品名に寸法ペアが無いのに **ロール単位長さが換算数量（100m 下限後）またはその 100m 切上と同じ**かつ **ロール長が 500m 以上**の行を **`INFER_ROLL_UNIT_LENGTH_DEFAULT_NO_MATCH_M` 経路の既定**へ戻す（旧シートに換算数量が誤って載っている FEL 等向け。500 未満は手入力とみなし触れない）。
  - 段階1再実行時、既存シートの **ロール単位長さ**は **`PLAN_STAGE1_MERGE_EXTRA_COLUMNS`** で旧値優先の行がある（依頼NO＋工程名）。**配台試行順番**は `MERGE_EXTRA` に含めず、段階1冒頭で空クリアしたうえで `fill_plan_dispatch_trial_order_column_stage1` が毎回付け直す。
- **段階2**: `build_task_queue_from_planning_df` は **`calc_done_qty_equivalent_from_row` にシートの換算数量をそのまま渡す**（実績換算はセル値ベース）。その直後、配台用の **換算数量（`total_qty_m`・残量・ロール長フォールバック等）だけ**を **`ROLL_UNIT_LENGTH_CEIL_STEP_M`（100m）の倍数に切り上げ**する（例: 40→100、125→200。`_ceil_roll_unit_length_m_to_next_step`・Excel は書き換えない）。**ロール単位長さ**はシート値を `unit_m` に用い、**空・0・不正**のときだけ **製品名推定**で補う（寸法解釈不能時は **既定 100m**、換算数量へは落とさない）。残りロール本数・`base_time_per_unit`・§B-2 のロール集計はこの `unit` に基づく。
- **業務ルール（最小加工長さとロール単位）**: 配台に使う **加工の最小長さ**は **ロール単位の長さを下回ってはならない**（リポジトリルール `dispatch-min-processing-length-vs-roll-unit.mdc`）。実装では **未加工 ≤ 0** の残量算定で **少なくともロール単位長さまで引き上げ**るほか、段階1の換算・ロール長の切上げ・矯正（§7.1 本文）と合わせ、通常運用で矛盾が出にくいよう寄せています。

### 7.2 在庫場所（加工計画DATA → 配台計画_タスク入力）

- **列名**: **`在庫場所`**（定数 `TASK_COL_STOCK_LOCATION`）。**加工計画DATA** に同一見出しの列があるとき、段階1で **原反投入日** の**右隣**に出力されます（`SOURCE_BASE_COLUMNS` / `plan_input_sheet_column_order`）。
- **配台計算**: 参照表示のみ。キュー構築（`build_task_queue_from_planning_df`）では内部キーに載せません。

### 7.3 原反投入日_上書き（配台計画_タスク入力）

- **列名**: **`原反投入日_上書き`**（定数 `PLAN_COL_RAW_INPUT_DATE_OVERRIDE`）。**上書き列ブロック**にあり、直前の参照列は **`（元）原反投入日_上書き`**（括弧内は列 **`原反投入日`** の値）。
- **段階1**: 新規抽出時は空。段階1再実行時は **`PLAN_STAGE1_MERGE_COLUMNS`** により、既存シートの当該セルが空でない行は **依頼NO＋工程名**キーで引き継ぎます。
- **段階2**: **`build_task_queue_from_planning_df`** で、本列に有効な日付があるとき **その日付を `raw_input_date` として採用**（開始日下限・同日13:00下限・**特別指定備考 AI の start_date**（該当する備考行のみ）との `max` など、原反関連はすべてこの日付基準）。空白のときは列 **`原反投入日`** のみを使います。
- **結果_タスク一覧**: 列 **`原反投入日`** の表示は、上書きがあるとき **上書き日付**（配台と同じ）、無いとき **シートの原反投入日**です。

## 9. 結果_タスク一覧のタスクID表示（加工内容との整合）

**目的**: 元データ側の **「加工内容」**（カンマ区切りの工程リスト）と、配台計画の **「工程名」** が食い違っている行を、段階2の結果ブック上で識別しやすくします。

**対象**: 段階2が出力する `production_plan_multi_day_*.xlsx` の **結果_タスク一覧** シートの **タスクID** 列（セルのみ）。

**判定**（`build_task_queue_from_planning_df` 内。加工内容の取り方は §A-1 と同じ `_collect_process_content_order_by_task_id`）:

- 同一 **依頼NO** について、表を上から見て最初に現れた **非空の加工内容** をカンマ `,` で分割したトークン列を「正しい工程名の一覧」とみなします。
- **トークンが1つも無い**依頼（加工内容が未記入のみ等）は、比較基準が無いため **ハイライトしません**。
- トークンがある依頼で、当該行の **工程名** が（NFKC・`_normalize_process_name_for_rule_match` による照合で）**いずれのトークンとも一致しない** とき、その行の **タスクID** セルに **塗りつぶし赤（`FF0000`）・文字白** を付与します。

**その他の列・既存の色**:

- **配台不可**／**配台残** の行に対する薄い黄の行全体の塗りより **後** にタスクIDだけ上書きするため、不整合がある行では **タスクIDセルは赤が優先**されます（他列は黄のまま）。

**結果_設備毎の時間割へのリンク**（`_apply_result_task_id_hyperlinks_to_equipment_schedule`）:

- **タスクID** の表示文字に **内部ハイパーリンク**を付け、**結果_設備毎の時間割** シートで当該タスクが **最初に現れるセル**（`[タスクID] 主:…` 形式のテキストが入る最初のマス）へジャンプできる。
- 時間割に一度も現れないタスク（例: 未割当のみ）はリンクなし。
- 同一依頼NOが複数行ある場合も、タスクID文字列が一致すれば同じ先頭セルを指すことがある。

**結果_設備毎の時間割の「補」表示のそろえ方**（`_equipment_schedule_unified_sub_string_map`）:

- メイン割付・余力追記の結果、**同一日・同一設備列・同一依頼NO** の加工でも、ブロックごとにタイムライン上の `sub` 文字列が食い違うことがある（例: 最終チャンクだけ別の補助者が乗る）。
- **タイムライン本体の `sub` は変更しない**（メンバー日程・ブロックテーブル等の前提を壊さない）。
- **設備時間割シートのセル表示だけ**、当該の日付・設備列・依頼NOについて `sub` に現れた補助者名を **和集合**し、**名前の昇順で `, ` 連結**した **1 種類の「補:…」**として出す。並べ替え・一覧性のための表示統一である。

**履歴列の黄色（need 追加人数）**（`_apply_result_task_history_need_surplus_highlight`）:

- **履歴1～n** のうち、**need の基本必要人数（req_num）を超えた人数**で割り当てたブロックに対応するセルのみ **薄黄（`FFEB9C`）** に塗る（配台不可／配台残の行全体の薄黄より **後** に上書き）。
  - **メイン割付**: `TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS` が有効 **かつ** 当該行の追加枠（`extra_max`）が **1 以上** **かつ** **採用人数 > req_num** のとき。
  - **全配台後の余力追記**（`append_surplus_staff_after_main_dispatch`）: 追記 **後** の当該ブロック人数が **req_num を超える** ときのみ（必要人数ちょうどまでの追記だけでは塗らない）。余力追記後は **履歴文字列の担当**をタイムラインの `op`/`sub` に合わせて更新する。

**履歴列のテキスト（組み合わせ表・追加メンバー）**（`_format_result_task_history_cell`）:

- **組合せ表#n**: マスタ「組み合わせ表」の **A列「組合せ行ID」**（VBA `master_組み合わせ表を更新` でデータ行ごとに 1 から採番）。`TEAM_ASSIGN_USE_MASTER_COMBO_SHEET` 有効時、**プリセット行として採用された**ロール、または **採用メンバー集合がいずれかのプリセットと一致**し **`_lookup_combo_sheet_row_id_for_preset_team`** で補完できたロールに付く。一致するプリセットが無い **純探索採用**では付かない。
- **列設定_結果_タスク一覧**（結果ブックの A:B）: 段階2は **`pandas` / `openpyxl` で毎回シートを新規作成**するため、セル以外の **図形は openpyxl では書き出せない**。**既定では**生成完了後（カレンダー日付表示整形の後）に xlwings が **TASK_INPUT_WORKBOOK** の同名シートから図形を結果ブックへ複製し、**貼り付け直後に入力側と同じ位置・大きさ**（COM の Left/Top/Width/Height、取れるとき Placement）を再設定する（`_stage2_try_copy_column_config_shapes_from_input`）。**無効化**は環境変数 **`STAGE2_COPY_COLUMN_CONFIG_SHAPES_FROM_INPUT=0`**（Excel 起動を避けたい場合など）。失敗時は警告ログのみ。
- **担当[…]**: **メイン割付まで**の担当（**余力追記**で足した名前は含めない。内部の `team` は追記後の全員に同期しているが、結果シートでは `_history_team_text_main_assignment_only` で表示から除外する）。
- **追加[名前,…]**: メインで **基本必要人数より多い人数**を採用したときの、その超過分（表示は担当の先頭から `req_num` 名のあとを **追加**とみなす）。
- **余力追記[名前,…]**: メイン完了後の余力追記で `sub` に足したメンバー。

## 10. 配台トレース（設定シート A3 のみ）

**目的**: 指定した **依頼NO** について、割付シミュレーション中に `[配台トレース task=…]` 形式の **詳細ログ**を `log/execution_log.txt` に出す（進捗が止まる原因調査用）。

**入力**

- マクロブック **「設定」シートの A 列 3 行目以降** … 依頼NOを縦に並べる（空セルはスキップ。詳細は `_read_trace_schedule_task_ids_from_config_sheet`）。**環境変数では指定しない**（旧 `TRACE_SCHEDULE_TASK_ID` は廃止）。

**集合**: 上記 A 列から読んだ依頼NOが `TRACE_SCHEDULE_TASK_IDS`。`generate_plan` **冒頭**（**マスタ skills 読込より前**）に読み込み・**INFO ログを必ず出力**する。以前は `load_skills_and_needs` の後にしか読んでおらず、**メンバー0人で段階2が即 return した場合は設定シートのログが一度も出ない**不具合があった。

**依頼NOごとのログファイル**: `[配台トレース task=…]` 行は **`log/execution_log.txt`** にも出し、あわせて **`log/dispatch_trace_<依頼NO>_r<外側ラウンド2桁>.txt`**（例: `dispatch_trace_W4-2_r00.txt`、依頼NOはサニタイズ）へ追記する。**外側ラウンド**は **0=初回のカレンダー通し**。**`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING=True`** のときのみ、**納期超過リトライで巻き戻すたびに +1**（`generate_plan` の `_outer_retry_round` に一致。既定オフでは通常 **r00 のみ**）。トレース行は **`[outer_round=00]` 形式**の接頭辞が本文に付く（`execution_log` と専用 txt で同じ文面）。**配台トレース有効時**、段階2冒頭で **`log/dispatch_trace_*.txt` をすべて削除**する。各ラウンド用ファイルのヘッダ付き**新規作成**は、そのラウンドのシミュレーションに入る **`while True` 先頭**で行う（過去実行の別依頼NOファイルが残らない）。

**制約**: ブックにシート **`配台_配台不要工程`**（openpyxl 非互換マーカー）があると、openpyxl で「設定」を読めないため **A3 以降は無視**され、**配台トレースは無効**になる（ログに理由を INFO で出す）。

## 11. デバッグ配台（設定シート B3 以降）

**目的**: 段階2で **配台の入力対象を一部の依頼NOに限定**する（全体シミュレーションを避けたいとき）。

**入力**

- マクロブック **「設定」シートの B 列 3 行目以降** … 配台対象にしたい **依頼NO** を縦に並べる（空セルはスキップ・連続空打ち切り・カンマ区切り複数可など、**読み取り仕様は A 列の配台トレースと同じ**。`_read_debug_dispatch_task_ids_from_config_sheet`）。

**動作**

- **B3 以降に 1 件も有効な依頼NOが無い**（すべて空白・読めない）… **通常モード**（`配台計画_タスク入力` の全行が対象）。**ダイアログは出さない**。
- **1 件以上ある** … **デバッグモード**。`generate_plan` 冒頭で **Windows のメッセージボックス**を出し（非 Windows では WARNING ログに同内容）、続けて **`配台計画_タスク入力` の DataFrame を当該依頼NOの行だけに絞る**（`planning_task_id_str_from_plan_row` と設定値の正規化 `planning_task_id_str_from_scalar` で突き合わせ）。絞り込み後に **0 行**なら段階2は **エラー終了**。

**トレースとの関係**: **A 列**は **ログ用トレースのみ**（§10）。**B 列**は **配台対象の限定**。両方に同じ依頼NOを書いてもよい。

**制約**: §10 と同様、openpyxl で「設定」シートを読めないときは **B 列は無視**され、**デバッグ配台は無効**（全件配台・ダイアログなし）。
