タスクの並び（割付の試行順・要約）

ここは generate_plan でのグローバル試行順の決め方と、段階2の日内ループの関係を見出し＋表で整理しています（2026-04 時点）。

1. グローバル（generate_plan 内）

条件試行順 dispatch_trial_order の決め方
キュー全行にシート列 配台試行順番（正の整数）が埋まっている その昇順。同値は planning_sheet_row_seq。EC 隣接繰り上げは行わない（手計算シート順を尊重）。
欠損行が1 行でもある task_queue.sort のキー _generate_plan_task_queue_sort_key で並べ替え（下表）→ _reorder_task_queue_b2_ec_inspection_consecutive で EC→後続を隣接 → _assign_sequential_dispatch_trial_order で 1…n。

自動ソート時のキー順（2026-04-06 改定②・欠損時のみ）:

順位キー意味（要約）
1（加工途中）途中タスクを先に
2納期基準due_basis_date（回答納期→指定納期）

2. 各日内・既定（STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1）

  1. 候補の並び: _trial_order_first_schedule_pass が、まず dispatch_trial_order 昇順で見る（主ループは _day_schedule_task_sort_key 単独では並べ替えない）。
  2. §B-2／§B-3 は完全二相:
    • フェーズ1 — 熱融着検査・巻返し（後続）以外を、連続ロールとして進める。
    • フェーズ2 — 後続行を配台。後続の入り幅は _roll_pipeline_inspection_assign_room（バッファ付きロール枠）。
  3. 本節ループ後の再走査: B-2 検査／B-3 巻返しは _run_b2_inspection_rewind_pass で日付の先頭から再スキャン。timeline_events と人・設備ブロックテーブルを再利用し、日をまたいだ占有を保ったまま詰める。
  4. 再走査中の関数: _roll_pipeline_inspection_assign_room、_roll_pipeline_b2_inspection_ec_completion_floor_dt。
  5. 設備の壁時計: machine_avail_dt を機械名占有キーで管理。同一物理機械では、試行順の整合に _equipment_line_lower_dispatch_trial_still_pending。
  6. 継続チーム優先はその日のうちに限り、翌日はまた当日候補から選び直す。

3. 各日内・フォールバック（STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0）

_day_schedule_task_sort_key 順の一本ループ。dispatch_trial_order は複合キーの一部に過ぎず、単独の最優先キーではない。

4. 納期シフト再試行の直前

対象依頼を再配台する前に、_apply_dispatch_trial_order_for_generate_plan で試行順を再確定（シート「全行そろい」優先、または自動ソート＋EC 隣接＋連番）。

5. 結果シートの見え方

結果_タスク一覧の行順の第一キーは、グローバルでソートした直後の配台試行順番（1…n）です。

日内割付・主な環境変数

  • STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST … 既定 1。配台試行順優先の _trial_order_first_schedule_pass。0 で従来の日内ソートキーループ。
  • STAGE2_SERIAL_DISPATCH_BY_TASK_ID … 既定相当 0。1 で依頼NO直列（アクティブ依頼のみ当日候補）。
  • PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE … 既定オン。§B-2 検査／§B-3 巻返しが占有する設備列では他タスクを試さない（変数名は歴史的）。
  • PLANNING_B2_EC_FOLLOWER_DISJOINT_TEAMS … 既定オン。同一依頼の EC と後続（検査／巻返し）で、履歴に出たメンバーを他側のチーム候補から除外（B-2.7）。0 / off 等で無効化すると除外しない。詳細は 配台ルール.md B-2.7。
  • 集約シート: マクロブックに 設定_環境変数（A=変数名・B=値）を置くと、段階1/2 等の Python 起動時に workbook_env_bootstrap.py が import planning_core より前に os.environ へ反映します（OS 環境に触れず運用可。VBA の TASK_INPUT_WORKBOOK は従来どおり）。VBA の 設定_環境変数_シートを確保 が段階1/2 先頭で不足行のみ追記します。段階1/2 の cmd 表示は STAGE12_CMD_HIDE_WINDOW（1=非表示・0=表示。シート B が非空なら優先、次に OS 環境、最後に VBA 既定）。設定 D3=false でも非表示指定時は headless 起動（ログ枠重ねは非表示時は行わない）。詳細は 配台ルール.md セクション8.0。
  • TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS … 未設定＝メインは基本人数のみ、全シミュレーション後に余力サブ追記。1＝従来どおりメインで req+追加上限 まで探索。
  • TEAM_ASSIGN_IGNORE_NEED_SURPLUS_ROW … 1 で need 追加人数行を全面無視（後追いもなし）。
  • TEAM_ASSIGN_*（人数優先・スラック）… 配台ルール.md セクション5。
  • 機械カレンダー: マスタ 機械カレンダー の占有は列キーに加え機械名集約キーでも machine_avail_dt に載り、_apply_machine_calendar_floor_for_date および _bump_machine_avail_after_roll_for_calendar で反映（配台ルール.md 3.2.1）。セルが数値 0 のときは空欄同様・非占有。特定時刻以降どの設備にも入らないときは午後スロットの誤占有・メイン A12/B12 終了・出勤退勤を確認。
  • 2026-04 追補（依頼切替・日次始業）: マスタ任意シート 設定_依頼切替前後時間 と 設定_機械_日次始業準備。依頼NO切替・「その日初回の加工」判定はタイムラインの加工イベントのみ（event_kind）。同一占有キー・同一依頼NOの連続加工ロールでは 加工前準備をタイムライン・壁時計下限にも載せない。日次始業は メイン A15（定常開始）が有効なとき勤怠と無関係に [定常開始, 定常開始+分) の壁時計ブロック（設備占有・加工不可）。A15 無効時のみ従来どおり代表スキルOPの勤務線で forward。加工前準備・後始末は引き続き代表OP／直前主の勤務線。スキルOPが誰もいないときのフォールバックは 配台ルール.md 3.2.1.1。結果_設備ガントの日次始業帯は黄色系（machine_daily_startup）。詳細は 配台ルール.md 3.2.1.1。
  • 2026-04 追補（休憩・終業直前）: 終業まで残り ASSIGN_END_OF_DAY_DEFER_MINUTES 分以下かつ ASSIGN_EOD_DEFER_MAX_REMAINING_ROLLS ロール分以上は回せないときは新たに加工を始めない（ASSIGN_END_OF_DAY_DEFER_MINUTES は Python 既定 45 分、0 で無効）。実装は (1) 依頼全体の残ロールが閾値以下で当日開始拒否、(2) その開始から終業までに収容できるロール数が閾値未満で候補却下。設備占有キー上の直前加工が同一依頼NOの連続ロールでは (1)(2) を適用しない（_eod_same_request_continuation_exempt）。休憩帯内に落ちた試行開始は休憩終了へ繰り下げ（下限を再適用）。帯外でも 1 ブロック分の連続実働が次の休憩まで足りないときはその休憩終了まで繰り下げ。それでも次の休憩までの連続枠に収まらない候補は却下（休憩またぎの連続配台なし）。旧 ASSIGN_DEFER_MIN_REMAINING_ROLLS / ASSIGN_PRE_BREAK_DEFER_GAP_MINUTES は無視。設備時間割の 14/54R 形式は累積完了/総ロール（左は残りではない）。詳細は 配台ルール.md 改定（2026-04）・4.2。

need 追加人数（余力・メイン後追記）

master.xlsm の need で、基本必要人数の直下（目安 Excel 5 行目付近）の「配台時追加人数／余力時追加人数」列は、工程×機械ごとに req を満たしたあと追加で入れられる人数の上限です。

  • 既定: メインの割付シミュレーションではこの上限を人数探索に使わず req_num ちょうどで組む。シミュレーション完了後、加工イベントのタイムラインブロックだけを対象に、枠が空き、かつその時間帯に他ブロックと重ならない・スキル OP/AS 適合・eligible の人を サブ（sub）に追記（append_surplus_staff_after_main_dispatch）。日次始業・後始末・準備の3種はサブ追記対象外（日次始業は人なし。準備・後始末の補はメインで確定した加工チームに合わせる）。
  • 時刻は変えない（表示・同席記録向け。メインで決めた end_dt のまま）。
  • 詳細・表は 配台ルール.md セクション 3.5。

  • 段階2: build_task_queue_from_planning_df はまずこの列の数値を unit_m に使う。空・0・不正のときだけ従来の製品名推定へフォールバック。

在庫場所（配台計画_タスク入力）

  • 列: 在庫場所（TASK_COL_STOCK_LOCATION）。加工計画DATA に同名の列があるとき、段階1で 原反投入日の右隣に出力されます（SOURCE_BASE_COLUMNS / plan_input_sheet_column_order）。
  • 配台計算: 参照表示のみ。キュー（build_task_queue_from_planning_df）の内部キーには載せません。

原反投入日_上書き（配台計画_タスク入力）

  • 列: 原反投入日_上書き（PLAN_COL_RAW_INPUT_DATE_OVERRIDE）。参照列 （元）原反投入日_上書き は括弧内に列 原反投入日 を表示。
  • 段階2: 日付があるときその値を raw_input_date として採用（開始下限・同日 13:00 下限・備考 AI の start_date（該当行）との max 等）。空のときは列 原反投入日 のみ。
  • 結果_タスク一覧の 原反投入日 表示も、上記と同じ優先で日付を出します。詳細は 配台ルール.md 2.1・7.3。

A. 全加工依頼に適用する追加ルール

加工内容列による工程順序（同一依頼NO）

  • 列: マクロブック 加工計画DATA および 配台計画_タスク入力 の 加工内容（段階1でコピーされます）。
  • 書式: カンマ , 区切り。各要素は 工程名に対応（例: EC,検査）。
  • 制約: 同一 依頼NO で、いずれの行も 配台不要でない とき、リストの 前の工程がすべて完了（残ロール0）するまで、後の工程には割り当てません。
  • 例外: 工程名が加工内容に含まれない行は、この順序制約の対象外です。配台不要行はキューに入らず、前工程としてもブロックしません。ただし加工内容にトークンがある依頼では、結果シートでタスクIDセルを赤背景・白文字にして不整合を示します（下記セクション）。
  • 照合: 工程名は NFKC・空白除去で加工内容の各トークンと突き合わせます。
  • 日内の動き: 同一日に、加工内容順（rank）・シート行順・§B-1 試行順・§B-2 パイプライン検査の前倒し・配台試行順・結果用ソートを組み合わせてタスクを複数周回し、各タスクあたり最大1チャンクずつ割り当てるため、前後工程を同日で交互に進められます。

同一依頼NOの直列（加工内容に rank が無い行）

  • 原則: 同一 依頼NO の 別工程（別行）は、設備が違っても 同時加工しない。配台計画シートを上から読んだ 行順（same_request_line_seq）で、上の行の残ロールがなくなるまで下の行に割り当てません。
  • §A-1 との関係: 両行に加工内容由来の rank があるときは rank だけで前後を決め、片方でも rank が無いときは 行順で直列化します。
  • §B-2 / §B-3: EC×後続（検査または巻返し）パイプラインはロール枠・設備占有など別フィルタで制御。同一依頼の前後関係は 配台ルール.md B-2.6・B-3・_trial_order_flow_eligible_tasks を参照。
  • 実装: planning_core/_core.py の build_task_queue_from_planning_df と _task_blocked_by_same_request_dependency。

B. 個別に指定する追加ルール

B-1 — 検査×熱融着機湖南で加工途中なら試行順優先

対象: 工程名 検査・機械名 熱融着機　湖南で 加工途中（in_progress）の行（§B-2 の後続行と同一条件）。

  • グローバル: b_tier = 0（§B-1）。対象行は常に加工途中のため §2.0 の先頭キーでは一般行と同様に並び、同一納期帯では B-2/B-3 帯より先。対象行同士は b2_queue_sub が同値のため need 列→依頼NO。試行番号はその後の付与。
  • 日内（STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0）: §2.0 と同趣旨の先頭キーのあと、dispatch_trial_order を §A-1 の rank より前に置く（_day_schedule_task_sort_key）。
  • §B 群内: B-1 を最優先。B-2 と B-3 は同一の b_tier = 1 帯。詳細は 配台ルール.md §B-1。

B-2 — EC → 検査（ロール単位パイプライン）

次の 工程名・機械名の組み合わせに限り、同一依頼NO内でロール単位の並行を認めます（コード内定数）。

役割工程名機械名
先行（EC）ECEC機　湖南
後続（検査）検査熱融着機　湖南
  • バッファ: EC 側で 2 ロール完了するまで、検査には 1 ロールも割り当てません（段取り時間の確保）。定数: ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2。
  • EC 行が同一依頼に無い検査: ロール枠を事実上かけず（ROLL_PIPELINE_INSP_UNCAPPED_ROOM）、通常どおり need・スキルで配台します。
  • パイプライン: バッファ後は、検査の累計完了ロール数が常に次の上限を超えないようにします。
    進行中: ec_done（EC 完了ロール数）・insp_done（検査完了）とすると、原則 insp_done ≤ max(0, ec_done - 2 + 1)。EC が 2 ロール終わると検査 1 ロール目が可能になり、その後は EC が 1 ロール進むごとに検査も最大 1 ロール進められます。
    EC 全ロール完了後: 同一依頼NOの EC パイプライン行がすべて残量ゼロのときは、上記だけでは検査が 1 本足りなくなるため、上限を ec_done まで引き上げ、検査も EC と同じ本数まで完了できます（_pipeline_ec_fully_done_for_tid / _roll_pipeline_inspection_assign_room）。
  • 当日候補への入り方: _trial_order_flow_eligible_tasks で §A 同一依頼ブロック・_roll_pipeline_inspection_assign_room（ロール枠）・（既定で）設備占有・試行順強制を順に判定。通過後にチーム割付。前進・再走査の両方で同ロール枠と時刻下限が効きます。詳細は 配台ルール.md B-2.6。
  • 担当者の排他（同一依頼NO）: 既定では EC 行と検査行で 履歴に出たメンバーは他側の候補から除外（_filter_capable_members_b2_disjoint_teams・NFKC 突合）。PLANNING_B2_EC_FOLLOWER_DISJOINT_TEAMS でオフにすると除外しません。配台トレース対象依頼NOでは、除外が発生した時点で ブロック判定: B-2担当者分離 ... を出力。さらに候補試行時に 候補固定: 担当OP指名=...・候補却下: ...（OP不在/実働不足など）も出力。
  • 変更方法: planning_core/_core.py の ROLL_PIPELINE_EC_PROCESS 等を編集してください。

B-3 — EC → 巻返し（ロール単位パイプライン・B-2 同趣旨）

同一依頼NOで B-2 と同じロールパイプライン挙動（完全二相・ロール枠・リワインド・担当者排他・設備占有）を適用します。B-2 の「検査」に相当する工程を 巻返し、機械名は先行 EC と同じ EC機　湖南 です。

役割工程名機械名
先行（EC）ECEC機　湖南
後続（巻返し）巻返しEC機　湖南
  • 実装: タスク辞書の roll_pipeline_rewind。定数 ROLL_PIPELINE_REWIND_PROCESS / ROLL_PIPELINE_REWIND_MACHINE。完了ロール集計は _pipeline_b2_follower_roll_done_units（検査＋巻返しを合算）。
  • 詳細: 配台ルール.md §B-3 および §B-2.2〜B-2.7（語の読み替え）。

納期超過検出と後ろ倒し再試行（任意・既定オフ）

既定: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = False。計画基準を過ぎても 巻き戻し再試行は行わず、カレンダーは 1 通しで完走（配台試行順を正とする方針）。

検出（_collect_task_ids_missed_deadline_after_day）: ある日のシミュレーション終了時に、残量あり・due_basis_date あり・start_date_req ≤ 当日・当日 ≥ due_basis_date の依頼NOを集める（納期当日を含む）。フラグがオフのときは 検出のみで再試行に入らない。

再試行（STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = True のとき）: 上記依頼について 内部 due_basis_date のみ +1 日、当該依頼のタイムラインと割当を巻き戻し、全日程を最初から再走査。表示用 due_basis_date_result_sheet は元日付のまま。上限 STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（依頼NOごと）。各ラウンド前に勤怠拡張日はマスタへ戻す。

  • 従来挙動を有効にする: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = True。
  • 上限: STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（既定 5、依頼NOごと。全体の合計回数ではない）。再試行オン時のみ意味を持つ。
  • シフトしない: 依頼全体で due_basis_date を持つ行が無い場合は、後ろ倒し対象にしません。
  • 勤怠最終日: +1 日がマスタ最終計画日を超える行がある依頼は、そのラウンドではシフト対象外になり得ます。

結果_タスク一覧・タスクIDの赤表示（加工内容との整合）

段階2が出力する production_plan_multi_day_*.xlsx の 結果_タスク一覧 で、元データの 加工内容（カンマ区切り）と行の 工程名 が一致しない可能性がある行の タスクID セルだけを、赤塗り・白文字にします。

  • 加工内容の取り方: セクション A と同じく、同一依頼NOについて表の上の方で最初に現れた非空の加工内容をトークン化します。
  • ハイライトしない: その依頼についてトークンが1つも無い（加工内容が未記入のみ等）場合は、比較基準が無いため色を付けません。
  • ハイライトする: トークンがあるのに、工程名が（NFKC 等の正規化後）いずれのトークンとも一致しないとき。
  • 配台不可／配台残の行全体の薄い黄より後にタスクIDだけ上書きするため、該当セルは赤が優先されます。
  • 結果_設備毎の時間割へのリンク: タスクIDセルに内部ハイパーリンクを付け、時間割で当該タスクが最初に現れるセルへジャンプ（未割当のみで時間割に無い行はリンクなし）。_apply_result_task_id_hyperlinks_to_equipment_schedule。
  • 履歴列（追加人数・色）: req_num を超えた人数のブロックのみ薄黄（FFEB9C）。メインは TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS ＋追加枠あり ＋採用人数>req_num。余力追記は追記後の人数が req_num を超えるときのみ（定員までの追記のみでは塗らない）。履歴の担当表記は追記後に同期。_apply_result_task_history_need_surplus_highlight。
  • 履歴テキスト: 組合せ表#nはプリセット行採用時に付くほか、探索のみで ID が欠けていても採用メンバー集合がいずれかのプリセットと一致すれば _lookup_combo_sheet_row_id_for_preset_team で補完（履歴1〜n 共通）。一致プリセットが無ければ付かない。プリセットは成立分をすべて候補に載せ、探索とまとめて最良を選ぶ。担当はメイン確定分のみ（余力追記は余力追記[…]）。追加[…]はメインで人数超過。_format_result_task_history_cell／_history_team_text_main_assignment_only。
  • 列設定_結果_タスク一覧（結果ブック）: 段階2は A:B を毎回再生成するが、既定で生成後に TASK_INPUT_WORKBOOK 側の同名シートから xlwings で図形（ボタン等）を複製し、位置・サイズを入力側と同じに戻す（_stage2_try_copy_column_config_shapes_from_input）。無効化は STAGE2_COPY_COLUMN_CONFIG_SHAPES_FROM_INPUT=0。

実装: planning_core/_core.py の process_content_mismatch、_process_name_matches_kakou_content_tokens、_apply_result_task_task_id_content_mismatch_highlight、_apply_result_task_id_hyperlinks_to_equipment_schedule、_apply_result_task_history_need_surplus_highlight、_format_result_task_history_cell、_lookup_combo_sheet_row_id_for_preset_team、_stage2_try_copy_column_config_shapes_from_input。

列 配完_回答指定16時まで: 配完_加工終了が、回答納期の日付＋既定 16:00（PLAN_DUE_DAY_COMPLETION_TIME）以前かを判定。回答納期が空なら 指定納期＋16:00 で判定。値は はい／いいえ／未割当／納期なし（回答・指定とも日付なし）。_result_task_plan_end_within_answer_or_spec_16_label。いいえのセルは赤塗り・白文字・太字（_apply_result_task_plan_end_answer_spec_16_no_highlight）。マクロブックへの xlsx 取り込み直後は VBA 結果_タスク一覧_配完回答指定16時_いいえを強調で同趣旨を再適用。

配台トレース（設定シート A3 のみ）

指定 依頼NO について [配台トレース task=…] 詳細を log/execution_log.txt に出します。

  • マクロブック 「設定」シート A 列 3 行目以降に依頼NOを縦に（_read_trace_schedule_task_ids_from_config_sheet）。環境変数では指定しない。
  • generate_plan 冒頭（skills 読込より前）に読み、必ず INFO ログ。メンバー0で即終了したときも設定シートの読み取り結果がログに残ります。
  • [配台トレース task=…] 行は log/execution_log.txt に加え、log/dispatch_trace_<依頼NO>_r<外側ラウンド2桁>.txt（例 dispatch_trace_W4-2_r00.txt、名はサニタイズ）へ追記。外側ラウンドは 納期巻き戻し再試行オン時のみリトライのたび増え（既定オフでは通常 r00 のみ）。文頭に [outer_round=00] が付く。段階2冒頭で log/dispatch_trace_*.txt を全削除。各ラウンドのヘッダ付きファイルは generate_plan の外側 while True 先頭で新規作成。
  • シート 配台_配台不要工程（openpyxl 非互換マーカー）があるブックは「設定」を openpyxl で読めず A3 以降は無視（配台トレースは無効）。

詳細は 配台ルール.md セクション 10。

デバッグ配台（設定シート B3 以降）

段階2の配台対象を 一部の依頼NOに限定し、全体シミュレーションを避けたいときに使います（generate_plan 冒頭の _read_debug_dispatch_task_ids_from_config_sheet）。

  • 入力: マクロブック 「設定」シートの B 列 3 行目以降に、配台対象にしたい 依頼NO を縦に並べます（空セルはスキップ。連続空で打ち切り・カンマ区切り複数可など読み取り仕様は A 列の配台トレースと同趣旨）。
  • 動作:
    • 有効な依頼NOが 1 件も無い（空白のみ等）… 通常モード（配台計画_タスク入力 の全行が対象）。ダイアログは出しません。
    • 1 件以上ある … デバッグモード。Windows のメッセージボックスを出し（非 Windows では WARNING ログ）、続けて 配台計画_タスク入力 を当該依頼NOの行だけに絞って配台します。絞り込み後に 0 行なら段階2は エラー終了です。
  • 配台トレースとの関係: A 列はトレース（ログ用）だけ、B 列は配台対象の限定です。両方に同じ依頼NOを書いても構いません。
  • 制約: シート 配台_配台不要工程（openpyxl 非互換マーカー）があるブックは「設定」を openpyxl で読めず B3 以降は無視（デバッグ配台は無効で、全件配台・ダイアログなし）。

詳細は 配台ルール.md セクション 11。
