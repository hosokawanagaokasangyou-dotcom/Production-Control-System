工程管理AI配台システム — Cursor 活用による社内運用・保守・開発（経営層向け）

文書の目的：ソフトウェア会社への外注に頼らず、社内で運用・保守・開発を進める場合に、Cursor（AI 支援型の開発環境） をどう位置づけ、経営・管理として何を決めればよいかを整理する。

想定読者：経営層、部門長（コードの詳細は不要）。社内に情報システム専任部署がない場合は、PC・ネットワーク・ツール導入は 経営または窓口担当 が 外部の IT 支援 と相談して決める想定で読む。

関連文書：経営層向け説明（HTML） ／ Markdown、要件定義書（HTML）

文書バージョン：1.2（2026-03-28）

目次： 1. 方針の整理（1.1 選任・1.2 教育）／ 2. Cursorとは ／ 3. 具体策 ／ 4. メリット・デメリット ／ 5. 承認事項 ／ 6. まとめ

1. 方針の整理：外注しない場合に経営が押さえること

観点社内推進で必要なこと
責任の所在 仕様変更・不具合・セキュリティ対応の 最終責任は社内（外部ベンダーに丸投げできない）。
人的リソース Excel／VBA と Python のいずれかに触れられる人（または習得する時間を与えられる人）が最低限必要。AI は 補助 であり、無人運転ではない。
情報ガバナンス ソース・マクロブック・認証情報が どこにあり、誰がアクセスできるか を規程化する。
継続性 特定個人依存（バス係数）を Git・ドキュメント・ペア作業 で下げる。

1.1 担当者の選任の考え方

選ぶ際に見ること（必須スキルより「適性・環境」）

観点内容
業務理解 配台・加工計画・勤怠マスタなど 現場の表が何を意味するか 説明できる人。仕様の正しさを最終判断しやすい。
表計算の習熟 複雑な Excel を日常的に触り、式・参照・シート構造を追える人。VBA 未経験でもここから入る例は多い。
学習意欲と継続時間 Python や Git は 数週間〜数ヶ月の学習が現実的。週あたりの確保時間（例: 半日〜1日相当）を 上司と合意してから選任する。
慎重さ 本番データを扱うため、変更前に確認・バックアップを取る姿勢が取れる人。

よくあるパターン

  • 現場の Excel 担当（関数・ピボットに強い）＋ プログラムを これから学ぶ 若手・中堅。
  • 工場／生管のベテラン（業務の芯）と 補助で実装する若手の 2 名ペア（どちらも「正しいか」の責任を分担）。
  • いずれも難しい場合、パートタイムの外部メンター（週1時間のレビューだけ）と組む選択肢もある。

避けた方がよい選び方

  • 兼務過多で学習時間がゼロになる人だけを指名する。
  • バックアップ要員を置かず 1 名に集約する（退職・休職で止まる）。

経営・部門長が決めること

  • 正（プライマリ）と副（セカンダリ） の 2 名体制を目標にすること。
  • 選任を 役職ではなく役割として明示し、評価・工数に反映する（「余暇でやる」前提は続かない）。

1.2 教育・習熟の進め方（例）

段階を踏むと負荷が分散しやすい（順序は組織で調整可）。

段階内容の例期間の目安（目安）
1 本システムの 業務フロー（段階1・2、マクロブックの役割）を 経営層向け説明・要件定義書 で把握する。 数日
2 Git の基本（クローン、コミット、履歴を見る、戻す）。開発用コピーだけ触るルールを徹底。 1〜2 週
3 Excel VBA — 既存マクロの 読み方、ブックと *_VBA.txt の対応、小さな表示変更から試す。 2〜4 週
4 Python — 変数・関数・ファイル読み書きの基礎。task_extract_stage1.py など 短いエントリから読む。 1〜2 ヶ月（並行可）
5 Cursor — チャットで「この関数の役割は？」「このエラーの意味は？」と 読解支援に使い、人が承認してからコミットする。 以降も継続

教育の手段の例

  • 自習＋質問会：週1回・1時間の 固定ミーティング（社内または外部メンターへの質問枠）。
  • ペア作業：慣れた者が 画面共有しながら 1 本の修正を通す（ノウハウが移転しやすい）。
  • 市販・オンライン講座：Python 入門、Git 入門（無料教材で足りることも多い）。
  • 仕事で小さく回す：最初から planning_core.py 全体ではなく、ログ文言変更・設定値の説明追記など リスクの低いタスクを課題にする。

経営が見る指標（例）

  • 副担当が同席できる状態になったか（バス係数の低下）。
  • 本番以外のフォルダで、1 件以上の変更を Git に残せたか。
  • 重大障害なく 1 四半期 運用できたか。

2. Cursor とは何か（非エンジニア向け）

  • Cursor は、プログラムを書くための PC 上の作業環境（エディタ） の一種で、生成AIを組み込んで、「説明をするとコード案を出す」「既存コードの修正を提案する」などを 対話しながら 進められる製品です（SaaS／サブスクリプションのサービス形態が一般的）。
  • 本システムのような「Excel マクロ ＋ Python ＋ 設定ファイル」のリポジトリを フォルダごと開き、変更・調査・小規模機能追加を 社内担当者が加速する用途に向く。
  • 経営層が誤解しやすい点：Cursor は 自走するシステムではない。入力した内容やコードの一部が クラウド上のAI処理に送られる場合がある（製品の設定・契約・「プライバシーモード」等で挙動が変わる）ため、導入前に 利用規約とプライバシー設定 を確認する（必要なら 外部の IT 専門家 に相談）。

3. 社内運用・保守・開発を推し進めるための手段（具体策）

3.1 バージョン管理（Git）を「正」とする

  • ソース（python/、VBA テキストバックアップ、要件定義）を Git リポジトリで管理する。
  • 効果：いつ・誰が・何を変えたかが残る。誤修正から 戻せる。外注に戻すときも 引き渡し単位が明確。
  • 経営メッセージ：「USB 一本の最新版」運用は 監査・障害時に不利。最低限 Git を正とする 運用ルールを決める。

3.2 バックアップと「本番フォルダ」のルール

  • マクロブック（.xlsm）・master.xlsm・output の保存場所を決め、クラウドストレージの履歴や 定期コピーを併用する。
  • 本番で動かすフォルダと 開発用のコピーを分ける（誤った Python を本番パスで実行しない）。

3.3 Cursor の位置づけ：開発プロセスのどこで使うか

フェーズCursor の典型的な使い方人の役割
調査 巨大な planning_core.py の該当箇所を探す・要約する 問い（何を直したいか）を言語化する
小改修 バグ修正・ログ出力・条件分岐の案を生成 内容を読み、動作確認し、採否を決める
VBA マクロの書き換え案（テキストバックアップとセット） Excel 上での動作確認・配布手順
ドキュメント 要件定義・手順書の下書き 事実確認と 責任ある承認

3.4 品質を保つ最低限の習慣

  • 変更は小さく：一度に大量改変せず、コミット単位を細かくする。
  • 再現手順：不具合報告は「操作手順・期待結果・実結果」を残す。
  • 影響範囲：配台ロジック・Gemini 連携・Excel 取り込みは 連鎖しやすいため、変更前に 要件定義書や LOG で範囲を確認する。

3.5 セキュリティ・コンプライアンス（必須の検討項目）

  • Cursor 利用時にコードやパスがクラウド側で処理される可能性 — 経営または実装担当が 利用規約・プライバシー設定（エンタープライズ契約の有無など）を確認する。判断が難しい場合は 外部の IT 専門家 に相談する。
  • Gemini 認証 JSON・API キー — リポジトリに 直書きしない、.gitignore と 暗号化手順（既存の encrypt_gemini_credentials.py 等）を運用に含める。
  • 個人情報・顧客情報を含む実データを AI に貼らない ルールを徹底する（匿名化サンプルで検証）。

3.6 役割分担の例（小規模組織）

役割内容
オーナー 優先順位・リリース判断・ベンダー契約（Gemini 等）
実装担当 Cursor＋Git でコード・VBA 保守、Excel 動作確認
業務担当 仕様の正しさ・受入試験・現場問い合わせの一次窓口
PC・ネットワーク（社内に情シスがない場合） 経営または窓口担当が、標準PC・インターネット・Cursor 導入可否を決める。地域の IT 支援事業者等へ スポット委託する運用も一般的。

4. ソフト会社と比べたときのメリット・デメリット

社内＋Cursor 推進外注中心
メリット 小変更の 待ち時間が短い、業務知識を直接反映しやすい、ランニングの SI 費を抑えやすい可能性 大規模開発・契約で応答時間や復旧の目安が数値化された保守（いわゆる SLA＝サービス品質を契約で約束する形）、第三者監査向き
デメリット 属人化・品質ばらつき、休職・退職リスク、AI 利用の ルール整理が自社負担 コスト・要件の言語化が必要、変更の 依頼〜見積サイクル

ハイブリッド（日常は社内、年次レビューや大改修だけスポット委託）は現実的な選択肢である。

用語メモ（SLA）：Service Level Agreement の略で、「障害が起きたら 何時間以内に連絡が返るか」「どの程度の時間で復旧を目指すか」などを 契約書に明記した保守のこと。社内運用ではこの約束を 自分たちで決め、自分たちで守る必要がある。

5. 経営層が承認・決定しておくとよい事項（チェックリスト）

  • 社内オーナー（技術ではなく 事業責任）の指名
  • Git リポジトリの保管場所（社内サーバ／許可されたクラウド）と アクセス権
  • Cursor（および利用する AI） の 利用可否 と 利用規約・プライバシー設定の確認（必要なら 外部 IT 専門家 に相談）
  • 本番フォルダと 開発フォルダの分離、マクロブックの配布手順
  • バックアップ頻度と リストア試験の有無
  • 外注に戻す条件（例：人月換算で割に合わない規模、セキュリティ事件、キーパーソン退職時）

6. まとめ

ソフト会社へ依頼せず Cursor を軸に社内で運用・保守・開発を進めることは、小さな改善の速度と 業務密着という面で有効になり得る。その一方で、責任・セキュリティ・Git による再現性は経営・管理層が明文化し、AI は補助手段であることを組織で共有することが成功の条件である。

本紙は方針・検討のたたき台である。Cursor 等の製品仕様・契約条件は各社の最新情報を正とする。製品名は説明のための例示であり、特定製品の推奨を意味しない。
