1. このシステムは何をするか（一言）

工場の 「いつ・どの機械で・誰が・何をするか」 を、加工計画データと勤務・スキル情報から 自動で組み立て、日次〜複数日の生産計画表と担当者別の予定を Microsoft Excel 上にまとめる仕組みです。現場が慣れ親しんだ表形式のまま、配台（割付）作業の負荷を下げることを狙います。

2. なぜ必要か（ビジネス上の位置づけ）

観点説明
現場負荷 依頼・納期・設備・人の制約が重なると、配台調整は時間がかかり属人的になりやすい。
変更・人員の急変 依頼内容や担当者が急に変わり再配台が必要な場面で、手作業に比べ再配台の難しさと工数の増大を抑え、回避しやすくする。
見える化 設備ごとの時間割、メンバーごとの予定を一覧化し、調整・説明の材料にする。
ルールの統一 優先度や納期、スキル条件などを、システム側のルールとして繰り返し適用しやすくする。
例外の扱い 「この案件だけ特別に」といった指示を、画面上の備考やメインシートの記載から反映する設計がある（詳細は要件定義書）。

3. 利用の流れ

現場の配台は、まず 「いま配台すべき仕事を揃える」、次に 「機械と担当者に振り分ける」 という流れになります。本システムは、この流れを Excel 上の表 でつなぎ、2 つの作業に分けて支援します。

  1. ステップ1 — 未完了の仕事を、配台用の入力表に並べる
    加工計画などに書かれた仕事のうち、まだ終わっていないものだけを抜き出し、システムが読み取る 入力表 に並べます。この表が、その日・その期間の やることリスト になります。
  2. ステップ2 — 割付案を計算し、計画表として出力する
    入力表に加え、マスタ（スキル、必要人数、その日の出勤など）を組み合わせて、どの機械に何をいつ、誰にどの仕事を 割り当てるかを計算します。結果は 生産計画表 と 担当者別の予定 として Excel に出力されます。

人の判断を挟むときは、ステップ1の表に 上書きしたい内容 や 特別な指示 を書き足してからステップ2を実行すると、配台案に反映されます。急ぎのときはステップ1と2を 続けて 実行してかまいません。

4. 「AI」は具体的に何をしているか

本節の「AI」には、① システムをつくる側の開発支援 と ② 稼働時に Excel やマスタの文言を解釈する Google Gemini（クラウド API） の二層があります。

4.1 開発について（コード作成・バイブコーディング）

当システムの ソースコードの作成・改修には、対話型の AI 支援開発（いわゆる バイブコーディング：意図や仕様を自然言語で伝えながら、AI と対話して実装を生成・調整するやり方）を活用しています。

メリットと意味：

  • 仕様の言語化から実装までの距離を短くできるため、現場の要望やルール変更を、比較的早くプロダクトに反映しやすい。
  • 現場の配台のやり方や Excel 上の運用ルールを、システムの仕様・プログラムの動きに落とし込むやりとりは、選任のエンジニアを配置しなくともよい形で継続しやすくなる。必要であれば、実演可能です。
  • 繰り返し起きる定型の実装・ドキュメント整備を支援し、保守コストの抑制に資する。

開発ツール（バイブコーディング）

項目内容
使用ツール Cursor（AI を統合したコードエディタ）。上記のバイブコーディングで使用しています。

4.2 稼働時に任せていること（Gemini API）

稼働時は、Google が提供するクラウド上の生成AI（Gemini）API を利用し、インターネット経由でサービスにアクセスします。

特記事項・備考の AI 解析については、セルに書かれた文章を AI が解釈し、プログラム側はその 解析結果をパラメータ（構造化された値・フラグ・日付など）として受け取り、配台ロジックや表示に渡します（人が細かい IF 文を逐一書かなくても、表の文言から意味を取り出す用途）。

現時点の試作バージョンでは、文言の解釈をパラメータ化することに課題があるため、オプション扱いとなっています。

主に任せていることの例：

対象・領域Gemini に任せること（概要）
配台計画上の 備考・特記事項 上記のとおり、解析結果をパラメータで受け取る
配台不要 としたい工程 条件を、機械が処理しやすい形に整理する
勤務や休暇まわりの備考 読み取り補助
メインシートの 自然言語による全体指示 解釈（例: 特定の案件の扱いを変える）

4.3 利用前提とコスト見込み（Gemini API）

重要：本システムの設計は、Gemini API を利用できる状態を前提としています。API キー（利用権限）を用意し、社内の情報セキュリティ方針に沿って管理する必要があります。

料金・枠の見込み：現時点の 計測に基づくと、無料枠の範囲内での運用が見込める水準です。一方、安定稼働（クォータや優先度の確保など）のためには、有料プランへの登録が必要となる場合があります。有料利用に移行する場合でも、利用上限（キャップ）を設定できるため、想定外の請求を抑えた運用が可能です。

5. システムの主要パーツと役割

いま表現しているもの実際のイメージ
Excel（マクロ付きブック） 画面・ボタン・表の置き場。現場が触る主たる場所。
マスタ用 Excel スキルや出勤など、計算の土台となるデータ。

6. 期待効果と留意点・リスク

<期待効果>

  • 配台表作成の 作業時間短縮、担当者による ブレの低減（ルールの再現性）。
  • 納期・優先度・スキル など複数条件を同時に考慮した案を素早く出せる。
  • 出力結果が慣れ親しんだ Excelであり、二次加工が容易。

留意点・リスク（経営・管理側の論点）

項目内容
クラウドAI依存 Gemini 利用には ネットワーク と 契約・キー管理 が必須。止まると設計どおりの機能が使えない。
コスト API の 従量課金・枠 は利用量次第。運用ルール（誰がどれだけ実行するか）の整理が望ましい。
情報の取り扱い 送信・保存の範囲は、個人情報・顧客情報・機密の観点で社内規程と照合すること。通信は通常 TLS（HTTPS）で暗号化されるが、データはクラウド上で処理され、保存・学習利用などの扱いはプラン・契約で異なる。無料・有料の別だけで「送信がない」「全面に保護される」とは限らないため、Google の最新の取り扱いと契約を確認すること。
Excel 前提 大規模な同時利用やモバイル中心運用には向かない構成。拡張時は Web 化など別アーキテクチャが検討対象になり得る。
環境の均一性 PC ごとにセットアップ差があると「動く／動かない」が分かれやすく、障害の発生も起きやすい。配布手順の整備が重要で、関係者が協調して環境をそろえる取り組みが欠かせません。

7.5 開発経緯と、ソフトウェア委託／内製の判断について

当初の想定と、実際に進んだ内容

  • 当初は、骨格を作成したうえで外部業者へ依頼する方向の想定でした。
  • 一方で、外部委託の方針は確定しておらず、また「骨格」の定義（何を指すか・どこまでが範囲か）も不明瞭でした。

その結果、特別ルールの指定漏れや不具合を含みつつも、完成形に近いイメージまで本格的にコーディングが進む形となりました。

開発体制とツール選定（Cursor）

開発担当者の専門は Java に偏っています。

3〜6 か月程度の期間をいただけるのであれば、AI によるフルバイブコーディングに頼らず、ロジックを完全に把握したうえでの実装も選択肢でした。

しかし今回は、骨格づくりにそれだけの期間を確保できない前提です。短期間で開発を進めるため、AI コードエディタ Cursor を初めて本格採用する方針をとりました。

ソフトウェア会社へ委託する場合の前提

現状、コードのロジック把握が十分に追いついていない部分があります。

そのため、ソフトウェア会社へ依頼する場合でも、おおむね次の流れになる見込みです。

  • Cursor で生成・編集した成果物をベースに、修正・追記しながら要件を固める
  • その結果、要件定義書の整備にも膨大な時間がかかる

当面の見立て（永続性・費用・運用）

永続性・保守性の観点では、ソフトウェア会社による再作成・再実装が望ましい一方、かかる費用は大きいと考えます。

当面は、このまま開発を完走させ、社内の開発リソースを増やす方策を模索しながら運用するのが現実的ではないか、という見立てです。

業務・工場稼働への影響

本システムが利用できない場合でも、従来どおり手動の配台に戻すだけです。

工場の稼働そのものに致命的な支障が生じるものではないと考えています。

9. まとめ

本システムは、工場の配台業務を Excel 中心の運用のまま支援し、Google Gemini による文章・ルールの解釈を組み合わせたものです。業務効率と再現性の向上が期待できる一方、保守を内製か外注かを選択する必要があり、経営判断が必要です。

本紙は意思決定・説明用の要約です。§8 の金額は保証しません。正式な金額は委託先の見積を正としてください。
