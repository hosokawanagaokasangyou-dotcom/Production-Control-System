1. このシステムは何をするか（一言）

工場の 「いつ・どの機械で・誰が・何をするか」 を、加工計画データと勤務・スキル情報から 自動で組み立て、日次〜複数日の生産計画表と担当者別の予定を Microsoft Excel 上にまとめる仕組みです。現場が慣れ親しんだ表形式のまま、配台（割付）作業の負荷を下げることを狙います。

2. なぜ必要か（ビジネス上の位置づけ）

観点説明
現場負荷 依頼・納期・設備・人の制約が重なると、配台調整は時間がかかり属人的になりやすい。
見える化 設備ごとの時間割、メンバーごとの予定を一覧化し、調整・説明の材料にする。
ルールの統一 優先度や納期、スキル条件などを、システム側のルールとして繰り返し適用しやすくする。
例外の扱い 「この案件だけ特別に」といった指示を、画面上の備考やメインシートの記載から反映する設計がある（詳細は要件定義書）。

3. 利用の流れ（業務の言葉で）

大きく 2ステップ です。

  1. ステップ1 — やることリストの整理
    加工計画に載っている仕事から、まだ完了していない仕事を取り出し、配台用の入力表に並べます。
  2. ステップ2 — 配台案の自動作成
    入力表と、マスタ（スキル・必要人数・出勤など） を踏まえて、機械への割付と担当者の割付を計算し、生産計画表や個人別スケジュールとして Excel に出力します。

担当者はステップ1のあと、必要に応じて 上書き欄や特別な指示 を表に書き込み、ステップ2でその内容が反映される運用を想定しています。ステップ1と2は 続けて実行 することもできます。

4. 「AI」は具体的に何をしているか

ここでいう AI は、Google が提供するクラウド上の生成AI（Gemini） を利用します。インターネット経由でサービスにアクセスします。

主に任せていることの例（人が細かくルールを書かなくても、文章や表の文言から意味を取り出す用途）：

  • 配台計画上の 備考・特記事項 の解釈
  • 配台不要 としたい工程の条件を、機械が処理しやすい形に整理すること
  • 勤務や休暇まわりの備考 の読み取り補助
  • メインシートの 自然言語による全体指示（例: 特定の案件の扱いを変える）の解釈

重要：本システムでは Gemini を利用できる状態を前提 に主要機能が設計されています。API キー（利用権限）を用意し、社内の情報セキュリティ方針に沿って管理する必要があります。利用には 通信料に加え、Google の課金・利用枠 が発生し得ます（契約内容に依存）。

5. 技術構成を「比喩」で理解する（詳細は不要な方向け）

いま表現しているもの実際のイメージ
Excel（マクロ付きブック） 画面・ボタン・表の置き場。現場が触る主たる場所。
マスタ用 Excel スキルや出勤など、計算の土台となるデータ。
裏側の計算プログラム 表の読み書きと配台ロジックを担当。Excel から自動で起動される。
出力フォルダ 計算結果のファイルやログが保存される場所。

「プログラムの名前」や「ライブラリ名」は経営判断には通常不要です。Windows PC と Excel が動く環境で、初回だけ 環境のセットアップ（社内ITまたは手順書に沿った作業）が必要、という理解で足ります。

6. メリットと、あらかじめ知っておきたいこと

メリット（期待しうる効果）

  • 配台表作成の 作業時間短縮、担当者による ブレの低減（ルールの再現性）。
  • 納期・優先度・スキル など複数条件を同時に考慮した案を素早く出せる。
  • 結果が Excel に残るため、説明・共有・印刷がしやすい。

留意点・リスク（経営・管理側の論点）

項目内容
クラウドAI依存 Gemini 利用には ネットワーク と 契約・キー管理 が必須。止まると設計どおりの機能が使えない。
コスト API の 従量課金・枠 は利用量次第。運用ルール（誰がどれだけ実行するか）の整理が望ましい。
情報の取り扱い 送信・保存されるデータの範囲を、個人情報・顧客情報・機密の観点で社内規程と照合すること。
Excel 前提 大規模な同時利用やモバイル中心運用には向かない構成。拡張時は Web 化など別アーキテクチャが検討対象になり得る。
環境の均一性 PC ごとにセットアップ差があると「動く／動かない」が分かれやすい。配布手順と IT 連携が重要。

7. 社内で決めておくとよいこと（チェックリスト）

  • Gemini API の契約主体・請求先・利用上限の監視方法
  • API キー の保管場所、アクセス権、漏えい時の対応
  • 計画データに含まれる情報の 機密度 と、クラウド送信の可否（必要に応じ 外部専門家 と確認）
  • 標準PC環境（Windows・Excel 版・ネットワーク）の定義と、テスト担当
  • 障害時の 代替運用（手作業配台に戻すまでの手順）

8.1 ① 本システムをベースに「一から作り直す」場合

現行のように Excel（マクロ）＋ローカル計算＋外部生成AI連携 が一体となった業務システムは、要件定義・設計・実装・結合試験・移行・教育まで含めると、外注では 複数人月〜十数人月以上 がかかる規模になり得ます。§8.0 の表がその金額イメージです。

区分一般的なイメージ（税抜・幅大）備考
初期開発（新規構築） 数百万円台後半〜数千万円規模 がよく議論される帯。大企業向けSI・高い可用性・監査対応・複数拠点展開まで含めると それ以上 になることもある。 既に 要件定義書・ソース・VBA バックアップ があると調査・仕様の言語化コストは下がりやすいが、「同じ見た目・同じ挙動で再実装」でも テストと移行 は必ず工数がかかる。Web化やDB正規化まで求めると別見積になる。
月次の保守契約 初期開発費をベースに、年 おおよそ 15〜25% を月割りした 定額保守 とする契約が、よくある話として挙げられる（業界慣行の一例）。実額は 月数万円〜月数十万円（小規模）、月数十万円〜数百万円（担当アサイン・応答時間などを契約に明記） まで幅がある。 「何時間まで無償／超過分は別料金」「本番障害の応答時間」を契約書で定義すると単価が変わる。

8.3 経営側で押さえると見積がぶれにくいポイント

  • スコープ：「Excel のまま」か「Web/API 化まで」か、AI は必須かオプションか。
  • 検収条件：誰が何をもって「完了」とするか（試験観点・本番データでの試行）。
  • 保守の範囲：バグのみか、OS アップデート対応 や Excel 新バージョン まで含むか。
  • セキュリティ：個人情報の取り扱い、クラウド送信の可否、委託先の作業環境（機密保持、接続方法など）。

9. まとめ

本システムは、工場の配台業務を Excel 中心の運用のまま支援し、Google Gemini による文章・ルールの解釈を組み合わせたものです。業務効率と再現性の向上が期待できる一方、クラウドAIのコスト・セキュリティ・可用性は経営・管理層での合意と運用設計が不可欠です。

本紙は意思決定・説明用の要約であり、契約・検収の唯一の根拠にはしません。§8.0 は 市場感の例示、§8.4 は ソース行数と仮定係数による試算 であり、いずれも見積・契約金額を保証するものではありません。正式な金額は委託先からの見積を正としてください。
