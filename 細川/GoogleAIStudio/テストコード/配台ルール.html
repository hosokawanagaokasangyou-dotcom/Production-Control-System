タスクの並び（割付の試行順・要約）

グローバル（generate_plan 内・2026-04）: キュー全行にシートの配台試行順番（正の整数）が揃っているときは、その昇順（同値は planning_sheet_row_seq）で並べ dispatch_trial_order に反映（EC 隣接繰り上げはスキップ）。欠損がある行が1行でもあるときは task_queue.sort（_generate_plan_task_queue_sort_key）: b_tier（0=§B-1、1=§B-2/§B-3 帯、2=その他）→ b2_queue_sub → 加工途中 → 計画基準納期 → need 列位置 → 依頼NOタイブレーク。続けて _reorder_task_queue_b2_ec_inspection_consecutive で EC→後続を隣接、_assign_sequential_dispatch_trial_order で 1..n。納期シフト再試行後も同じ _apply_dispatch_trial_order_for_generate_plan で確定。

各日内（既定）: STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1 では _trial_order_first_schedule_pass が候補を dispatch_trial_order 昇順にし、§B-2/§B-3 は 完全二相（フェーズ1: 熱融着検査・巻返し（後続）以外を連続ロール → フェーズ2: 後続行を配台）。後続候補は _roll_pipeline_inspection_assign_room（バッファ付きロール枠）で制御されます。通常の日次ループ完了後、B-2 検査／B-3 巻返しを _run_b2_inspection_rewind_pass で日付先頭から再走査し、timeline_events を人・設備のブロックテーブルとして再利用して日跨ぎ占有を保持したまま詰めます。再走査中は _roll_pipeline_inspection_assign_room と _roll_pipeline_b2_inspection_ec_completion_floor_dt。壁時計の設備占有は machine_avail_dt を機械名占有キーで管理し、同一物理機械上の試行順は _equipment_line_lower_dispatch_trial_still_pending。_day_schedule_task_sort_key は主ループのソートには使わない。継続チーム優先は同日内のみで、翌日は当日候補から再選定します。

各日内（フォールバック）: STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0 のとき _day_schedule_task_sort_key 順の 1 本ループ。dispatch_trial_order は複合キーの一部（最優先単独ではない）。

納期シフト再試行: 対象依頼の再配台前に _apply_dispatch_trial_order_for_generate_plan で試行順を再確定（シート優先条件または自動ソート＋隣接＋連番）。

結果_タスク一覧の行順は 配台試行順番（グローバルソート直後の 1..n）が第一キーです。

A. 全加工依頼に適用する追加ルール

加工内容列による工程順序（同一依頼NO）

  • 列: マクロブック 加工計画DATA および 配台計画_タスク入力 の 加工内容（段階1でコピーされます）。
  • 書式: カンマ , 区切り。各要素は 工程名に対応（例: EC,検査）。
  • 制約: 同一 依頼NO で、いずれの行も 配台不要でない とき、リストの 前の工程がすべて完了（残ロール0）するまで、後の工程には割り当てません。
  • 例外: 工程名が加工内容に含まれない行は、この順序制約の対象外です。配台不要行はキューに入らず、前工程としてもブロックしません。ただし加工内容にトークンがある依頼では、結果シートでタスクIDセルを赤背景・白文字にして不整合を示します（下記セクション）。
  • 照合: 工程名は NFKC・空白除去で加工内容の各トークンと突き合わせます。
  • 日内の動き: 同一日に、加工内容順（rank）・シート行順・§B-1 試行順・§B-2 パイプライン検査の前倒し・優先度・結果用ソートを組み合わせてタスクを複数周回し、各タスクあたり最大1チャンクずつ割り当てるため、前後工程を同日で交互に進められます。

同一依頼NOの直列（加工内容に rank が無い行）

  • 原則: 同一 依頼NO の 別工程（別行）は、設備が違っても 同時加工しない。配台計画シートを上から読んだ 行順（same_request_line_seq）で、上の行の残ロールがなくなるまで下の行に割り当てません。
  • §A-1 との関係: 両行に加工内容由来の rank があるときは rank だけで前後を決め、片方でも rank が無いときは 行順で直列化します。
  • §B-2 / §B-3: EC×後続（検査または巻返し）パイプラインはロール枠・設備占有など別フィルタで制御。同一依頼の前後関係は 配台ルール.md B-2.6・B-3・_trial_order_flow_eligible_tasks を参照。
  • 実装: planning_core/_core.py の build_task_queue_from_planning_df と _task_blocked_by_same_request_dependency。

B. 個別に指定する追加ルール

B-1 — 検査×熱融着機湖南で加工途中なら試行順優先

対象: 工程名 検査・機械名 熱融着機　湖南で 加工途中（in_progress）の行（§B-2 の後続行と同一条件）。

  • グローバル: b_tier = 0。対象行同士の順は §2.0（加工途中→計画基準納期→need 列→依頼NO）。試行番号はその後の付与。
  • 日内（STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0）: §2.0 と同趣旨の先頭キーのあと、dispatch_trial_order を §A-1 の rank より前に置く（_day_schedule_task_sort_key）。
  • §B 群内: B-1 を最優先。B-2 と B-3 は同一の b_tier = 1 帯。詳細は 配台ルール.md §B-1。

B-2 — EC → 検査（ロール単位パイプライン）

次の 工程名・機械名の組み合わせに限り、同一依頼NO内でロール単位の並行を認めます（コード内定数）。

役割工程名機械名
先行（EC）ECEC機　湖南
後続（検査）検査熱融着機　湖南
  • バッファ: EC 側で 2 ロール完了するまで、検査には 1 ロールも割り当てません（段取り時間の確保）。定数: ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2。
  • EC 行が同一依頼に無い検査: ロール枠を事実上かけず（ROLL_PIPELINE_INSP_UNCAPPED_ROOM）、通常どおり need・スキルで配台します。
  • パイプライン: バッファ後は、検査の累計完了ロール数が常に次の上限を超えないようにします。
    進行中: ec_done（EC 完了ロール数）・insp_done（検査完了）とすると、原則 insp_done ≤ max(0, ec_done - 2 + 1)。EC が 2 ロール終わると検査 1 ロール目が可能になり、その後は EC が 1 ロール進むごとに検査も最大 1 ロール進められます。
    EC 全ロール完了後: 同一依頼NOの EC パイプライン行がすべて残量ゼロのときは、上記だけでは検査が 1 本足りなくなるため、上限を ec_done まで引き上げ、検査も EC と同じ本数まで完了できます（_pipeline_ec_fully_done_for_tid / _roll_pipeline_inspection_assign_room）。
  • 当日候補への入り方: _trial_order_flow_eligible_tasks で §A 同一依頼ブロック・_roll_pipeline_inspection_assign_room（ロール枠）・（既定で）設備占有・試行順強制を順に判定。通過後にチーム割付。前進・再走査の両方で同ロール枠と時刻下限が効きます。詳細は 配台ルール.md B-2.6。
  • 担当者の排他（同一依頼NO）: 既定では EC 行と検査行で 履歴に出たメンバーは他側の候補から除外（_filter_capable_members_b2_disjoint_teams・NFKC 突合）。PLANNING_B2_EC_FOLLOWER_DISJOINT_TEAMS でオフにすると除外しません。配台トレース対象依頼NOでは、除外が発生した時点で ブロック判定: B-2担当者分離 ... を出力。さらに候補試行時に 候補固定: 担当OP指名=...・候補却下: ...（OP不在/実働不足など）も出力。
  • 変更方法: planning_core/_core.py の ROLL_PIPELINE_EC_PROCESS 等を編集してください。

B-3 — EC → 巻返し（ロール単位パイプライン・B-2 同趣旨）

同一依頼NOで B-2 と同じロールパイプライン挙動（完全二相・ロール枠・リワインド・担当者排他・設備占有）を適用します。B-2 の「検査」に相当する工程を 巻返し、機械名は先行 EC と同じ EC機　湖南 です。

役割工程名機械名
先行（EC）ECEC機　湖南
後続（巻返し）巻返しEC機　湖南
  • 実装: タスク辞書の roll_pipeline_rewind。定数 ROLL_PIPELINE_REWIND_PROCESS / ROLL_PIPELINE_REWIND_MACHINE。完了ロール集計は _pipeline_b2_follower_roll_done_units（検査＋巻返しを合算）。
  • 詳細: 配台ルール.md §B-3 および §B-2.2〜B-2.7（語の読み替え）。

納期超過検出と後ろ倒し再試行

検出（_collect_task_ids_missed_deadline_after_day）: ある日のシミュレーション終了時に、残量あり・due_basis_date あり・start_date_req ≤ 当日・当日 ≥ due_basis_date の依頼NOを集める（納期当日を含む）。

再試行: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = True のとき、上記依頼について 内部 due_basis_date のみ +1 日、当該依頼のタイムラインと割当を巻き戻し、全日程を最初から再走査。表示用 due_basis_date_result_sheet は元日付のまま。上限 STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（依頼NOごと）。各ラウンド前に勤怠拡張日はマスタへ戻す。

  • オフにする: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = False で 1 パスのみ。
  • 上限: STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（既定 5、依頼NOごと。全体の合計回数ではない）。
  • シフトしない: 依頼全体で due_basis_date を持つ行が無い場合は、後ろ倒し対象にしません。
  • 勤怠最終日: +1 日がマスタ最終計画日を超える行がある依頼は、そのラウンドではシフト対象外になり得ます。

結果_タスク一覧・タスクIDの赤表示（加工内容との整合）

段階2が出力する production_plan_multi_day_*.xlsx の 結果_タスク一覧 で、元データの 加工内容（カンマ区切り）と行の 工程名 が一致しない可能性がある行の タスクID セルだけを、赤塗り・白文字にします。

  • 加工内容の取り方: セクション A と同じく、同一依頼NOについて表の上の方で最初に現れた非空の加工内容をトークン化します。
  • ハイライトしない: その依頼についてトークンが1つも無い（加工内容が未記入のみ等）場合は、比較基準が無いため色を付けません。
  • ハイライトする: トークンがあるのに、工程名が（NFKC 等の正規化後）いずれのトークンとも一致しないとき。
  • 配台不可／配台残の行全体の薄い黄より後にタスクIDだけ上書きするため、該当セルは赤が優先されます。
  • 結果_設備毎の時間割へのリンク: タスクIDセルに内部ハイパーリンクを付け、時間割で当該タスクが最初に現れるセルへジャンプ（未割当のみで時間割に無い行はリンクなし）。_apply_result_task_id_hyperlinks_to_equipment_schedule。
  • 履歴列（追加人数・色）: req_num を超えた人数のブロックのみ薄黄（FFEB9C）。メインは TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS ＋追加枠あり ＋採用人数>req_num。余力追記は追記後の人数が req_num を超えるときのみ（定員までの追記のみでは塗らない）。履歴の担当表記は追記後に同期。_apply_result_task_history_need_surplus_highlight。
  • 履歴テキスト: 組合せ表#nはプリセット行採用時に付くほか、探索のみで ID が欠けていても採用メンバー集合がいずれかのプリセットと一致すれば _lookup_combo_sheet_row_id_for_preset_team で補完（履歴1〜n 共通）。一致プリセットが無ければ付かない。プリセットは成立分をすべて候補に載せ、探索とまとめて最良を選ぶ。担当はメイン確定分のみ（余力追記は余力追記[…]）。追加[…]はメインで人数超過。_format_result_task_history_cell／_history_team_text_main_assignment_only。
  • 列設定_結果_タスク一覧（結果ブック）: 段階2は A:B を毎回再生成するが、既定で生成後に TASK_INPUT_WORKBOOK 側の同名シートから xlwings で図形（ボタン等）を複製し、位置・サイズを入力側と同じに戻す（_stage2_try_copy_column_config_shapes_from_input）。無効化は STAGE2_COPY_COLUMN_CONFIG_SHAPES_FROM_INPUT=0。

実装: planning_core/_core.py の process_content_mismatch、_process_name_matches_kakou_content_tokens、_apply_result_task_task_id_content_mismatch_highlight、_apply_result_task_id_hyperlinks_to_equipment_schedule、_apply_result_task_history_need_surplus_highlight、_format_result_task_history_cell、_lookup_combo_sheet_row_id_for_preset_team、_stage2_try_copy_column_config_shapes_from_input。

列 配完_回答指定16時まで: 配完_加工終了が、回答納期の日付＋既定 16:00（PLAN_DUE_DAY_COMPLETION_TIME）以前かを判定。回答納期が空なら 指定納期＋16:00 で判定。値は はい／いいえ／未割当／納期なし（回答・指定とも日付なし）。_result_task_plan_end_within_answer_or_spec_16_label。いいえのセルは赤塗り・白文字・太字（_apply_result_task_plan_end_answer_spec_16_no_highlight）。マクロブックへの xlsx 取り込み直後は VBA 結果_タスク一覧_配完回答指定16時_いいえを強調で同趣旨を再適用。

配台トレース（設定シート A3 のみ）

指定 依頼NO について [配台トレース task=…] 詳細を log/execution_log.txt に出します。

  • generate_plan 冒頭（skills 読込より前）に読み、必ず INFO ログ。メンバー0で即終了したときも設定シートの読み取り結果がログに残ります。
