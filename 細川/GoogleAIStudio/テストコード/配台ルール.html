配台ルール

全加工依頼に適用するルールと、個別指定ルールを分けて記載しています。詳細は 配台ルール.md、実装の正は python/planning_core.py です。2026-04 反映: 計画基準納期は「グローバル AI task_planning_basis_dates →（特別指定_備考は納期系キーワードがある行のみ）備考 AI の目標日／開始日 → 指定納期_上書き → 原反投入日 → 回答納期 → 指定納期」。開始日は max(基準日, 原反投入日)＋上書き。原反当日は 13:00 以降開始。日内は既定で STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1（配台試行順優先マルチパス）。STAGE2_SERIAL_DISPATCH_BY_TASK_ID=1 で依頼NO直列。§B-2 は PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE（既定オン・変数名は歴史的）で設備占有。優先度同数重複は段階2中止。納期リトライは _collect_task_ids_missed_deadline_after_day 条件（本文 2.3）。need 配台時追加人数は既定でメインは基本人数のみ→全シミュレーション後に未割当×スキルでサブ追記（下記）。配台ロジックを変えたら 配台ルール.md / 本 HTML を必ず更新（開発ルール）。業務案の順次1件・16:00締め込みは次段。

改定（2026-03）要約

  • グローバル AI: JSON に task_planning_basis_dates（依頼NO→日付）を追加。特別指定備考 AI より先に計画基準納期へ反映。
  • 優先度重複: 配台対象行で同じ優先度整数が2行以上→段階2中止、終了コード3、メッセージファイル経由で Excel に表示（999は除外）。
  • 同日同機械: 計画基準納期と機械名が同じ帯では、依頼NOのタイブレーク（Y4-1-1 は先頭-後の数値）。
  • 日内配台（実装）: STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST 既定オンで配台試行順の昇順にマルチパス。同一設備列は _equipment_line_lower_dispatch_trial_still_pending で試行順を強制。
  • 依頼NO直列: STAGE2_SERIAL_DISPATCH_BY_TASK_ID=1 のときのみ、残のある最上位依頼に限定。
  • 未実装（次段）: 業務案どおり「近い納期から1件ずつ」型メインループ、計画基準納日16:00での割付強制（定数 PLAN_DUE_DAY_COMPLETION_TIME の組み込み）。

タスクの並び（割付の試行順・要約）

グローバル（generate_plan 内の task_queue.sort・_generate_plan_task_queue_sort_key）: 納期ソース順 → 完了日上書き行 → b_tier（0=§B-1 検査×熱融着×加工途中、1=§B-2 帯＝roll_pipeline_ec または未着手検査、2=その他）→ b2_queue_sub（§B-2 帯内のみ EC が検査より先）→ §B-1 試行順タイブレーク（planning_sheet_row_seq）→ §B 対象外の加工途中 → 配台計画の優先度 → 納期緊急度 → 納期基準日 → 機械名 → 依頼NOタイブレーク → 開始日 → 依頼NO → 必要OP人数（多いほど先）。§B は §A より先。§B 群内は B-1 ＞ B-2 帯。ソート直後に 配台試行順番 1..n（シート物理行順ではない）。

各日内（既定）: STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1 では _trial_order_first_schedule_pass が _trial_order_flow_eligible_tasks で絞った候補を dispatch_trial_order 昇順に並べ、1 パスでは先頭から見るが、1 ロールでも進んだタスクは可能な限り連続ロール割付し、進展があればそのパスは終了（同一パス内では試行順の後続には回らない）。外側マルチパスで再び先頭から。同一設備列では若い試行順がまだ詰めうる間、後続を当該設備では試さない（_equipment_line_lower_dispatch_trial_still_pending）。_day_schedule_task_sort_key はこの経路の主ループのソートには使わない。

各日内（フォールバック）: STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0 のとき _day_schedule_task_sort_key 順の 1 本ループ。dispatch_trial_order は複合キーの一部（最優先単独ではない）。

納期シフト再試行: ラウンドごとに task_queue は再ソートされるが、dispatch_trial_order は付け直さない（初回 1..n のまま）。

結果_タスク一覧の行順は 配台試行順番（グローバルソート直後の 1..n）が第一キーです。

日内割付・主な環境変数

  • STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST … 既定 1。配台試行順優先の _trial_order_first_schedule_pass。0 で従来の日内ソートキーループ。
  • STAGE2_SERIAL_DISPATCH_BY_TASK_ID … 既定相当 0。1 で依頼NO直列（アクティブ依頼のみ当日候補）。
  • PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE … 既定オン。§B-2 検査が占有する設備列では他タスクを試さない（変数名は歴史的）。
  • TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS … 未設定＝メインは基本人数のみ、全シミュレーション後に余力サブ追記。1＝従来どおりメインで req+追加上限 まで探索。
  • TEAM_ASSIGN_IGNORE_NEED_SURPLUS_ROW … 1 で need 追加人数行を全面無視（後追いもなし）。
  • TEAM_ASSIGN_*（人数優先・スラック）… 配台ルール.md セクション5。

need 追加人数（余力・メイン後追記）

master.xlsm の need で、基本必要人数の直下（目安 Excel 5 行目付近）の「配台時追加人数／余力時追加人数」列は、工程×機械ごとに req を満たしたあと追加で入れられる人数の上限です。

  • 既定: メインの割付シミュレーションではこの上限を人数探索に使わず req_num ちょうどで組む。シミュレーション完了後、各タイムラインブロックで枠が空き、かつその時間帯に他ブロックと重ならない・スキル OP/AS 適合・eligible の人を サブ（sub）に追記（append_surplus_staff_after_main_dispatch）。
  • 時刻は変えない（表示・同席記録向け。メインで決めた end_dt のまま）。
  • 詳細・表は 配台ルール.md セクション 3.5。

A. 全加工依頼に適用する追加ルール

加工内容列による工程順序（同一依頼NO）

  • 列: マクロブック 加工計画DATA および 配台計画_タスク入力 の 加工内容（段階1でコピーされます）。
  • 書式: カンマ , 区切り。各要素は 工程名に対応（例: EC,検査）。
  • 制約: 同一 依頼NO で、いずれの行も 配台不要でない とき、リストの 前の工程がすべて完了（残ロール0）するまで、後の工程には割り当てません。
  • 例外: 工程名が加工内容に含まれない行は、この順序制約の対象外です。配台不要行はキューに入らず、前工程としてもブロックしません。ただし加工内容にトークンがある依頼では、結果シートでタスクIDセルを赤背景・白文字にして不整合を示します（下記セクション）。
  • 照合: 工程名は NFKC・空白除去で加工内容の各トークンと突き合わせます。
  • 日内の動き: 同一日に、加工内容順（rank）・シート行順・§B-1 試行順・§B-2 パイプライン検査の前倒し・優先度・結果用ソートを組み合わせてタスクを複数周回し、各タスクあたり最大1チャンクずつ割り当てるため、前後工程を同日で交互に進められます。

同一依頼NOの直列（加工内容に rank が無い行）

  • 原則: 同一 依頼NO の 別工程（別行）は、設備が違っても 同時加工しない。配台計画シートを上から読んだ 行順（same_request_line_seq）で、上の行の残ロールがなくなるまで下の行に割り当てません。
  • §A-1 との関係: 両行に加工内容由来の rank があるときは rank だけで前後を決め、片方でも rank が無いときは 行順で直列化します。
  • §B-2: EC×検査パイプラインはロール枠・設備占有など別フィルタで制御。同一依頼の前後関係は 配台ルール.md B-2.6・_trial_order_flow_eligible_tasks を参照。
  • 実装: planning_core.py の build_task_queue_from_planning_df と _task_blocked_by_same_request_dependency。

B. 個別に指定する追加ルール

B-1 — 検査×熱融着機湖南で加工途中なら試行順優先

対象: 工程名 検査・機械名 熱融着機　湖南で 加工途中（in_progress）の行（§B-2 の後続行と同一条件）。

  • グローバル: b_tier = 0。加工途中同士は planning_sheet_row_seq でタイブレーク（試行番号付与前の「シート順の土台」）。
  • 日内: 同じ納期・機械・タイブレークの帯で、dispatch_trial_order を §A-1 の rank より前に置く（_day_schedule_task_sort_key）。
  • §B 群内: B-1 ＞ B-2。詳細は 配台ルール.md §B-1。

B-2 — EC → 検査（ロール単位パイプライン）

次の 工程名・機械名の組み合わせに限り、同一依頼NO内でロール単位の並行を認めます（コード内定数）。

役割工程名機械名
先行（EC）ECEC機　湖南
後続（検査）検査熱融着機　湖南
  • バッファ: EC 側で 2 ロール完了するまで、検査には 1 ロールも割り当てません（段取り時間の確保）。定数: ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2。
  • EC 行が同一依頼に無い検査: ロール枠を事実上かけず（ROLL_PIPELINE_INSP_UNCAPPED_ROOM）、通常どおり need・スキルで配台します。
  • パイプライン: バッファ後は、検査の累計完了ロール数が常に次の上限を超えないようにします。
    進行中: ec_done（EC 完了ロール数）・insp_done（検査完了）とすると、原則 insp_done ≤ max(0, ec_done - 2 + 1)。EC が 2 ロール終わると検査 1 ロール目が可能になり、その後は EC が 1 ロール進むごとに検査も最大 1 ロール進められます。
    EC 全ロール完了後: 同一依頼NOの EC パイプライン行がすべて残量ゼロのときは、上記だけでは検査が 1 本足りなくなるため、上限を ec_done まで引き上げ、検査も EC と同じ本数まで完了できます（_pipeline_ec_fully_done_for_tid / _roll_pipeline_inspection_assign_room）。
  • 当日候補への入り方: _trial_order_flow_eligible_tasks で §A 同一依頼ブロック・_roll_pipeline_inspection_blocked_until_ec_done・ロール枠・（既定で）設備占有・試行順強制を順に判定。通過後にチーム割付・ロール枠が効く。詳細は 配台ルール.md B-2.6。
  • 変更方法: planning_core.py の ROLL_PIPELINE_EC_PROCESS 等を編集してください。

納期超過検出と後ろ倒し再試行

検出（_collect_task_ids_missed_deadline_after_day）: ある日のシミュレーション終了時に、残量あり・due_basis_date あり・start_date_req ≤ 当日・当日 ≥ due_basis_date の依頼NOを集める（納期当日を含む）。

再試行: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = True のとき、上記依頼について 内部 due_basis_date のみ +1 日、当該依頼のタイムラインと割当を巻き戻し、全日程を最初から再走査。表示用 due_basis_date_result_sheet は元日付のまま。上限 STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（依頼NOごと）。各ラウンド前に勤怠拡張日はマスタへ戻す。

  • オフにする: STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING = False で 1 パスのみ。
  • 上限: STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS（既定 40、依頼NOごと。全体の合計回数ではない）。
  • シフトしない: 依頼全体で due_basis_date を持つ行が無い場合は、後ろ倒し対象にしません。
  • 勤怠最終日: +1 日がマスタ最終計画日を超える行がある依頼は、そのラウンドではシフト対象外になり得ます。

結果_タスク一覧・タスクIDの赤表示（加工内容との整合）

段階2が出力する production_plan_multi_day_*.xlsx の 結果_タスク一覧 で、元データの 加工内容（カンマ区切り）と行の 工程名 が一致しない可能性がある行の タスクID セルだけを、赤塗り・白文字にします。

  • 加工内容の取り方: セクション A と同じく、同一依頼NOについて表の上の方で最初に現れた非空の加工内容をトークン化します。
  • ハイライトしない: その依頼についてトークンが1つも無い（加工内容が未記入のみ等）場合は、比較基準が無いため色を付けません。
  • ハイライトする: トークンがあるのに、工程名が（NFKC 等の正規化後）いずれのトークンとも一致しないとき。
  • 配台不可／配台残の行全体の薄い黄より後にタスクIDだけ上書きするため、該当セルは赤が優先されます。

実装: planning_core.py の process_content_mismatch、_process_name_matches_kakou_content_tokens、_apply_result_task_task_id_content_mismatch_highlight。

列 配完_回答指定16時まで: 配完_加工終了が、回答納期の日付＋既定 16:00（PLAN_DUE_DAY_COMPLETION_TIME）以前かを判定。回答納期が空なら 指定納期＋16:00 で判定。値は はい／いいえ／未割当／納期なし（回答・指定とも日付なし）。_result_task_plan_end_within_answer_or_spec_16_label。

配台デバッグ（ロールトレース JSONL）

割付で 1 ロール確定ごとに、依頼・設備・時刻などを JSONL（1 行 1 JSON）へ追記します。

  • DISPATCH_ROLL_TRACE_JSONL … 出力パス。相対はマクロブック（TASK_INPUT_WORKBOOK）と同じフォルダ基準。未設定または空ではファイルを作りません（xlwings_console_runner の段階2デバッグ入口は既定で log/dispatch_roll_trace.jsonl を設定）。
  • DISPATCH_DEBUG_STOP_AFTER_ROLLS … 正の整数で、指定ロール数でシミュレーション打ち切り。
  • 初期化: generate_plan の先頭でファイルを空にし、早期終了時もトレース有効なら 空の JSONL が残ります（以前は日次ループ直前のみ初期化）。
  • 追記: ループ内の _dispatch_roll_trace_after_roll。ロールが無ければ 0 行のまま。

詳細は 配台ルール.md セクション 10。

配台トレース（設定シート A3・環境変数）

指定 依頼NO について [配台トレース task=…] 詳細を log/execution_log.txt に出します。

  • TRACE_SCHEDULE_TASK_ID … カンマ区切り（例 W4-1,Y3-26）。
  • マクロブック 「設定」シート A 列 3 行目以降に依頼NOを縦に（_read_trace_schedule_task_ids_from_config_sheet）。
  • 和集合が有効集合。generate_plan 冒頭（JSONL 初期化の直後・skills 読込より前）に読み、必ず INFO ログ。メンバー0で即終了したときも設定シートの読み取り結果がログに残ります。
  • [配台トレース task=…] 行は log/execution_log.txt に加え、log/dispatch_trace_<依頼NO>.txt（名はサニタイズ）へ同一文面を追記。段階2開始時に対象ごとにファイルを上書き初期化。
  • シート 配台_配台不要工程（openpyxl 非互換マーカー）があるブックは「設定」を openpyxl で読めず A3 以降は無視（環境変数のみ）。

詳細は 配台ルール.md セクション 11。

デバッグ限定配台（設定シート B3・環境変数）

指定した 依頼NO に該当する 配台計画_タスク入力 の行だけで段階2を試す（検証用）。

  • 優先: マクロブック 「設定」シート B 列 3 行目以降（空スキップ・連続30空で打ち切り・最大500行・カンマ区切り可）。
  • B 列が空のとき: 環境変数 DISPATCH_DEBUG_ONLY_TASK_IDS（カンマ区切り）。
  • generate_plan 冒頭で集合を確定し、タスク読込直後にフィルタ。0 行なら中断。有効時は WARNING で対象一覧をログへ。
  • VBA: 段階2_コア実行 と xlwings の配台デバッグ入口で、B 列に値があるとき MsgBox 通知のあと Save してから Python 実行（ディスク読み）。
  • openpyxl 非互換ブックでは B 列は読めず環境変数のみ（セクション11と同様）。
  • A 列はトレース用、B 列は対象絞り込み用（同時指定可だが集合は一致しないことがある）。

詳細は 配台ルール.md セクション 12。
