# 配台ルール（現行仕様）

実装の正は `python/planning_core.py` の `generate_plan`・`build_task_queue_from_planning_df`・日次割付ループおよび周辺関数です。本書はその挙動を業務向けに要約したものです。

**開発運用**: 配台ロジックをソースで変更したときは、**本書（`配台ルール.md`）と `配台ルール.html` を必ず同内容に更新する**（Cursor ルール `.cursor/rules/dispatch-docs-sync.mdc`）。

## 改定（2026-03）計画基準納期・配台まわり

### 1. 計画基準納期（`due_basis_date`）の決定順

実装は `build_task_queue_from_planning_df` 内の分岐です（上から最初に成立したもの）。

| 順位 | 内容 |
|------|------|
| ① | メインシート **グローバルコメント** を Gemini で解釈した JSON の **`task_planning_basis_dates`**（依頼NO → YYYY-MM-DD）。該当キーがあればそれが計画基準納期。 |
| ② | **特別指定_備考** を AI 解析した **完了・出荷目標日**（`_ai_planning_target_due_date`）。**備考セルに納期・期限・最優先等のキーワードがある行だけ**がこの経路を使う（`_special_remark_implies_due_dispatch_priority`）。 |
| ③ | 同上条件を満たす行の AI **`start_date`**（目標日が無い場合の締め用）。 |
| ④ | **指定納期_上書き**（空白は無視）。 |
| ⑤ | **原反投入日**（列に日付がある行は、計画基準納期＝原反投入日）。 |
| ⑥ | **回答納期**。 |
| ⑦ | **指定納期**。 |

**優先度列の重複チェック**: 配台対象行のうち、シートの **優先度** が **同一整数で2行以上**あると段階2を中止します（**999** は未入力扱いで重複検査から除外）。`PlanningValidationError` → `log/stage2_blocking_message.txt` に1行で理由を残し、終了コード **3**。VBA は `m_lastStage2ErrMsg` に取り込み **計画生成** で MsgBox し、**結果ブックの取り込みはスキップ**します。

### 2. 順次配台・日内フロー（業務案と実装の対応）

- **業務案（未実装）**: 「納期が近い1件ずつだけ配台し、超過時に都度計画基準+1」型のメインループは、現行の **全日程×日次マルチパス**構造とは別のため **未実装**です。
- **実装されている日内配台**: 環境変数 **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST`**（既定 `1`）で、**`_trial_order_first_schedule_pass`** が有効なとき、当日候補を **配台試行順（`dispatch_trial_order`）の昇順**で回し、**試行順が先でもその日入らなければ同じパス内で次の試行順へ進む**（1タスクあたり複数ロール詰め可能）。`0` にすると従来の **`_day_schedule_task_sort_key` ソート順**の一日内ループにフォールバックします。
- **依頼NO直列**: 環境変数 **`STAGE2_SERIAL_DISPATCH_BY_TASK_ID=1`** のときのみ、**残がある最も早い依頼NO**の行だけが当日候補になり、該当依頼が詰まると他依頼は進みません（既定は無効）。
- **納期超過後の全体リトライ**はセクション **2.3**（`_collect_task_ids_missed_deadline_after_day` トリガ）のとおりです。

### 3. 計画基準納期当日の完了時刻 16:00

定数 **`PLAN_DUE_DAY_COMPLETION_TIME`（16:00）** を `planning_core.py` に定義済みです。**割付終了時刻への組み込みは今後の拡張**です（現状はシフト終了・既定 17:00 枠など従来ロジック）。

### 4. 同一計画基準納期・同一機械名の試行順

**日内ソート**および **グローバルキュー**に、**計画基準納期 → 機械名（正規化）→ 依頼NOタイブレーク**を挿入済みです。

- 通常の依頼NO（例 `Y3-24`）: **最後の `-` 以降が数値**ならその整数が小さいほど先。
- **ハイフンが2つ以上**（例 `Y4-1-1`）: **最初の `-` の直後の区切りの数値**を比較（例では **1**）。

### 5. 同一依頼NO・異なる工程×機械の順序

**加工内容**をカンマで分割した工程順と **工程名** の照合は従来どおり **§A-1**（`process_sequence_rank`・`_task_blocked_by_same_request_dependency`）です。

## 適用の優先順位

矛盾があるときは次のとおりです（上ほど強い）。

**ルール群の階層（明示）**

① **§B（個別ルール群）**は **§A（全依頼ルール群）より優先**します。§B と §A が食い違うときは、該当する依頼・工程・機械の組み合わせでは **§B を採用**します。  
② **§B 群内**の適用の強さは **`B-1` ＞ `B-2` ＞ …** とします（番号が小さいほど先に効く。実装ではグローバル `task_queue.sort` のキーで **B-1（`roll_pipeline_inspection` かつ `in_progress`＋試行順タイブレーク）→ B-2（同じくパイプライン検査だが未着手）** → その他、の順です）。

**本文セクションとの対応**

1. **§B 個別に指定する追加ルール**（例: B-1 検査×熱融着機湖南の加工途中と試行順、B-2 EC→検査ロールパイプライン）
2. **§A 全加工依頼に適用する追加ルール**（例: A-1 加工内容の工程順）
3. **本文の番号付き一般ルール**（「1. 配台の対象外」以降）

**§B-1** は **検査×熱融着機湖南かつ加工途中**の行に対し、**配台試行順**（グローバルでは `planning_sheet_row_seq`、日内では `dispatch_trial_order`）を **§A-1 の rank より先に**効かせます（下記 §B-1・§2.0）。**§B-2** は **ロールパイプライン定数・ロール枠・（既定の）設備占有・キュー／日内ソートでの検査前倒し** 等で制御します。**§A-1 は B-2 検査行にも適用されうる**ため、実際に検査が候補に残るかは **§B-2.6** の **`_trial_order_flow_eligible_tasks` チェーン**（§A ブロック → EC 完走待ち → 枠・占有・試行順）で決まります。**§B 対象外**は **§A-1 / §A-2** を従来どおり適用し、**§B-2** 対象の未着手検査はグローバルキューで **`b_tier = 1`** により §A より先の帯に並びます。

## A. 全加工依頼に適用する追加ルール

### A-1 「加工内容」列に基づく工程順序（同一依頼NO）

**データ**: マクロブック **「加工計画DATA」** および段階1経由の **「配台計画_タスク入力」** に列 **`加工内容`** がある場合に有効です（段階1で `SOURCE_BASE_COLUMNS` に含まれるため、抽出データに引き継がれます）。

**ルール**:

- 同一 **依頼NO** で複数行（複数工程）があり、いずれも **配台不要ではない** とき、**加工内容** の文字列を **カンマ `,` 区切り**で分割した順序が **工程の正しい順序**です（先頭が最初の工程）。
- 例: `EC,検査` → **先行工程に残ロールがある間、後工程には `_task_blocked_by_same_request_dependency` により割付を試みません**（§A-1）。  
  **§B-2**（`EC機　湖南`→`熱融着機　湖南`）の検査行は、**`_trial_order_flow_eligible_tasks`** 内の **`_roll_pipeline_inspection_blocked_until_ec_done`**（EC 残がある間は候補外）· **ロール枠** · **設備占有** などで制御され、§A より上位のフィルタチェーンは **B-2.6** 参照（§B が本文一般ルールより優先）。
- 各行の **工程名** が分割後のどの要素と一致するかで順序番号を決めます（NFKC・空白除去で照合）。**加工内容に現れない工程名**の行には、この順序制約はかかりません（ただし **結果シートでは不整合としてタスクIDを赤表示**します。下記「9.」）。
- **配台不要**の行はキューに入らないため、ブロッカーにもなりません。

**実装上の補足**: 既定の日内フロー（**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`**）では **配台試行順**を軸にマルチパスしますが、**「全タスクを 1 ロールずつ公平に回す」方式ではありません**（§2.0.2）。旧フロー（`0`）では **§2.0.1** の複合キー順でループします。

### A-2 同一依頼NOの直列化（rank が無い行・加工内容に載らない工程）

**目的**: 同一 **依頼NO** で **異なる工程（別行）** は、**別設備でも同時に加工しない**（現場の流れを1本化）。

**データ**: 段階1→配台計画シートを **上から**読み込んだときの **同一依頼NO内の行番号**（`same_request_line_seq`）。

**ルール**:

- **§A-1** で両行に `process_sequence_rank` が付くときは **rank のみ**で前後を判定（本節は追加判定しない）。
- **どちらか一方でも rank が無い**ときは、**シート上で上にある行**（`same_request_line_seq` が小さい方）の **残ロールがゼロになるまで**、下の行には割付を試みません。
- **§B-2**（上記 EC×検査）は **`_task_blocked_by_same_request_dependency` を特別に抜ける実装ではなく**、**B-2.6** のフィルタ（EC 完走待ち・ロール枠等）で並行度が決まります。

**実装**: `planning_core.py` の `build_task_queue_from_planning_df`（`same_request_line_seq` 付与）と `_task_blocked_by_same_request_dependency`。

## B. 個別に指定する追加ルール

**§B は「配台ルール（現行仕様）」本文の一般ルール（§1 以降）より優先**します。§A と食い違う場合も、**§B に該当する依頼・工程の組み合わせでは §B を採用**します（上記「適用の優先順位」参照）。

**ブロック構成**: 今後 **`B-1`, `B-2`, …** とルールを追加する想定です。各 **`B-n` を独立したブロック**（対象・ルール・§A との関係・定数）として記述します。

---

### B-1 — 検査（熱融着機　湖南）かつ加工途中のとき、配台試行順を優先

**対象**: **§B-2.1 の後続行と同一**（工程名 **`検査`**、機械名 **`熱融着機　湖南`**）かつ **`in_progress`（加工途中・実績あり）** が真のタスク。

**ルール**

- **グローバル** `task_queue.sort`（`_generate_plan_task_queue_sort_key`）: **`b_tier = 0`** として §B 群内で最優先。対象行同士のタイブレークでは **配台計画シートの読み込み順**（`planning_sheet_row_seq`、上にある行ほど小さい）を先に効かせます。**`dispatch_trial_order` はこのソートの直後に 1..n で付与**されるため、初回グローバルソートではシート順が「試行順の土台」になります。
- **日内** `_day_schedule_task_sort_key`: 同一 **計画基準納期・機械名・依頼NOタイブレーク** の帯で、対象行は **`dispatch_trial_order`（昇順）を `process_sequence_rank`（§A-1）より前**に置きます。**§B-2 帯の並び（`b2_roll_pipeline_stage`）より前**のキー列に置くため、**加工途中の検査（§B-1）**では試行順が **EC 先行の §B-2 段**よりも先に効きます（§2.0.1）。
- **§B 群内の順位**: **B-1**（本節）を **B-2**（ロールパイプライン）より **常に先**にソートキーへ反映しています。

**実装**: `planning_core.py` の `_generate_plan_task_queue_sort_key`、`_day_schedule_task_sort_key`。

---

### B-2 — EC（EC機　湖南）→ 検査（熱融着機　湖南）ロールパイプライン

**ロジック（方針）**

1. **後続の検査**は他のどのタスクよりも優先とするため、**先行の EC の配台の直後**に配台を行う（日内ループの試行順・キュー順で EC に続けて検査を試し、**ロール単位の並行枠・段取りバッファ**は B-2.2 の式に従う）。
2. **§B 群内**では **§B-1**（加工途中の熱融着検査と試行順）に次いで、**EC 行と未着手の検査行**を **§A より先の同一帯**（`b_tier = 1`）にまとめ、**同一帯内では EC を検査より先**（`b2_queue_sub` / 日内は `b2_roll_pipeline_stage`）に並べます。ロール枠・バッファは B-2.2 の式に従います。

#### B-2.1 対象（コード定数・同一依頼NOの行同士のみ）

| 段階 | 工程名 | 機械名 |
|------|--------|--------|
| 先行 | `EC` | `EC機　湖南` |
| 後続 | `検査` | `熱融着機　湖南` |

#### B-2.2 ルール（ロールパイプライン・並行）

- **ロール単位**でパイプラインします。EC 側の **完了ロール数**を `ec_done`、検査側を `insp_done` とします（計画上の残量から算出）。
- **段取りバッファ**: **EC が 2 ロール完了するまで、検査には 1 ロールも割り当てません**（`ROLL_PIPELINE_INITIAL_BUFFER_ROLLS = 2`）。
- **同一依頼のキューに EC 行が無い**検査行は、ロール枠を事実上かけない（`ROLL_PIPELINE_INSP_UNCAPPED_ROOM`）。`need`・スキルに従い通常配台します（EC 不在で永久スキップしない）。
- バッファ後は **検査済** `insp_done` が次の上限を超えないようにします。  
  **上限（パイプライン進行中）**: `max(0, ec_done - 2 + 1)`（`ec_done < 2` の間は実質 0）。  
  **EC 全ロール完了後**: 同一依頼NOの EC パイプライン行が **すべて残量ゼロ**のときは、上記式だけでは検査の許容本数が `ec_done` より 1 本少なくなり最終ロールが詰まるため、上限を **`ec_done` まで引き上げ**、検査も EC と **同じ本数まで**完了できるようにします（`_pipeline_ec_fully_done_for_tid` / `_roll_pipeline_inspection_assign_room`）。  
  解釈: EC が 2 ロール終わった時点で検査 1 ロール目が許可され、その後は **EC で 1 ロール進むごとに、検査でも最大 1 ロール**（並行のずらし積み）。最後の本数は EC 完走後に検査が取り残されないよう補正されます。
- **検査の優先・並行試行**  
  - **グローバル** `task_queue.sort`: **§B-1**（加工途中）の次に **`b_tier = 1`**（本節の検査行・未着手）が来ます。  
  - **同一日内** `_day_schedule_task_sort_key`: **§B-1** 対象は **`dispatch_trial_order` を rank より前**へ。同じ `加工内容` 由来の **rank 内**では **§B-2** の検査を先に試すほか、**パイプラインに割当可能枠がある**（`_roll_pipeline_inspection_assign_room` > 0）ときは **工程順 rank を 1 段繰り上げ**、**他依頼の先行工程（rank 相当）と同じ波**で割付を試みます（EC に完了ロールがあるのに他依頼ばかり先になるのを防ぐ）。

#### B-2.3 §A-1・§A-2 との関係

- **`_task_blocked_by_same_request_dependency`** は B-2 ペアだけを自動免除する分岐は持ちません。**同一依頼で加工内容上 EC が検査より先なら**、先行 EC に残がある間は §A-1 により検査はスキップされ得ます。
- **`_roll_pipeline_inspection_blocked_until_ec_done`** は「同一依頼に EC パイプライン行があり、いずれかに **残ロールがある**間は B-2 検査を候補から外す」動作で、**EC 完走後**に初めて検査が試行されます（実装と B-2.2 のロール並行イメージは読み替えが必要な場合があります。詳細はコード参照）。
- **同一依頼内の他の前工程**（B-2 以外）に残がある場合も §A-1 が検査をブロックし得ます。**他依頼NO**には B-2 定数は適用されません。

#### B-2.4 同一設備列での検査占有（環境変数）

- 既定オン: **`PLANNING_B1_INSPECTION_EXCLUSIVE_MACHINE`**（`1` 相当。`0` / `false` / `off` 等で無効。名称は歴史的なもの）。**§B-2 検査**が**既に割付開始**し残ロールがある行がある設備列（`equipment_line_key`）では、**他タスクを試行しない**（`_exclusive_b1_inspection_holder_for_machine`）。オフにすると時間割上は依頼が混在し得るが、長期占有ブロックは避けられる場合があります。

#### B-2.5 定数の変更

- `planning_core.py` 内の `ROLL_PIPELINE_*` および `ROLL_PIPELINE_INITIAL_BUFFER_ROLLS` を編集してください。

#### B-2.6 実装上の候補フィルタ（`_trial_order_flow_eligible_tasks`）

当日候補に入る前に、同一依頼の **`_task_blocked_by_same_request_dependency`**（§A）· **`_roll_pipeline_inspection_blocked_until_ec_done`**（同一依頼の EC パイプライン行に残がある間は B-2 検査を外す）· **ロール枠ゼロ**· **上記 B-2.4 の設備占有**· **同一設備の試行順強制**の順で判定されます。ドキュメント上の「EC 直後に検査を試す」表現は、**これらのフィルタを通過した後**の日内ソート・1ロール割付の話として解釈してください。

## 1. 配台の対象外（キューに載せない）

- **完了扱い**の行（完了キーワード等で除外）
- 配台計画シートの **「配台不要」** がオン
- **同一依頼NO × 同一機械名**で複数行あるグループのうち、工程名が **「分割」** の行は、セルが空なら自動で配台不要
- マクロブック **「設定_配台不要工程」**  
  - C列＝常に配台不要の工程×機械  
  - または E列の条件（AI が JSON 化したロジック）が真のとき、その組み合わせの行は配台不要

## 2. タスクの並び順（割付の試行順）

`generate_plan` 内で構築した `task_queue` は、**日付ループに入る前**に `task_queue.sort` で並べ替えます（先に並んだタスクほど、日をまたいだときも相対的に先に扱われます）。その直後に **`dispatch_trial_order` = 1..n** を付与します（**配台計画シートの物理行順そのものではなく**、このソート後の並びが試行番号の基準です）。

**各日の割付順**は環境変数 **`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST`** で分岐します。

- **`=1`（既定）**: 当日の割付本体は **`_trial_order_first_schedule_pass`**（§2.0.2）。`_day_schedule_task_sort_key` はこの経路では**割付ループの主ソートには使いません**。
- **`=0`（旧フロー）**: 当日は **`_day_schedule_task_sort_key`** でソートした順に 1 本のループで試行します（試行順はキーの一部）。

### 2.0 グローバルキュー（`task_queue.sort` のタプル順）

実装は `planning_core.generate_plan` 内の `task_queue.sort`（関数 **`_generate_plan_task_queue_sort_key`**）と一致させています（納期シフト再試行の各ラウンドでも同じキーで再ソート）。

1. **`due_source_rank`**（小さいほど先）… 計画基準納期の由来（**改定 2026-03**）  
   - 0: グローバルコメント AI の `task_planning_basis_dates`  
   - 1: 特別指定備考 AI の目標完了日等  
   - 2: 特別指定備考 AI の開始日  
   - 3: 指定納期_上書き  
   - 4: 原反投入日を計画基準納期にした行  
   - 5: 回答納期  
   - 6: 指定納期  
   - 9: なし
2. **`has_done_deadline_override`** … AI 等で締め・開始日を上書きした行を先（`True` を先）
3. **`b_tier`（§B の段）** … **0** = §B-1（`roll_pipeline_inspection` かつ `in_progress`）、**1** = §B-2 帯（`roll_pipeline_ec` または、未着手の `roll_pipeline_inspection`）、**2** = 上記以外。§B を §A より先に並べるための段
4. **`b2_queue_sub`（§B-2 帯内の順）** … `b_tier = 1` のときだけ意味があり、**0** = `roll_pipeline_ec`（EC 先行）、**1** = 未着手の `roll_pipeline_inspection`（検査）、**それ以外の `b_tier` では 0**。同一依頼内で **配台試行順番（結果一覧）が EC → 検査**になる土台
5. **§B-1 試行順タイブレーク** … `roll_pipeline_inspection` かつ `in_progress` の行だけ **`(0, planning_sheet_row_seq)`**（シート上で上ほど先）。それ以外は **`(1, 0)`**
6. **`in_progress`（§B 対象外のみ）** … `roll_pipeline_inspection` でない行について **加工途中**を先
7. **`priority`** … 配台計画の優先度（**小さいほど先**。セル優先、無ければ特別指定備考 AI）
8. **`due_urgent`** … 計画基準日において納期基準が「遅れ気味」扱いのものを先
9. **`due_basis_date`** … 早いほど先（無い行は後回し）
10. **機械名** … NFKC 正規化キー（同一計画基準納期内の同機械をまとめる）
11. **依頼NOタイブレーク** … `_task_id_same_machine_due_tiebreak_key`（改定 2026-03・§4）
12. **`start_date_req`** … 早いほど先
13. **依頼NO** … `_task_id_priority_key`（ハイフン後が数値なら小さい方が先）
14. **必要OP人数** … **大きいほど先**（`resolve_need_required_op` の結果を負号でソート）

§B-1 行では **`planning_sheet_row_seq`** がキーに入るため、**付与後の `dispatch_trial_order` はシート上の相対順と概ね一致しやすい**です（それでも番号の基準はあくまで上記ソート後の並びです）。

### 2.0.1 日内の試行順（`_day_schedule_task_sort_key`）

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=0`** のときだけ、同一日内の割付ループがこのキーで **`sorted(...)`** されます。**計画基準納期 → 機械名 → 依頼NOタイブレーク（§4）** → **§B-1**: `roll_pipeline_inspection` かつ `in_progress` の行は **`(0, dispatch_trial_order)`**、それ以外 **`(1, 0)`** → **`process_sequence_rank`** → **`same_request_line_seq`** → **`dispatch_trial_order`** → **`b2_roll_pipeline_stage`（0=EC、1=未着手検査、2=その他）** → **配台計画の優先度** → **`_result_task_sheet_sort_key`**（配台試行順番・依頼NO・機械名…）です。  
このため **`dispatch_trial_order` は並びの要素の一つ**であり、納期・rank 等と**同時に**効きます（試行順だけが絶対の第一キーではありません）。

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`（既定）** のときは、割付の主経路は **§2.0.2** を参照してください。

**結果_タスク一覧**の行順は **`_result_task_sheet_sort_key`** により **配台試行順番**（`generate_plan` 冒頭の `task_queue.sort` 後に付与した 1..n）が第一キーです。

### 2.0.2 配台試行順優先の日内割付（`_trial_order_first_schedule_pass`・既定）

**`STAGE2_DISPATCH_FLOW_TRIAL_ORDER_FIRST=1`** のとき、各日・各マルチパスで次を行います。

1. **`tasks_today`**（その日 `start_date_req` 以前で残あり）を起点に、**`_trial_order_flow_eligible_tasks`** で候補を絞る。除外され得るものの例: §A-1 の先行工程待ち、§B-2 の EC 完走待ち・検査ロール枠ゼロ、（既定で）§B-2 検査の同一設備占有、**同一設備列でより小さい `dispatch_trial_order` がまだ当日詰めうる**ときの後続試行順（`_equipment_line_lower_dispatch_trial_still_pending`）など。
2. 残った候補を **`dispatch_trial_order` 昇順**に並べる。
3. **1 パス**では、その並びの**先頭から順に**各タスクを見るが、**あるタスクで 1 ロールでも割付が進んだら**、実装は **`_drain_rolls_for_task`** により**そのタスクについて可能な限り連続でロール割付**を試み、**進展があればそのパスはそこで終了**する（**同一パス内では、試行順の後続タスクには回らない**）。
4. 外側のマルチパス（`while`）が続く限り、**再度 1 から**同様のパスを繰り返す。

**読み替えのポイント**

- **試行順の若いタスクが優先**される（各パスは常に昇順リストの先頭側から検討）。
- ただし **「1 パス＝全タスクに 1 ロールずつ」ではない**。先頭側でまだ詰められる間は**同じタスクに複数ロール**が続けて割り当てられ、**パスをまたいで**再び先頭から詰める、という挙動に近い。
- **候補に入らなかったタスク**は、その日は試行順に関係なく割付対象にならない。

### 2.0.3 `dispatch_trial_order` と納期シフト再試行

各ラウンドの冒頭で `task_queue` は **§2.0** と同じキーで**再ソート**されますが、**`dispatch_trial_order` 自体は付け直していません**（初回 `generate_plan` 冒頭で付与した 1..n のまま）。並びが変わっても番号が古いまま残るため、**再試行後の「試行順」の解釈がキュー上の実順とずれる**場合があります（実装・ログ上の注意点）。

### 2.1 開始日 `start_date_req`

- **原反投入日が無い**とき: **`start_date_req` = 計画基準日**（データ抽出日＝`run_date`）。
- **原反投入日がある**とき: **`max(計画基準日, 原反投入日)`**（原反前倒しはしない）。  
- **特別指定**（セル／特別指定備考 AI の `start_date` マージ結果）に開始日上書きがあれば **`start_date_req` をそれに置き換え**（原反と同日またはそれ以降の指定が可能。原反より前の上書きはログのみで、実効日は上書き値）。
- メイン **再優先特別記載** の **工場休業日** は勤怠上全日非稼働。
- `generate_plan` 冒頭で、**`start_date_req` が誰も出勤しない日**のとき **直近の稼働日**へ繰り上げ。

### 2.2 原反投入日と同日開始

- 原反投入日と**同日**に加工を開始し、かつ**開始日上書きが無い**場合、**その日の加工開始下限**は **`same_day_raw_start_limit` = 13:00**（`time(13, 0)`。`_trial_order_flow_day_start_floor` でも原反当日は 13:00 帯を織り込む）。

### 2.3 配台残と納期の後ろ倒し再試行

**トリガ**（実装 **`_collect_task_ids_missed_deadline_after_day`**）: **各日のシミュレーション終了時点**で、**残量があり**、**`due_basis_date`（内部の計画基準納期）が存在し**、`start_date_req` ≤ 当日 で、かつ **当日 ≥ `due_basis_date`**（納期当日を含み「期限内に完遂できなかった」）となった **依頼NO** を集めます。

この **ミスした依頼NO** 集合に対し、`STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING` がオンのとき **納期後ろ倒し再試行**に入ります（**割当履歴の有無は条件に含めません**。以前の「配台残」の口語定義とは異なる点に注意）。

- **フラグ**: `planning_core.py` の `STAGE2_RETRY_SHIFT_DUE_ON_PARTIAL_REMAINING`（既定 **`True`**）。`False` にすると **1 パス**の日ループのみです。
- **1 ラウンドの内容**: 対象依頼NOの各行について **内部の `due_basis_date`（計画基準納期）のみ +1 日**（`_shift_task_due_calendar_fields_one_day`）。ソート・納期超過判定・再試行の優先度はこのシフト後の値を使う。**結果_タスク一覧の「計画基準納期」列**は **`due_basis_date_result_sheet`**（キュー構築時の元日付）のまま表示し、+1 連鎖は反映しない。**回答納期・指定納期・指定納期_上書き**もシート取り込み値のまま。`remaining_units` と `assigned_history` を初期化し、**勤怠の自動拡張で増えた日付キーはマスタの日付集合へ巻き戻して**から、再度全日程を走査します。
- **シフトしない依頼**: いずれの行も **`due_basis_date` が無い**依頼は、後ろ倒しの対象にせず通常の配台残のままです。
- **勤怠最終日を超える場合**: `due_basis_date` を持つ行があり、**+1 日が勤怠マスタ上の最終計画日を超える**依頼は、そのラウンドではシフト対象外（カレンダー不足側の分類）になります。
- **上限**: `STAGE2_RETRY_SHIFT_DUE_MAX_ROUNDS`（既定 **40**）は **依頼NOごと**（シミュレーション全体の合計ではない）。当該依頼について計画基準+1の巻き戻し再シミュをこれ以上行わず、ログに警告し得ます。上限到達後も未完了が残る当該依頼の行には「納期見直し必要」を付けます。

各ラウンドの冒頭で `task_queue` は **2.0 グローバルキュー**と同じキーで再ソートされます。**§2.0.3** のとおり **`dispatch_trial_order` は再付与されません**。

## 3. 誰を候補にするか（スキル・人数）

### 3.1 稼働

- `master.xlsm` 各メンバー勤怠で、**加工ライン配台の候補**は **`eligible_for_assignment`** が真の人のみ（勤怠備考 AI の **`配台不参加`** が true の日は除外）。**出勤表示**は **`is_working`**（全日休み・欠勤は `is_holiday` 由来で false）。月次点検など「出勤はするが配台だけ不可」は `is_working` true・`配台不参加` true になり得る。
- 勤怠 **備考** が空でも **休暇区分** だけ入っている行は、文脈解釈のため **同じ勤怠 AI** に渡す（表記揺れ・婉曲表現はキーワードマッチではなく AI で解釈）。
- 個人の空き開始は **`start_dt`**（退勤は `end_dt`）

### 3.2 スキル（`skills` シート）

- **工程名＋機械名**の交差セルが **OP** または **AS**（＋優先度整数）のメンバーのみ候補
- チームは **少なくとも1名が OP** でなければ不採用（AS のみの組は不可）
- 優先度は **数値が小さいほど**チーム選定で有利

### 3.3 必要人数 `req_num`

1. 配台計画シートの **必要OP** 列に正の整数があればそれを使用
2. 無ければ **`need` シート**：`工程+機械名` → 機械名のみ → 工程名のみの順で参照
3. **特別指定1〜99** は、条件が依頼NOに一致したうえで上書き（**order が小さいほど優先**）

### 3.4 メイン「再優先特別記載」（グローバル AI）

- **`ignore_need_minimum`** … `req_num` を **1** に固定（`need`・行の必要OPより優先）
- **`ignore_skill_requirements`** … 全員 **OP 扱い**かつ優先度を実質同一にし、スキルマトリクスを無視
- **`abolish_all_scheduling_limits`** … 設備専有・原反同日13:00下限・指定開始時刻・マクロ実行時刻下限などを適用しない（併せて skill / need も緩和される経路あり）

### 3.5 余剰増員（`need` の「配台時追加人数／余力時追加人数」行）

`master.xlsm` の **need** シートで、基本必要人数行の**直下**（Excel 上は概ね **5 行目付近**）に置かれる行を読み、工程×機械列ごとに **必要人数を満たしたうえで追加で入れられる人数の上限**（`extra_max`／`surplus_map`）を定義します。

**既定（2026-04 以降）**

1. **メインの配台シミュレーション**では、この上限を **チーム人数の組み合わせ探索には使わない**。メインでは **`req_num`（基本必要人数）ちょうど**でチームを組む（`_assign_one_roll_trial_order_flow` および非試行順優先フローで `extra_max` を 0 扱い）。
2. **全日程の割付シミュレーションが終了した後**、`append_surplus_staff_after_main_dispatch` が各タイムラインブロックを時系列に見て、**まだ `req_num + extra_max` に達していない**ブロックに対し、次をすべて満たす人を **サブ（`sub`）に追記**する。  
   - 当日 **`eligible_for_assignment`**  
   - **OP または AS** で当該工程×機械にスキル適合（メイン割付と同じキー解決）  
   - そのブロックの **`[start_dt, end_dt)` と時間が重ならない**他の配台イベントに**未参加**（いわゆる配台未割当の時間帯）  
   - 追記順はスキル優先度のうえ、**AS を OP より先**に並べる  
3. **ブロックの開始・終了時刻は再計算しない**（メインで決めた `end_dt` のまま）。追記は **結果表示・サブ欄・個人スケジュール上の同席表現**が主目的。

**環境変数**

| 値 | 挙動 |
|----|------|
| （`TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS` 未設定） | 上記「メインは req_num のみ → 後追いサブ」 |
| `TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS=1` | **従来どおり**、メイン割付中に `req_num`〜`req_num+extra_max` を探索し、`_surplus_team_time_factor` も当該 `extra_max` で評価 |
| `TEAM_ASSIGN_IGNORE_NEED_SURPLUS_ROW=1` | need の追加人数行を **無視**（メインも **後追いも** なし。従来どおり） |

## 4. 時刻・設備の制約

### 4.1 チーム開始時刻

- 各メンバーの **`avail_dt`（その時点の空き）** の **最大値**（最も遅い人に合わせる）
- **`abolish_all_scheduling_limits` がオフ**のとき、次を順に適用  
  - **同一設備**は1タスクずつ：`machine_avail_dt` より前には開始しない  
  - 原反同日の **13:00 下限**（`same_day_raw_start_limit`／`_trial_order_flow_day_start_floor`）

- **`DEFAULT_START_TIME`** … 8:45  
- **`DEFAULT_END_TIME`** … 17:00

## 5. チーム選定の優先基準

そのタスク×その日について **成立する候補チームをすべて集めたうえで**、候補ごとにタプルを比較し、**辞書式で小さい方**を採用します。  
比較には **その日の候補全体の最早開始時刻 `t_min`** を使います。

### 5.0 環境変数の優先関係

1. **`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF=1`**（従来モード）… 下記を **無条件で** 人数最優先にします（スラック分は使いません）。
2. 上記が **オフ**（既定 `"0"`）で、**`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES`**（既定 **60**）が **正**のとき … **開始優先＋1時間以内は人数を厚く**（下記「スラック付き」）。
3. **`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES=0`** かつ余剰優先オフ … **開始のみ**優先（下記「開始のみ」）。

### 5.0.1 従来モード（`TEAM_ASSIGN_PRIORITIZE_SURPLUS_STAFF=1`）

- **`(-チーム人数, 開始時刻, -その日の単位数, 優先度合計)`**  
  → **人数が多い** → **開始が早い** → **その日の処理量が多い** → **スキル優先度合計が小さい**

### 5.0.2 スラック付き（既定の割合）

- **`t_min`** … その日の **すべての成立候補**のうち **最も早い** `team_start`
- 候補の遅れ **`team_start - t_min`** が **`TEAM_ASSIGN_START_SLACK_WAIT_MINUTES` 分以下**のとき（**同じ「帯」**）:  
  **`(0, -チーム人数, 開始時刻, -その日の単位数, 優先度合計)`**  
  → **スラック時間以内の開始の遅れ**であれば **人が集まる（人数の多い）チーム**を優先
- 上記の **スラックを超えて遅い**候補は:  
  **`(1, 開始時刻, -チーム人数, -その日の単位数, 優先度合計)`**  
  → **開始が早い**方を優先し、同帯では人数・単位数・優先度でタイブレーク

### 5.0.3 開始のみ（スラック 0）

- **`(開始時刻, -その日の単位数, 優先度合計)`**

### 5.1 主担当（lead OP）

- **担当OP指名**が当日の OP 候補に解決でき、かつチームに含まれる → その人を lead
- それ以外 → OP のうち **(スキル優先度, 名前)** が最小の人

### 5.2 担当OP指名の入力源と照合

- 配台計画の **担当OP_指定** と、特別指定備考 AI の **`preferred_operator`**（セル優先）
- メイン再優先の **`task_preferred_operators`** がある場合は **最優先で上書き**
- 名前は NFKC 正規化・**富田/冨田**（姓側の表記寄せ）で、当日 **`op_today`**（その設備で OP の人）と照合
- **姓・名の表記ゆれルール**（`skills` の全メンバーで判定）  
  - **姓**は正規化後 **完全一致のみ**（別姓を名のゆれで誤マッチさせない）。  
  - **`skills` に同一姓のメンバーが2人以上いない** ときだけ **名**は、完全一致に加え **どちらかの文字列が他方を含む**場合も一致とみなす（略称・表記のゆれ）。  
  - **同一姓がロスターにいる** ときは **名も完全一致**のみ。  
  - 姓・名の区切りは **最初の半角／全角空白**（マスタの「姓　名」形式に合わせる）。名ゆれモードで **姓だけ**入力したとき、同姓候補が複数いれば **解決しない**（曖昧のため）。
- **一致しない場合**はログに記録し、**指名は無効**となり、通常の組み合わせ探索のみ（別の OP で埋まり得る）

## 7. 計画の日付範囲

- 勤怠データに含まれる日のうち、**計画基準日（データ抽出日）以降**が対象です（`generate_plan` 内で `sorted_dates` を基準日以降に絞り込み）。
- **現行（段階2）**: **`STAGE2_EXTEND_ATTENDANCE_CALENDAR = False`** のため、上記の勤怠に載っている日付の範囲**だけ**で各パスの日ループを行い、**勤怠の自動拡張（日付の複製追加）は行いません**。最終日までに割り切れないタスクは **配台残**または **配台不可**のまま残り得ます。
- **配台残の再試行**: 既定では **セクション 2.3** のとおり納期を後ろ倒しして**同じ勤怠範囲でもう一度**全体を回します（勤怠日付そのものを増やすのとは別メカニズム）。
- **オプション**: `planning_core.py` で **`STAGE2_EXTEND_ATTENDANCE_CALENDAR = True`** にすると、**残タスクがあるのに最終勤怠日までに終わらない場合**、直近の勤怠日と同じ出勤パターンを **1 日ずつ複製**して日付を延長し、残ロールがなくなるまで配台を続けます。拡張の上限は **`SCHEDULE_EXTEND_MAX_EXTRA_DAYS`（既定 366 日）**です。

## 9. 結果_タスク一覧のタスクID表示（加工内容との整合）

**目的**: 元データ側の **「加工内容」**（カンマ区切りの工程リスト）と、配台計画の **「工程名」** が食い違っている行を、段階2の結果ブック上で識別しやすくします。

**対象**: 段階2が出力する `production_plan_multi_day_*.xlsx` の **結果_タスク一覧** シートの **タスクID** 列（セルのみ）。

**判定**（`build_task_queue_from_planning_df` 内。加工内容の取り方は §A-1 と同じ `_collect_process_content_order_by_task_id`）:

- 同一 **依頼NO** について、表を上から見て最初に現れた **非空の加工内容** をカンマ `,` で分割したトークン列を「正しい工程名の一覧」とみなします。
- **トークンが1つも無い**依頼（加工内容が未記入のみ等）は、比較基準が無いため **ハイライトしません**。
- トークンがある依頼で、当該行の **工程名** が（NFKC・`_normalize_process_name_for_rule_match` による照合で）**いずれのトークンとも一致しない** とき、その行の **タスクID** セルに **塗りつぶし赤（`FF0000`）・文字白** を付与します。

**その他の列・既存の色**:

- **配台不可**／**配台残** の行に対する薄い黄の行全体の塗りより **後** にタスクIDだけ上書きするため、不整合がある行では **タスクIDセルは赤が優先**されます（他列は黄のまま）。

**結果_設備毎の時間割へのリンク**（`_apply_result_task_id_hyperlinks_to_equipment_schedule`）:

- **タスクID** の表示文字に **内部ハイパーリンク**を付け、**結果_設備毎の時間割** シートで当該タスクが **最初に現れるセル**（`[タスクID] 主:…` 形式のテキストが入る最初のマス）へジャンプできる。
- 時間割に一度も現れないタスク（例: 未割当のみ）はリンクなし。
- 同一依頼NOが複数行ある場合も、タスクID文字列が一致すれば同じ先頭セルを指すことがある。

**履歴列の黄色（need 追加人数）**（`_apply_result_task_history_need_surplus_highlight`）:

- **履歴1～n** のうち、**need の基本必要人数（req_num）を超えた人数**で割り当てたブロックに対応するセルのみ **薄黄（`FFEB9C`）** に塗る（配台不可／配台残の行全体の薄黄より **後** に上書き）。
  - **メイン割付**: `TEAM_ASSIGN_USE_NEED_SURPLUS_IN_MAIN_PASS` が有効 **かつ** 当該行の追加枠（`extra_max`）が **1 以上** **かつ** **採用人数 > req_num** のとき。
  - **全配台後の余力追記**（`append_surplus_staff_after_main_dispatch`）: 追記 **後** の当該ブロック人数が **req_num を超える** ときのみ（必要人数ちょうどまでの追記だけでは塗らない）。余力追記後は **履歴文字列の担当**をタイムラインの `op`/`sub` に合わせて更新する。

**履歴列のテキスト（組み合わせ表・追加メンバー）**（`_format_result_task_history_cell`）:

- **組合せ表#n**: マスタ「組み合わせ表」の **A列「組合せ行ID」**（VBA `master_組み合わせ表を更新` でデータ行ごとに 1 から採番）。`TEAM_ASSIGN_USE_MASTER_COMBO_SHEET` 有効時、当該行のプリセット編成が **メイン割付で採用された**ロールにだけ付く（組合せ探索のみで決まった場合は ID なし）。
- **追加[名前,…]**: メインで **基本必要人数より多い人数**を採用したときの、その超過分（表示は担当の先頭から `req_num` 名のあとを **追加**とみなす）。
- **余力追記[名前,…]**: メイン完了後の余力追記で `sub` に足したメンバー。

## 10. 配台デバッグ（ロールトレース JSONL）

**目的**: 割付シミュレーションで **1 ロール確定ごと**に、どの依頼・設備・時刻に入ったかを **JSONL**（1 行 1 JSON）へ追記する。

**環境変数**

| 変数 | 内容 |
|------|------|
| `DISPATCH_ROLL_TRACE_JSONL` | 出力ファイルパス。**相対パス**はマクロブック（`TASK_INPUT_WORKBOOK`）と同じフォルダ基準。**未設定または空**のときはトレースファイルを作らない（`xlwings_console_runner.run_stage2_with_roll_trace_jsonl_then_refresh_sheet_for_xlwings` は既定で `log/dispatch_roll_trace.jsonl` を設定）。 |
| `DISPATCH_DEBUG_STOP_AFTER_ROLLS` | 正の整数で、指定ロール数に達したら割付シミュレーションを打ち切る（部分結果の調査用）。 |

**初期化のタイミング**: `generate_plan` の **先頭**で `_dispatch_debug_reset_roll_trace` を実行し、`log` 等の親フォルダを作成したうえでファイルを **空にトランケート**する。これにより、メンバー0人・タスクシート読込エラー・優先度重複などで **日次ループに入る前に return しても**、トレースが有効なら **空の JSONL が残る**（以前はループ直前のみ初期化のためファイルが一切できないことがあった）。

**追記**: 日次割付でロールが記録されるたびに `_dispatch_roll_trace_after_roll` が **追記**する。シミュレーションにロールが1つも無い場合は **0 行の空ファイル**のまま。

## 11. 配台トレース（設定シート A3 のみ）

**目的**: 指定した **依頼NO** について、割付シミュレーション中に `[配台トレース task=…]` 形式の **詳細ログ**を `log/execution_log.txt` に出す（進捗が止まる原因調査用）。

**入力**

- マクロブック **「設定」シートの A 列 3 行目以降** … 依頼NOを縦に並べる（空セルはスキップ。詳細は `_read_trace_schedule_task_ids_from_config_sheet`）。**環境変数では指定しない**（旧 `TRACE_SCHEDULE_TASK_ID` は廃止）。

**集合**: 上記 A 列から読んだ依頼NOが `TRACE_SCHEDULE_TASK_IDS`。`generate_plan` **冒頭**（ロールトレース JSONL 初期化の直後、**マスタ skills 読込より前**）に読み込み・**INFO ログを必ず出力**する。以前は `load_skills_and_needs` の後にしか読んでおらず、**メンバー0人で段階2が即 return した場合は設定シートのログが一度も出ない**不具合があった。

**依頼NOごとのログファイル**: `[配台トレース task=…]` 行は従来どおり **`log/execution_log.txt`** にも出し、あわせて **`log/dispatch_trace_<依頼NO>.txt`**（ファイル名は依頼NOをサニタイズ）へ**同一文面**を追記する。**配台トレース有効時**、`generate_plan` 冒頭で **`log/dispatch_trace_*.txt` をすべて削除**したうえで、今回の対象依頼NO分だけヘッダ付きで**新規作成**する（過去実行の別依頼NOファイルが残らない）。

**制約**: ブックにシート **`配台_配台不要工程`**（openpyxl 非互換マーカー）があると、openpyxl で「設定」を読めないため **A3 以降は無視**され、**配台トレースは無効**になる（ログに理由を INFO で出す）。

## 12. デバッグ限定配台（設定シート B3・環境変数）

**目的**: 全依頼ではなく、指定した **依頼NO** に該当する **配台計画_タスク入力** の行だけで段階2の配台シミュレーションを試す（検証・デバッグ用）。

**入力**

1. **優先**: マクロブック **「設定」シートの B 列 3 行目以降** … 依頼NOを縦に並べる（空セルはスキップ。連続 30 セル空で打ち切り・最大 500 行。1 セルにカンマ区切りで複数可。`_read_dispatch_debug_only_task_ids_from_config_sheet_b`）。
2. **B 列が実質空のときの代替**: 環境変数 **`DISPATCH_DEBUG_ONLY_TASK_IDS`** … カンマ区切り（例: `W4-1,Y3-26`）。

**処理**: `generate_plan` **冒頭**（配台トレース A 列の確定の直後）で上記を読み、モジュール集合 **`DISPATCH_DEBUG_ONLY_TASK_IDS`** を確定する。`load_planning_tasks_df` 成功直後、**依頼NO** 列がこの集合に含まれる行だけ残し、0 行なら **ERROR ログのうえ段階2を中断**する。有効時は **WARNING** で対象依頼NO一覧をログに出す。

**VBA**: `段階2_コア実行` および xlwings 経由の **配台デバッグ**入口では、B 列に値があるとき **段階2開始前**に **MsgBox** でデバッグ限定配台である旨を通知する。**cmd / xlwings いずれも** Python が **ディスク上のブック**から B 列を読むため、**通知のあと `ThisWorkbook.Save` してから**段階2を実行する（未保存だと B 列が反映されない）。

**制約**: セクション 11 と同様、ブックに **`配台_配台不要工程`** があると openpyxl で「設定」を読めず **B3 以降は無視**される。ログに理由を出す。**限定配台**については環境変数 **`DISPATCH_DEBUG_ONLY_TASK_IDS`** を代替として使える（**配台トレースの A 列には環境変数は無い**）。

**注意**: **A 列**は **配台トレース**用、**B 列**は **対象依頼の絞り込み**用で、役割が異なる。両方指定した場合、トレース対象と実際に配台する行の集合は一致しないことがある。
